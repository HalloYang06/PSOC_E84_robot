# AI 协作平台产品化功能路线图

版本：v0.1  
日期：2026-04-19  
用途：将平台能力按阶段拆分，明确哪些功能属于第一版必须完成、哪些属于增强、哪些属于上市前必须补齐、哪些属于后续高价值扩展

---

## 1. 文档目标

本路线图的目标不是继续堆功能，而是回答以下问题：

1. 第一版到底应该把什么做完。
2. 哪些功能是为了“能用”。
3. 哪些功能是为了“好用”。
4. 哪些功能是为了“能上市”。
5. 哪些功能是后续产品化和商业化的高价值扩展。

平台必须始终围绕一个核心定位：

**人类工程师主导，AI 团队协作，服务嵌入式和机器人研发。**

---

## 2. 功能分层原则

所有能力按四层划分：

### L1 基础可用层

目标：

让团队能真正用平台管理项目、任务、AI、Runner、审批和交接。

特点：

1. 必须可运行。
2. 必须能闭环。
3. 可以朴素，但不能是假功能。

### L2 协作增强层

目标：

减少会议、减少重复沟通、减少上下文浪费。

特点：

1. 让平台比普通任务系统更有效率。
2. 让人和 AI 的协作更清晰。

### L3 上市必备层

目标：

达到对外交付时最基本的稳定性、安全性、权限边界和可信度。

特点：

1. 可以让外部团队使用。
2. 可以做私有化交付或正式售卖。
3. 可以承受真实项目压力。

### L4 高价值扩展层

目标：

形成差异化竞争力和产品卖点。

特点：

1. 能提升长期留存和产品壁垒。
2. 更适合机器人和嵌入式场景。

---

## 3. 第一版必须完成的功能

这部分是**第一版必须做完的底线**。

### 3.1 用户与成员系统

必须具备：

1. 用户注册
2. 用户登录
3. 成员邀请
4. 项目成员管理
5. 成员角色管理
6. 协作者状态查看

第一版目标：

让一个项目至少能把：

1. 人类成员
2. 审查人
3. 实验人员
4. 项目负责人

都纳入平台，而不是只管理 AI。

---

### 3.2 项目与任务系统

必须具备：

1. 新建项目
2. 编辑项目
3. 项目列表
4. 任务创建
5. 任务拆分
6. 任务状态流转
7. 任务负责人
8. 任务详情页
9. 任务相关日志、审批、交接、审计

第一版目标：

让项目真正从“任务”开始运转，而不是停在首页展示。

---

### 3.3 AI 工位与线程工位系统

必须具备：

1. AI 工位列表
2. 工位详情
3. 所属电脑
4. 所属模型
5. 职责
6. 权限等级
7. token 状态
8. 当前任务
9. 上下文健康

第一版目标：

让每个 AI 线程成为一个可管理对象，而不是一个分散的聊天窗口。

---

### 3.4 多电脑 / Runner 系统

必须具备：

1. Runner 注册
2. Runner 心跳
3. Runner 领取任务
4. Runner 日志回传
5. Runner 结果回传
6. Runner 所属电脑展示

第一版目标：

让“多电脑协作”从概念变成真实能力。

---

### 3.5 需求管理库

必须具备：

1. 需求单创建
2. 需求单列表
3. 需求单详情
4. 指定接收方
5. 采纳 / 升级 / 关闭
6. 结构化回复
7. 需求与任务关联

第一版目标：

替代 AI 之间无序聊天。

---

### 3.6 审批门岗

必须具备：

1. 审批单创建
2. 审批列表
3. 审批详情
4. 审批通过 / 驳回 / 要求修改
5. 审批备注
6. 与任务和实验楼联动

第一版目标：

把高风险动作都挂到人类确认点上。

---

### 3.7 上下文健康与交接站

必须具备：

1. 上下文健康列表
2. 上下文等级
3. 摘要生成
4. 交接包创建
5. 交接包接手
6. 交接记录查看

第一版目标：

防止线程上下文过载，把接手机制做实。

---

### 3.8 审计账簿

必须具备：

1. 审计日志列表
2. 按任务查看
3. 按项目查看
4. 审批相关留痕
5. 交接相关留痕
6. 需求相关留痕

第一版目标：

保证所有关键动作可追溯。

---

### 3.9 游戏化研发基地前端

必须具备：

1. 基地总览
2. 任务大厅
3. AI 工位区
4. 代码车间
5. 硬件实验楼
6. 财务室
7. 需求信箱
8. 审批门岗
9. 交接站
10. 知识图书馆

第一版目标：

让平台不是普通表格后台，而是更直观、更统一的研发基地。

注意：

游戏化是表现层，底层必须对应真实工程数据。

---

## 4. 第二版增强功能

这部分属于第一版跑通后最值得补强的能力。

### 4.1 正式的多电脑 / 多 provider 管理面

增强内容：

1. 电脑节点 CRUD
2. AI provider CRUD
3. 线程工位 CRUD
4. 电脑能力标签
5. provider 凭证状态

目标：

把“配置结构存在”升级成“用户真正能管理”。

---

### 4.2 模块责任地图

增强内容：

1. 模块列表
2. 模块负责人
3. 相关目录
4. 风险等级
5. 最近变更

目标：

特别适合机器人和嵌入式项目。

---

### 4.3 接口契约中心

增强内容：

1. 接口契约列表
2. owner / consumer
3. 协议字段
4. 示例数据
5. 版本号
6. 关联文件

目标：

降低跨模块沟通成本。

---

### 4.4 实验记录流

增强内容：

1. 实验创建
2. 实验参数
3. 照片 / 视频
4. 数据上传
5. 结果记录
6. 结论沉淀

目标：

让平台真正适合实验室和现场研发。

---

### 4.5 任务接力模式

增强内容：

1. 任务链
2. 接力顺序
3. 每棒输入输出
4. 接手人 / AI
5. 交接风险

目标：

适配机器人项目跨模块协作。

---

## 5. 上市前必须补齐的能力

这部分不一定先做视觉，但必须做，否则不建议上市。

### 5.1 鉴权与权限收严

必须完成：

1. 所有写接口要求已认证用户或 Runner 身份
2. 项目成员校验
3. 权限等级校验
4. 审批权限校验
5. 线程 / 工位写权限校验

---

### 5.2 敏感信息保护

必须完成：

1. 不返回邀请 token 明文
2. 不返回明文平台密钥
3. token 使用审计
4. 最小权限原则

---

### 5.3 高风险动作硬门岗

必须完成：

1. H3 / H4 动作必须真实人类确认
2. 审批必须有备注
3. 审批必须有责任人
4. 未审批不能继续执行危险动作

---

### 5.4 构建和发布稳定性

必须完成：

1. 前端稳定 build
2. 后端稳定启动
3. Docker 部署稳定
4. 关键页面不报错
5. 主链演示可重复

---

### 5.5 协作主链统一

必须完成：

1. 登录
2. 成员
3. 邀请
4. 协作者
5. 讨论区
6. 任务留言
7. 需求沟通

统一成一条清晰主链。

---

### 5.6 数据治理

必须完成：

1. 正式 migration 体系
2. 样板数据与正式数据隔离
3. 启动不自动改生产数据
4. 审计日志可保留

---

## 7. 推荐推进顺序

建议按以下阶段推进。

### 阶段 A：把第一版真正做实

目标：

1. 所有页面不是摆设
2. 所有关键入口有真实动作
3. 任务、审批、交接、审计主链打通

### 阶段 B：把协作能力做强

目标：

1. 多成员协作清晰
2. 多线程工位可管理
3. 多电脑和多 provider 真的能配置

### 阶段 C：把嵌入式和机器人特性做出来

目标：

1. 硬件实验楼实用
2. 实验记录流可用
3. 模块责任地图落地
4. 接口契约中心落地

### 阶段 D：把上市门槛补齐

目标：

1. 权限和安全收严
2. 构建和部署稳定
3. 审计链真实可靠
4. 数据治理正规化

### 阶段 E：做差异化卖点

目标：

1. 整机版本快照
2. 任务接力模式
3. 设备管理
4. 数据与模型管理
5. 机器人专项模式

---

## 8. 最终建议

不要把平台做成“什么都想先做”的大杂烩。

最稳的路线是：

1. 先把第一版主链做实
2. 再把协作能力收口
3. 再把机器人和嵌入式特色做出来
4. 最后补上市门槛和差异化

一句话总结：

**先从“能用”做到“好用”，再从“好用”做到“可上市”，最后从“可上市”做到“有壁垒”。**
