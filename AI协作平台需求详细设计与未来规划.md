# AI 协作平台需求详细设计与未来规划

版本：v0.1  
用途：把平台需求从 MVP 扩展成可长期演进的产品需求池  
定位：当前能开发、后续能增强、未来能产品化

---

## 1. 产品长期定位

本平台不是一次性工具，也不是单项目脚本，而是一个面向嵌入式、机器人、软硬件协作项目的长期产品。

核心定位：

```text
人类总工程师主导的 AI 研发团队管理平台。
```

长期目标：

1. 支持多个项目。
2. 支持多个团队。
3. 支持多个 AI 模型和 Agent 工具。
4. 支持多个 Runner 和多台电脑协同。
5. 支持 GitHub/Gitea/Forgejo/GitLab 等 Git 平台。
6. 支持嵌入式、ROS、机器人、Web、App、AI 模型训练等项目模板。
7. 支持需求、任务、代码、文档、硬件调试、成本、安全的统一管理。
8. 支持游戏化研发基地前端，形成差异化产品体验。

产品边界：

```text
AI 是研发助手，不是最终决策者。
人类负责硬件调试、安全确认、主分支合并、发布和重大架构决策。
```

---

## 2. 需求分层

平台需求按 4 层规划：

```text
L1 MVP 必须有:
  证明平台能跑通核心闭环。

L2 可用版本:
  能支持真实团队日常开发。

L3 增强版本:
  提升自动化、成本控制、安全性和跨项目复用。

L4 产品化版本:
  支持多团队、插件、模板、部署、商业化和生态。
```

---

## 3. 核心闭环需求

### 3.1 MVP 闭环

第一版必须跑通：

```text
创建项目
  -> 添加 AI
  -> 注册 Runner
  -> 创建任务
  -> 指派 AI
  -> 创建任务分支
  -> AI/Runner 执行
  -> 查看日志和 diff
  -> 记录 token
  -> 审查
  -> 人类确认
  -> 合并或回滚
```

验收：

```text
[ ] 可以完成一次从任务创建到代码审查的完整演示
[ ] 可以看到 AI 状态和 token 消耗
[ ] 可以阻止 AI 执行高风险硬件操作
[ ] 可以回滚到指定 commit
```

### 3.2 未来闭环

未来完整闭环：

```text
项目模板导入
  -> AI 团队自动生成
  -> 需求自动拆分
  -> AI 需求路由
  -> 多 AI 协作
  -> 上下文健康监控
  -> 自动生成交接包
  -> 代码/文档/测试/硬件调试分流
  -> 多级审查
  -> 发布检查
  -> 版本归档
  -> 每日/每周战报
```

## 24. 总结

这套需求规划的核心是：第一版要小而完整，后续要能自然长大。

```text
第一版做闭环。
第二版做协作。
第三版做嵌入式特色。
第四版做团队可用。
第五版做产品化。
```

所有需求都要围绕一个中心：

```text
让人类工程师更容易管理 AI 研发小队，
而不是让 AI 无边界地替代工程师。
```
