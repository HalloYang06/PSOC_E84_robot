## 2) 每个线程职责（第一版）

### AI-Boss（本机）

职责：

1. 把第一版范围钉死在“核心闭环”，阻止发散。
2. 与 AI-PM 共同维护本周优先级和阻塞列表。
3. 发现线程/任务卡住时，安排接手或拆小任务。
4. 组织每日检查清单与日报草稿（不写业务代码）。
5. 明确哪些事项必须人类确认，避免越权。

输出物：

1. 本周 6 个最优先任务列表（含验收标准）。
2. 阻塞清单和解阻建议。
3. 当日战报草稿（完成/阻塞/风险/需人类处理）。

第一版期间建议固定保留的线程（数量越少越好，避免管理成本）：

## 5) 本周调度规则（第一版）

### 5.1 调度优先级

本周只围绕“核心闭环”调度：

1. 项目/AI/Runner 基础管理
2. 任务管理
3. Runner 执行与日志回传
4. Git 分支与 diff（最小可用）
5. token 统计（最小可用）
6. 人工确认与审计日志（最小可用）
7. 研发基地首页（轻量版）

禁止抢优先级的内容：

1. 多租户/商业计费/插件市场
2. 复杂 3D/复杂动画/娱乐化系统
3. 自动控制真实硬件/自动烧录
4. 没有闭环支撑的大范围架构重构

### 5.2 任务颗粒度

规则：

1. 单任务应能在 0.5-2 天内完成并可验收。
2. 任务必须有明确“验收标准”。
3. 任务必须有“允许修改目录”和“禁止修改目录”。

### 5.3 合并节奏

规则：

1. 每天最多安排 1-2 次合并窗口。
2. 所有合并必须通过最小检查清单（见第 6 节）。
3. 任何涉及权限/密钥/硬件边界的改动，必须先走人类确认。

### 5.4 阻塞处理

规则：

1. 卡住超过 60 分钟必须标记为阻塞。
2. 阻塞超过半天必须拆分任务或安排接手线程。
3. 阻塞必须写“下一步最小行动”，避免空转。

## 6) 每日检查清单（第一版）

每天由人类总工程师或 AI-Boss 执行一次：

1. 今日最优先 3 个任务是否明确（负责人/验收/目录边界都齐全）。
2. 是否有阻塞任务超过 1 天未处理（列出并决定：拆分/接手/降低优先级）。
3. 是否有人在做“禁止发散项”（发现即停止，改回闭环任务）。
4. 是否有任何疑似密钥泄漏（日志/commit/文档里出现 token/key）。
5. 是否有任何越权操作尝试（例如主分支直接提交、绕过人工确认）。
6. Runner 是否在线（至少 1 台可用）且能回传日志。
7. 今日合并窗口是否安排（最多 1-2 次）及审查人是谁。
8. 前端是否能在手机端打开（至少主要页面不崩、不溢出）。
9. 后端健康检查是否正常（API 可用、DB 可用）。
10. 今日 token 消耗是否异常（突增就先排查重复工作/日志未摘要）。

---

## 7) 人类总工程师需要看的 5 个关键指标（第一版）

每天只看这 5 个就够：

1. **闭环完成度**
   - 是否已经跑通：创建项目 -> 添加 AI -> 注册 Runner -> 创建任务 -> 执行 -> 日志/diff -> 人工确认 -> 合并/回滚
2. **阻塞数与阻塞时长**
   - 当前阻塞任务数量；最长阻塞已多久
3. **Runner 健康度**
   - 在线 Runner 数 / 总 Runner 数；是否能成功执行并回传日志
4. **合并安全性**
   - 今日合并次数；是否存在未审查合并；是否触发高风险确认点
5. **成本与上下文风险**
   - 今日 token 消耗；是否出现上下文过长/需要交接的任务

## 附：人类必须确认的事项（第一版提醒）

1. 合并到 `main`、分支保护变更
2. 回滚到任意 commit/tag
3. 任何密钥/token/凭证的新增与修改
4. 权限模型与人工确认等级（H0-H4）变更
5. 任何部署配置、端口暴露、服务器操作
6. 任何可能触达真实硬件的操作（第一版默认禁止自动化）
