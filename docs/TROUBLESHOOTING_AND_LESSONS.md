# Troubleshooting and Lessons

## 2026-07-08

- Symptom: users could tap the bottom `社区` tab on several mobile pages and see no useful result.
- Root cause: the rebuilt Stitch mobile pages still had `href="#"` placeholders for a product area that was not ready.
- Fix: generate a Stitch-first `社区内测` screen and route every `社区` tab to `community.html`; closed-beta features now show honest status and route users to the working康复师 Agent.
- Lesson: a disabled product area still needs a user-facing destination. `href="#"` reads as a broken App, not as a roadmap.

- Symptom: the App device page reported no Bluetooth bridge in APK-oriented flows even though the Android wrapper already had a tested Classic SPP plugin.
- Root cause: the current page looked for `RehabArmBluetooth.scanDevices`, while the Android wrapper exposes `RehabArmSpp.listBondedDevices/connect/sendLegacyFrame`.
- Fix: add a compatibility adapter in the mobile runtime that maps `RehabArmSpp.listBondedDevices()` into the existing device-search UI. This proves the APK can surface paired SPP devices without reintroducing BLE firmware code into M33.
- Lesson: do not add another Bluetooth stack before checking the installed native bridge name. Keep account device binding, SPP connection, frame send, ACK upload, and M33 acceptance as separate visible states.
- Agent UX lesson: a server-backed message page still feels fake if it opens with hardcoded sample user/assistant messages. Keep only a real empty/welcome state, then render user messages and `/agent/messages` responses. If a safety example is needed, keep it in docs/tests, not as visible default chat history.
- Mobile auth QA lesson: the current mobile runtime reads `localStorage.access_token`. Browser-injected QA that writes only an older key such as `rehab_arm_access_token` will hit login gates and falsely report that device scanning or binding is broken.
- Phone-binding product lesson: a button that calls a missing endpoint is worse than a disabled control. The shipped profile page called `/account/phone-verifications`, so the backend must expose a debug-SMS contract during internal testing and persist `phone_number` / `phone_verified_at` before the user can trust that account setup is complete.
- Production schema lesson: a local fresh SQLite test can pass while the live database has an older partial table. For `rehab_app_phone_verifications`, repair both new fields (`phone_number`, `debug_code`, `status`, `confirmed_at`) and legacy not-null fields (`phone`, `code_hash`, `attempts`, `consumed_at`) before inserting, or production will fail with `sqlite3.OperationalError` / `sqlite3.IntegrityError` even though the endpoint exists.
- Profile bootstrap lesson: the current `/api/rehab-arm/app/v1/me` response exposes `profile` at the top level. Mobile pages that only read `patient_view.profile` can show `未登录/未绑定` after a successful 200 response. Keep the frontend tolerant of both shapes until the contract is fully consolidated.
- Release deployment lesson: updating `apps/web/public/...` on the server is not enough if the user installs an APK. For App-visible fixes, sync Web assets into the Capacitor wrapper, rebuild the APK, update public-config/download-page version and SHA, upload the new APK, then verify the downloaded bytes hash.
- Agent QA lesson: icon-only send buttons can confuse broad Playwright selectors. For user-path QA on `ai-plan.html`, click `[data-action="send-agent-message"]` explicitly and assert the network request to `/api/rehab-arm/app/v1/agent/messages`; otherwise a test can click a visually nearby control and falsely report that the Agent is disconnected.
- Device no-bridge lesson: "browser cannot use Bluetooth Classic SPP" is correct but incomplete. The UI must also say the next user action: install the Android APK, pair the M33/SPP device in system Bluetooth, then return to App search. This keeps the no-hardware state honest without making the device flow feel like a dead end.

- Symptom: the message page looked like an intelligent assistant but the current repo did not expose the exact `/agent/messages` App route the page was calling.
- Root cause: frontend and cloud deployment had drifted ahead of the local backend contract.
- Fix: add `POST /api/rehab-arm/app/v1/agent/messages` with explicit App relay metadata, model/fallback status, audit logs, and dangerous request refusal.
- Lesson: every App chat/agent surface needs the same source/purpose/scope separation as the training planner, otherwise platform logs can mix patient guidance with XiaoZhi/L semantic routing.

## 2026-07-04

- Symptom: the `3D / 孪生` IK coordinate inputs still felt impossible to edit after a prior persistence fix, and the visible Stitch URDF upload area could show a new filename without changing the real Three.js/URDF model.
- Root cause: the iframe IK panel was still rebuilt while focused because `doc.activeElement instanceof HTMLElement` fails across iframe realms; additionally, the Stitch upload input was a separate DOM control from the React `Arm3DOverview` file input, so label changes did not prove the real loader received a file.
- Fix: detect focused iframe inputs without cross-realm `instanceof`, preserve the existing IK panel while editing, keep draft values in `ikDraftRef`, and bridge the Stitch file input into `Arm3DOverview` via an external URDF file request/nonce. QA must inspect `#codex-twin-runtime-stage`, not only the outer Stitch `body.innerText`.
- Status: fixed and deployed. Cloud QA uploaded `codex_bridge_test.urdf`; the real portal host reported `已导入 codex_bridge_test.urdf` and listed `codex_test_joint`.
- Legacy Bluetooth lesson: the old tested rehab-arm App protocol is Bluetooth Classic SPP/RFCOMM, not BLE GATT. It uses UUID `00001101-0000-1000-8000-00805F9B34FB`, UTF-8 JSON, and `\n` packet framing. A browser/PWA cannot open this RFCOMM link directly, so the Android package needs a native Capacitor/Kotlin bridge before phone-to-M33 communication is real.
- Protocol safety lesson: do not convert App hello/status/diagnostic requests into guessed old-protocol packets. The old protocol formally defines `mode`, `control`, `memory`, `execute_memory`, and `stop`; only send compatible frames with `sendable=true`, and keep unsupported status-like messages as backend evidence until M33 firmware confirms a command.
- Workflow-form lesson: phone workflow buttons are not closed-loop if each frontend shell guesses the payload shape. Return `payload_schema` and `form_contract` with each `/me/workflow.action_queue` item so Stitch/PWA/APK can render fields from backend-authored rules and submit through `/me/workflow/actions`.
- Workflow-preflight lesson: `PREFLIGHT_CHECK_REQUIRED` can be App-executable only after the current plan version has real `m33_accepted` evidence. Execute it by reusing the normal preflight service so pain thresholds, checklist completeness, sync id, and plan-version gates stay centralized.
- Shared AI relay lesson: using one model provider configuration for XiaoZhi/L and App training planning is acceptable only when every record carries channel metadata. XiaoZhi/L uses semantic routing records; App training planning must carry `relay_channel=app_training_planner`, `client_type=app`, `purpose=training_plan_draft`, `scope=rehab_training_planning`, and `does_not_touch_xiaozhi_l=true`. Do not point the App at XiaoZhi WebSocket or treat an App draft as L/A motion permission.
- App AI audit lesson: do not only store the App relay split inside the draft payload. Mirror `relay_channel`, `client_type`, `purpose`, and `scope` into `rehab_app.ai_training_draft.generated` audit events as well, otherwise a unified platform log can show App and XiaoZhi records but cannot reliably separate their control chains. The report-to-next-plan endpoint must carry the same metadata as direct `/ai-training-drafts/generate`.
- App AI sanitization lesson: a model prompt saying "do not output motor/CAN fields" is not enough. Recursively remove dangerous key variants from the persisted `generated_plan`, including camelCase or dashed spellings such as `emergencyStopRelease` and `motor-command`, before a draft can be accepted into a normal training plan.
- Stitch App AI page lesson: a generated Stitch screen can preserve the right visual direction but still need integration cleanup before landing in the App. Check for fake static values such as `SYS.ONLINE` or a hardcoded shoulder prescription, ensure `mobile-bridge.js` is loaded, keep all `data-arm-ai-*` hooks, and run a 390 px browser QA for horizontal overflow and touch targets.
- Cloud-call lesson: a successful `/ai-training-drafts/generate` response only proves the endpoint is callable. If the returned draft lacks `context_snapshot.ai_planner`, the cloud backend is stale or not on the latest App AI relay metadata commit; do not claim model relay/audit separation is deployed until the cloud response includes `relay_channel=app_training_planner`.
- Cloud external-model lesson: enabling `REHAB_ARM_MODEL_RELAY_API_KEY` changes App AI tests from deterministic fallback behavior to real model output. Run regression tests with `REHAB_ARM_MODEL_RELAY_EXTERNAL_ENABLED=false` for deterministic CI, then separately smoke the cloud model path and confirm `status=external_used`.
- App AI user-flow lesson: accepting a draft must be tied to the draft the user just generated. If `/me.latest_open_ai_draft` still returns older open drafts, the mobile page can appear to accept one plan and then redraw another. Disable accept while regenerating and prioritize the just-accepted plan in local state until the user starts a new generation.
- App AI serialization lesson: training reports and reviews can carry `datetime` values. Before sending report-derived context to an external model relay, pass the prompt payload through the JSON-safe normalizer; otherwise the report-to-next-plan endpoint only works in fallback mode and fails when real model relay is enabled.
- Mobile profile calendar lesson: a visually polished Stitch calendar is still a demo if it contains hardcoded month labels, heatmap cells, counts, or log buttons. For `我的 / 训练活动`, render from `/me.care_timeline` only; if no `training_session` evidence exists this week, show `暂无本周训练记录` and a real next step rather than a fake activity count.
- Calendar closure lesson: users expect plan creation and AI-draft acceptance to show up on the calendar immediately, not only after a finished session. Include `training_plan` entries in `/me.care_timeline`, highlight calendar days for all persisted rehab activity, and separately label how many of those activities were actual `training_session` records so the UI does not confuse a plan with completed exercise.
- User-view deployment lesson: local QA can pass while the user still sees the old cloud package. Capture a cloud screenshot before/after deployment when the complaint is "I cannot see the change"; the stale cloud profile page still showed `Oct 2023`, `4 本周训练次数`, `System OK`, and debug panels even after the local fix.
- Profile empty-state lesson: do not let first paint imply device readiness. Replace `System OK`, `患者 A`, fake M33/EMG bindings, and enabled voice/collaboration labels with authenticated backend state or clear `待登录/待同步/待邀请` language.
- Mobile full-page QA lesson: keyword scans catch obvious fake values, but `bodyStart` review catches split text such as `8` + `/12`, residual M33 "执行中" copy, and static EMG confidence that a single search term can miss. For every App page that implies live physiology or motion, read the rendered text after bridge hydration and force unknown values to `待同步/待报告/待审核`.
- Mobile bridge replacement lesson: broad text replacement can corrupt bridge-owned UI such as `[data-arm-status]`, toast, workflow, and evidence panels. When cleaning Stitch demo copy, skip bridge nodes and avoid replacing generic brand fragments like `ArmControl`; otherwise the App can show misleading states such as `待绑定后端` or `灵动康复 等待绑定 M33` even though the backend is connected.

## 2026-07-03

- Symptom: NanoPi V page could show an apparently stable end-effector box even though the recognition looked visually unreasonable.
- Root cause: the live end-effector ONNX helper was applying sigmoid to confidence values that were already raw probabilities, turning weak scores such as about `0.21` into seemingly acceptable `0.55`. It also treated YOLO `cx,cy,w,h` output in `416x416` model space as if it were platform-frame `x,y,w,h` coordinates.
- Fix: patch the NanoPi runtime to keep raw confidence by default, convert center boxes to top-left boxes, and scale `416x416` boxes back to `640x480` image coordinates before building `stereo_vision_context`. The cloud payload now correctly reports no visual lock when the current frame lacks a stable target/end-effector pair.
- Lesson: before trusting a detector in a VLA loop, verify three numbers together: confidence transform, coordinate convention (`cxcywh` vs `xywh`), and image-space scaling. A detector can appear to "work" while feeding the planner a wrong pixel delta.
- Symptom: local labeled images could show end-effector detections, but the live/platform stereo context still reported waiting or COCO/pretrained target detection state for the robot end effector.
- Root cause: the trained `end_effector` / `gripper_tip` ONNX model existed locally, but the NanoPi collector wrapper did not forward that model path into the read-only agent. The deployed V chain could therefore upload frames while never invoking the user-trained detector.
- Fix: keep the detector optional and pass `--end-effector-onnx`, confidence, NMS, and image-size options through `nanopi-rehab-arm-collect-and-upload.py` to `nanopi-rehab-arm-readonly-agent.py`. A dry-run with `D:\vla_dataset\20260627_213112\runs\end_effector_v1_cpu_416_e10\weights\best.onnx` produced real `end_effector` and `gripper_tip` detections in `stereo_context`.
- Lesson: when a model works offline but not on the platform, first verify the runtime chain is actually loading that exact model. Do not debug kernel, camera, frontend boxes, or L routing before checking the detector field and model path in the evidence payload.
- Mobile release lesson: do not publish a new APK as user-ready just because backend endpoints pass tests. The public config and authenticated `/me.mobile_readiness_guide` must keep saying `blocked` while the Android wrapper still falls back to static preview data or lacks `Authorization: Bearer {access_token}` bootstrap wiring.
- Hardware protocol lesson: missing BLE/M33/M55 packet maps are a product blocker, not a reason to invent demo behavior. The App backend should expose `awaiting_protocol` readiness and continue recording plans, diagnostics, evidence, and safety reviews without pretending it can certify motion.
- Symptom: the platform could prove `stereo_left` / `stereo_right` keyframes and a `stereo_vision_context` existed, but the Python helper payload did not preserve the physical OpenCV camera indices or flip parameters that produced those logical frames.
- Root cause: `nanopi-rehab-arm-collect-and-upload.py` accepted `--left-camera-index` / `--right-camera-index` for capture, but `nanopi-rehab-arm-readonly-agent.py` only uploaded logical `stereo_left` / `stereo_right` IDs. Physical `/dev/video*` mapping and orientation therefore lived in operator memory or C++ runtime logs rather than the durable evidence record.
- Fix: add optional camera-index and flip arguments to the read-only agent, pass them through from the collector, apply requested OpenCV flips before saving Python-captured keyframes, and record the values under `stereo_vision_context.capture_loop.camera_mapping`. Forwarded image files without explicit indices now keep `mapping_state = waiting_physical_index_confirmation`.
- Lesson: left/right image existence is not enough for stereo calibration. Every calibration sample and depth result must carry the physical camera mapping and frame orientation used at capture time; otherwise a later baseline, rectify, or hand-eye result may be numerically plausible but tied to the wrong camera.
- Scanner lesson: USB inventory alone is not a camera inventory. `lsusb` may reveal "USB Camera", but stereo bring-up needs V4L2 nodes, sysfs names, optional udev IDs, and optional `v4l2-ctl` formats. Record those under `camera_device_details` before deciding which OpenCV index is logical left/right.
- Live NanoPi mapping lesson: the real C++ loop on `NanoPi-M5` was not using OpenCV index `0/1`; it was launched as `--left /dev/video45 --right /dev/video47 --left-flip hv --right-flip hv`, and the probe argv matched `/dev/video45 /dev/video47 ... hv hv`. Treat platform `stereo_left` / `stereo_right` as correct logical channels, but record the physical device paths in `capture_loop.camera_mapping` and calibration metadata. Do not infer mapping from old Stitch static text that mentions `/dev/video0` and `/dev/video1`.
- Restart lesson: avoid `pkill -f nanopi_vla_cpp_upload_loop.py` from a shell whose own command line contains the same match string; it can kill the SSH command before restart executes. If restarting the vision upload loop, use PID-specific kill or a pattern that cannot match the current shell, then verify only one upload loop remains to avoid camera-open failures from duplicate readers.
- Calibration capture lesson: do not force stereo calibration scripts to accept only integer camera indices. On NanoPi M5 the stable, evidence-backed sources are `/dev/video45` and `/dev/video47`; calibration capture should accept device paths and write those exact sources into `capture_manifest.json`. A non-chessboard smoke capture can validate image saving, but `calibrate` must remain `waiting` when no real chessboard corners are found.
- Upload mapping lesson: the read-only collector should also accept `/dev/video*` sources, not just integer indices. When `/dev/video45` and `/dev/video47` are supplied, preserve them as `opencv_source` and leave `opencv_index` null unless the operator explicitly supplied an index. Do not silently map device paths back to `0/1`, because that recreates the original left/right ambiguity.
- Calibration scale lesson: once the printed board is confirmed as `zhang_chessboard_9x6_inner_20mm_A4.pdf`, use `--board-cols 9 --board-rows 6 --square-size-m 0.02` as the metric scale. The earlier unit-square and `baseline06m_provisional` files were useful sanity checks, but the final depth path should use `chessboard_real_20260703_01_A4_20mm_9x6.json` / `.yaml`, whose measured baseline is about `0.06276 m`. Do not keep normalizing to the user's rough `6 cm` baseline after the square size is known.
- Depth evidence lesson: loading calibration is not enough to publish depth. The agent must also have a same-label left/right detection center, positive rectified disparity, and a small rectified vertical mismatch. If the current frame is a chessboard, empty scene, or one-eye-only detection, report `stereo_depth_evidence.state=waiting_stereo_match` and keep `estimated_depth_m`, `target_3d_camera_frame`, and `target_3d_robot_frame` null.
- Live wrapper lesson: the current C++ upload wrapper has historically populated `detections.left` only. That is useful for 2D target/effector evidence, but calibrated stereo depth requires a real right-eye bbox/center from the backend/OpenCV/model path as well. Do not compute disparity from duplicated left boxes, UI overlays, or inferred right centers.
- NanoPi access note: `paramiko` may fail at SSH banner while the board is busy or the SSH service is unhealthy. Treat that as a transport issue, not permission to restart vision blindly. Do not kill/restart the live camera loop unless the user asks or SSH/service health is confirmed; continue with repo-local dry-runs and cloud payload checks.
- Real stereo depth rejection note: with the placed bottle, right-eye YOLO found the bottle body, but a lowered-threshold left-eye detection locked onto the bottle base/table reflection. The rectified vertical mismatch was about `125 px`, so the depth gate rejected the pair even though disparity was positive. This is the desired behavior: do not publish `target_3d_camera_frame` from two boxes that do not mark the same physical point. Next improvement should use a trained target detector/segmentation, a specific keypoint on the bottle, or manual paired point annotation for depth proof.
- Detector upgrade lesson: after switching the target model to `yolo11s.onnx` at `--target-conf 0.45`, the same bottle produced stable full-body left/right boxes and accepted stereo depth. This confirms the calibration/depth math was not the blocker; the previous blocker was unreliable paired detection. Keep the vertical mismatch gate because it cleanly separates a real same-object match from misleading positive disparity.
- Process hygiene reminder: a `ps | awk` cleanup pattern that includes `nanopi_vla_cpp_upload_loop.py` can match the current shell command and interrupt itself. Use `pgrep -af '^python3 /home/pi/rehab_vla/nanopi_vla_cpp_upload_loop.py'` to inspect, then kill explicit PIDs and verify exactly one wrapper remains.
- Symptom: the Stitch V page showed real `stereo_left` / `stereo_right` frames but still displayed static HUD copy such as `GRASP_BOTTLE`, `98.4%`, `CALIB_VALID`, and fake coordinates while the API payload had `detections=[]`, `stereo_calibration_id=""`, and `target_3d_camera_frame=null`.
- Root cause: the Stitch HTML contained optimistic placeholder detection/calibration text, and the React bridge only replaced a few label-neighbor values. Some nested status rows were left untouched, so the page implied a successful detection/calibration that did not exist.
- Fix: add explicit data-role slots for V page evidence values and bind them from real `stereo_vision_context`; default the static HTML itself to waiting states so slow hydration never flashes fake boxes, labels, depth, or calibration.
- Lesson: for perception evidence, image binding is not enough. Every detection label, confidence, depth, disparity, and calibration state shown beside the image must come from backend/OpenCV/model payloads or display a waiting/rejected state.
- Symptom: uncalibrated depth briefly displayed as `depth 0 m` even though the payload depth was null.
- Root cause: `Number(null)` is `0`; a generic finite-number helper treated null coordinate fields as valid zeroes.
- Fix: use a strict stereo-depth extractor that skips `null`, `undefined`, and empty strings before numeric conversion.

- Mobile history placement lesson: a global history/evidence panel can improve trust on training-library, but it makes task pages such as `设备` and `AI` feel buried. Keep detailed timeline/history on training-library/profile and leave action pages focused on their immediate CTA and blockers.
- Local static QA lesson: serving `apps/web/public` from `127.0.0.1` is useful for bridge syntax and visual checks, but authenticated cloud API calls can fail CORS. Use same-origin cloud PWA QA for `/public-config`, `/me`, AI draft generation, and plan-sync blocker validation.
- Gripper single-class ONNX lesson: the new fixed-camera `gripper` model exports as `1x5xN`, not the old two-class `1x6xN` end-effector format. If the parser still assumes two class rows, it can treat coordinate values as confidence and produce nonsense scores such as `39.x`. Parse `1x5xN` as `cx,cy,w,h,score` and normalize the class to the existing downstream `gripper_tip` evidence label.
- Live gripper generalization lesson: a validation mAP near `0.994` on the fixed-camera split does not guarantee the current live pose will detect if the gripper is on the image edge, partially occluded, or visually fragmented by dark rods/boxes. In the 2026-07-05 live NanoPi frame, the new gripper ONNX score max was only about `0.0766` while the gripper was at the right edge and partly occluded, so lowering confidence would only reintroduce weak evidence. First verify framing and collect hard examples for edge/occluded poses before relaxing the VLA quality gate.
- L/A Stitch QA server lesson: if the outer platform route shows a Next build overlay after static `/rehab-stitch/router.html` and `/rehab-stitch/planner.html` build cleanly, check for a stale local dev process before debugging the module HTML. A stale `127.0.0.1:3000` process can serve an old compiler state such as missing `app/not-found.module.css` even when `npm --workspace apps/web run build` passes.
- VLA closed-loop lesson: the A page should not independently guess whether the system is blocked by L, V, IK, MuJoCo, or M33. Use dashboard `vla_closed_loop_status` as the single evidence explanation layer, and keep camera-frame-only depth blocked from robot-frame IK until hand-eye calibration or an explicit manual/simulation target source exists.
- Cloud schema compatibility lesson: the cloud runtime can lag newer stereo payload fields such as `end_effector_object` even when it still stores `detections`. Closed-loop readiness should tolerate older payload shapes by inferring target/end-effector visibility from detection labels, while still requiring robot-frame coordinates before A/IK consumes V as a motion target.
