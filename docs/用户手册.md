## M33 Rehab Bench Current-Mode QA

Purpose:
- Validate the local MSH bench rehab path on the M33 shell before involving NanoPi, ROS2, or MuJoCo.
- Keep this classified as `bench-debug`: it may move the motor only under explicit local operator commands.

Boundary:
- MSH bench commands may bypass the NanoPi heartbeat, but they must still require fresh feedback, current limits, stop handling, and memory playback calibration.
- NanoPi/MuJoCo are not required for this local bench test. Use them later for the formal or shadow route.
- Formal product motion remains `JointTrajectory -> NanoPi -> M33 -> motor`.

Current default:
- Rehab `elbow` maps to M33 joint/motor slot `5`.
- Active, assist, and resist rehab strategies now output private-protocol current commands through `iq_ref(0x7006)`.
- Rehab mode commands may also take an explicit M33 motor/joint slot, for example `rehab active 5`, `rehab assist motor5`, or `rehab resist m33:5`.
- Follow direction defaults to `-1` for active/assist after the bench direction finding. It can be changed at runtime through `rehab cfg direction <1|-1>`.
- Assist and resist gains are runtime service parameters, so future LVGL screens can edit the same values without reimplementing shell logic.
- Memory playback still uses position control and refuses to run unless the joint is calibrated.

Power-on bench sequence:
```text
cmd_control_init
cmd_can_bitrate 1000000
cmd_motor_report 5 1
cmd_motor_fb 5
rehab status
```

Runtime tuning examples:
```text
rehab cfg direction -1
rehab cfg active_gain 1.6
rehab cfg active_max 1.0
rehab cfg assist_gain 2.5
rehab cfg assist_max 1.0
rehab cfg adaptive_enable 0
rehab cfg assist_pid_enable 0
rehab cfg resist_pid_enable 0
rehab cfg resist_gain 3.0
rehab cfg resist_max 1.0
rehab cfg
```

Parameter meaning:
- `direction`: active/assist current direction. Use `1` or `-1`.
- `active_gain`: active-follow current gain in A/Nm after torque deadband.
- `active_min`: minimum active-follow current once torque or velocity leaves the deadband.
- `active_max`: active-follow current cap.
- `assist_gain`: assist current gain in A/Nm while hysteresis is engaged.
- `assist_max`: assist current cap.
- `adaptive_enable`: optional adaptive assist gain. Default is `0`.
- `adaptive_base`: base gain used when adaptive assist is enabled.
- `adaptive_load`: added gain per measured Nm while adaptive assist is enabled.
- `adaptive_max`: maximum effective adaptive gain. This is still limited by `assist_max` current.
- `adaptive_step`: maximum effective-gain change per service step.
- `assist_pid_enable`: enables the bounded adaptive PID trim in assist mode.
- `resist_pid_enable`: enables the bounded adaptive PID trim in resist mode.
- `pid_load_low` / `pid_load_high`: torque range used to map load into `0..1`.
- `pid_speed_low` / `pid_speed_high`: velocity range used to map speed into `0..1`.
- `assist_pid_target`: assist-mode residual torque target in Nm.
- `assist_pid_kp_base` / `assist_pid_kp_load` / `assist_pid_kp_speed`: scheduled assist PID Kp terms.
- `assist_pid_ki_base` / `assist_pid_ki_load` / `assist_pid_ki_speed_reduce`: scheduled assist PID Ki terms.
- `assist_pid_kd_base` / `assist_pid_kd_speed`: scheduled assist PID Kd terms.
- `assist_pid_i_limit`: assist PID integral-state limit.
- `assist_pid_trim`: maximum assist PID current trim in amps.
- `resist_pid_target`: resist-mode target velocity in rad/s.
- `resist_pid_kp_base` / `resist_pid_kp_load` / `resist_pid_kp_speed`: scheduled resist PID Kp terms.
- `resist_pid_ki_base` / `resist_pid_ki_load` / `resist_pid_ki_speed_reduce`: scheduled resist PID Ki terms.
- `resist_pid_kd_base` / `resist_pid_kd_speed`: scheduled resist PID Kd terms.
- `resist_pid_i_limit`: resist PID integral-state limit.
- `resist_pid_trim`: maximum resist PID current trim in amps.
- `resist_gain`: resist damping current gain in A/(rad/s).
- `resist_max`: resist current cap.
- `resist_dir`: resistance direction sign if the damping direction is wrong.

Expected checks:
- `rehab status` should show `m33_joint=5`.
- `fresh=1` is required before active/assist/resist should command current.
- `torque_x1000`, `vel_x1000`, `current_x1000`, `limit_x1000`, and `gain_x1000` are scaled by 1000. For example, `current_x1000=500` means `0.500 A`.
- `pid_kp_x1000`, `pid_ki_x1000`, `pid_kd_x1000`, `pid_load_x1000`, `pid_speed_x1000`, `pid_err_x1000`, and `pid_trim_x1000` show the scheduled PID state when assist/resist PID is enabled.
- `sat=1` means the strategy output reached the configured current cap. If force still feels too small while `sat=1`, raising gain alone will not help; raise the relevant `*_max` only within the bench safety limit.
- `rehab stop` and `cmd_motor_stop 5 0` must immediately stop the bench command path.
- If feedback becomes stale, rehab service should stop the motor and latch a fault/detail state until the mode is reset.

Adaptive assist bench example after direction and fixed gain are confirmed:
```text
rehab cfg adaptive_base 2.0
rehab cfg adaptive_load 0.5
rehab cfg adaptive_max 4.0
rehab cfg adaptive_step 0.1
rehab cfg adaptive_enable 1
rehab assist motor5
rehab status
rehab stop
```

Watch `gain_x1000` while changing load. It should change gradually and never
let `current_x1000` exceed `limit_x1000`.

Adaptive PID bench example after fixed assist/resist signs are confirmed:
```text
rehab cfg pid_load_low 0.2
rehab cfg pid_load_high 1.5
rehab cfg pid_speed_low 0.03
rehab cfg pid_speed_high 0.20

rehab cfg assist_pid_target 0.25
rehab cfg assist_pid_kp_base 0.10
rehab cfg assist_pid_kp_load 0.35
rehab cfg assist_pid_ki_base 0.02
rehab cfg assist_pid_ki_load 0.04
rehab cfg assist_pid_kd_speed 0.01
rehab cfg assist_pid_trim 0.30
rehab cfg assist_pid_enable 1
rehab assist motor5
rehab status
rehab stop

rehab cfg resist_pid_target 0.05
rehab cfg resist_pid_kp_base 0.10
rehab cfg resist_pid_kp_speed 0.50
rehab cfg resist_pid_kd_speed 0.02
rehab cfg resist_pid_trim 0.30
rehab cfg resist_pid_enable 1
rehab resist motor5
rehab status
rehab stop
```

If `pid_load_x1000` or `pid_speed_x1000` stays at `0`, the scheduler is not
seeing enough load/speed to raise gains. If it stays at `1000`, the chosen range
is too low or the physical motion is already outside the tuning window.

Do not use memory playback until calibration is explicitly valid:
```text
cmd_m33_joint_calib 5
rehab play
```

Expected safe result while uncalibrated:
```text
calibrated=0
rehab play ret=<negative>
```


## CM55 Wi-Fi Provisioning QA

Purpose:
- Configure CM55 Wi-Fi for XiaoZhi/WebSocket and later platform communication without hardcoding credentials in firmware.
- Keep user-facing provisioning reusable from three entries: LVGL touchscreen, CM55 shell, and M33/App IPC.

Safety and credential boundary:
- `/flash/rehab_wifi.cfg` stores only Wi-Fi SSID/password and auto-connect state.
- Do not store platform scoped relay tokens in the Wi-Fi config file.
- Do not store vendor LLM API keys on M33, CM55, NanoPi, or in Git.

Touchscreen path after flashing CM55:
1. Power the Infineon board and wait for the LVGL screen. New M55 builds should start the LVGL thread automatically.
2. Confirm the boot log contains:
   ```text
   [m55] starting LVGL thread
   [m55] LVGL thread init ret=0
   ```
3. If an older image is still flashed or the screen stays blank, try starting the GUI thread manually:
   ```text
   thread_init
   ```
4. If the LVGL thread is running but the Wi-Fi panel is not visible, open it manually:
   ```text
   rehab_wifi_panel_cmd
   ```
5. If `thread_init` succeeds but the display is still blank, run `lcd_test` to check the LCD driver/hardware separately.
6. Press `Scan`.
7. Wait up to about five seconds. Nearby networks should appear in the list with SSID, RSSI, security mode, and channel.
8. Tap the target Wi-Fi network. The SSID field is filled automatically.
9. Tap the password field and enter the Wi-Fi password on the LVGL keyboard.
10. Keep `Auto connect` checked for product-like boot behavior.
11. Press `Connect` to save and connect immediately, or press `Save` if you only want to persist credentials.
12. Press `Diag` after a few seconds and verify the screen shows a netdev, WLAN ready/connected state, and `saved:1 auto:1 storage:0`.

Touchscreen notes:
- If the target network is hidden or not listed, manually type the SSID field and continue from the password step.
- `Scan` refreshes the list automatically for a short period after tapping it. `Diag` also refreshes the cached list/status.
- The scan list is a convenience UI over the shared CM55 Wi-Fi config service; it does not store platform relay tokens or vendor API keys.

CM55 shell path, if the interactive shell is on CM55:
```text
m55_wifi_ssid <ssid>
m55_wifi_password <password>
m55_wifi_auto 1
m55_wifi_save
m55_wifi_connect
m55_wifi_status
```

M33 shell path, when the visible shell is M33:
```text
m55qa_wifi_ssid <ssid>
m55qa_wifi_password <password>
m55qa_wifi_auto 1
m55qa_wifi_save
m55qa_wifi_connect
m55qa_wifi_diag
m55qa_whd_diag
m55qa_status
```

Forget credentials:
```text
m55qa_wifi_forget
```
or on CM55:
```text
m55_wifi_forget
```

Expected pass criteria:
- `m55qa_status` prints `ipc_ready=1`.
- The netdev line eventually shows `saved=1 auto=1 storage=0`.
- `wlan=1` or `ready=1` appears after connection succeeds.
- IP/GW/mask/DNS fields are no longer all zero after DHCP succeeds.

If Wi-Fi was previously working but now does not enumerate:
```text
m55qa_whd_diag
m55qa_wifi_diag
m55qa_wifi_scan
m55qa_status
```

CM55 local scan-list fallback when the shell is on CM55:
```text
m55_wifi_scan
# wait 3-5 seconds
m55_wifi_aps
```

Expected `m55_wifi_aps` output:
```text
[wifi_config] cached_ap_count=<n>
[wifi_config] ap[0] ssid="<ssid>" rssi=-45 security=WPA2 channel=6 bssid=xx:xx:xx:xx:xx:xx
```

Use this when the LVGL screen or touch panel is not yet trusted. If `m55_wifi_aps` lists nearby APs but LVGL does not, debug the LVGL list/touch layer. If both show no APs, debug WHD/WLAN scan, antenna, power, or RF environment first.

Useful interpretation:
- `whd_stage=5` with a negative result usually means the SDIO/WHD probe path timed out; check reset timing, WLAN power, and WHD resource partitions before changing XiaoZhi logic.
- `storage` is the result of the last Wi-Fi config file operation. `0` means saved/loaded successfully; negative values mean the `/flash` filesystem or config file was not available.

XiaoZhi audio counters to watch:
- `xz_cur` = current in-flight audio bytes/chunks being accumulated or sent.
- `xz_last` = last completed listen window sent to the platform.
- `xz_fail` = WebSocket binary send failures.
- `xz_rx` = platform text/binary reply counters.
- `frame_len` = current partial frame bytes waiting to be sent.
- `tts_fwd` = TTS audio chunks/bytes forwarded from CM55 to M33 playback.
- `tts_fail` = TTS forwarding failures or pending-queue drops.
- `pcm_reject` = downlink binary packets rejected as non-PCM before speaker playback.

For the current board contract, the CM55 follows the official XiaoZhi path and declares `audio_params.format=opus`. A normal 60 ms upstream frame is an Opus packet, so several seconds of speech should usually total tens of KB rather than hundreds of KB. Do not expect a local 4-byte binary wrapper in protocol version 1.

Optional token loader from Windows PowerShell:


This default test opens the same platform WebSocket, sends the XiaoZhi `hello`, sends `listen start`, streams synthetic 60 ms `pcm_s16le` frames, then sends `listen stop`. Notes:

- `xz_ws=0` means the XiaoZhi WebSocket is not connected.
- `xz_listening=1` means CM55 is actively streaming the post-wake utterance to the platform.
- `audio_playback_tone_cmd 500` is a low-level speaker/I2S check and sounds like a test tone, not a user-facing voice.
- `audio_playback_voice_cmd` plays a softer voice-like local sample for speaker listening QA without relying on platform TTS.
- COM4 is the M33 shell on this bench. Use `m55qa_status` to inspect M55 XiaoZhi/LVGL state via IPC; M55-only finsh commands are not expected to be directly callable from COM4.
- If LCD stays in 鈥滄鍦ㄦ€濊€冣€? wait at least 20 seconds after capture/listen stop. The M55 UI state should now return to `鍦ㄧ嚎寰呭敜閱抈 with a retry hint when the platform does not reply.
- `latest_pcm_len=320` in `m55qa_status` can still be normal because it is the local mic driver chunk length. The cloud WebSocket frame is separately reframed and Opus-encoded to 60 ms packets in CM55.
- A matching platform token should make `xz_token=1`. A successful WebSocket connection should make `xz_ws=1 xz_stage=70 xz_errno=0`.
- During a good capture, `frames`/`pcm_seq` should increase and `probe_lwip=<chunks>/<bytes>` should become nonzero after capture stops. If those pass but no `stt/tts` arrives, debug platform ASR/codec handling rather than WiFi.
- During a good TTS downlink, `xz_rx` binary count should increase, `tts_fwd=<chunks>/<bytes>` should become nonzero, and M33 should log `tts audio rx` / `tts audio write`. If `xz_rx` binary increases but `tts_fwd` stays zero, inspect server audio format and CM55 Opus/PCM classification.
- The firmware default endpoint already targets `/xiaozhi/ws?robot_id=rehab-arm-alpha`; use `m55qa_xz_url <ws://...>` only when overriding it for diagnostics.
- `cmd=1004` begins a chunked XiaoZhi platform-token update on CM55.
- `cmd=1005` appends one token chunk; keep each chunk short enough for the embedded shell line, usually 48 to 60 characters.

- `m55qa_xz_token_clear` clears the CM55 platform token and should be used after dummy-token tests.
- On this bench the safe token refresh flow is to use `tools/load_xiaozhi_token.ps1` from the M33 repo against COM4, not to paste the full token manually into the shell.
- Current validation note: the loader can write the token and show ACKs, but `m55qa_status` may still stay at `xz_ws=0 xz_errno=-403` if the platform rejects the token.

- `cmd=3` is start wake listening.
- `cmd=1` is start capture.
- `cmd=4` is stop wake listening.
- Repeated `[drv_can] direct tx pending ...` lines indicate CAN/motor bus acknowledgement issues and do not by themselves mean CM55 IPC failed.
## M33 BLE restoration bench check (2026-07-11)

1. Build or flash `build/rtthread.hex` from the M33 project.
2. Confirm the serial log reaches `[bt] Bluetooth stack ready` and advertising starts.
3. In nRF Connect, connect to `OpenClaw-NUS` and enable notifications on UUID `6E400003-B5A3-F393-E0A9-E50E24DCCA9E`.
4. Write `heartbeat` and then `stream:on` to UUID `6E400002-B5A3-F393-E0A9-E50E24DCCA9E`.
5. Require an `OK:` response and growing telemetry notifications.
6. While BLE remains connected, run the existing M33/M55 status command and CAN status/diagnostic command used on the bench. Their counters must keep advancing without stale IPC, CAN errors, reset loops, or control-loop starvation.
7. Use `stream:off` and disconnect; advertising should become available again.

Do not issue movement commands during the first BLE restoration check. Validate transport and concurrent subsystem health before any motion test.
