## 1. 新的总方向

这个项目未来不是只卖软件，也不是只卖一个网页游戏平台。

新的总方向是：

**以网页游戏为统一前台，承载教育模式与开发模式；以 NanoPi 等 Linux 开发板作为商业交付载体之一；以账号服务、NPC 协作、开发/教学闭环作为核心能力。**

关键点：
1. 前台本质上是同一套网页游戏产品。
2. 这套网页游戏既可以运行在 NanoPi 上，也可以运行在普通电脑浏览器上。
3. NanoPi 不是唯一运行环境，而是优先商业化交付形态之一。
4. 电脑浏览器端仍然重要，用于开发、教学、运营、验收、远程协作和备用登录入口。
5. 当前正在建设的 AI 协作平台，不应被废弃，而应被视为未来“开发模式”的内核。

## 2. 统一前台：同一套网页游戏，双载体运行

必须明确记住：

**不是“NanoPi 专属游戏”和“电脑后台系统”两套东西，而是同一套网页游戏前台，根据载体不同切换运行形态。**

两种主要载体：

### 2.1 NanoPi / Linux 开发板载体

定位：
- 面向商业交付
- 面向硬件整机体验
- 面向“开机即进产品”的专用终端体验

要求：
- 上电直接进入登录界面或受控首页
- 尽量不暴露 Linux 桌面、终端、文件系统和系统后台
- 用户感觉自己在使用一台专用学习/开发设备，而不是普通 Linux 电脑

### 2.2 电脑浏览器载体

定位：
- 面向开发者、运营者、教师、进阶用户
- 面向日常持续使用
- 面向没有随身带 NanoPi 时的继续工作入口

要求：
- 与 NanoPi 版本共用同一套账号体系和核心产品逻辑
- 允许在电脑上继续进入教育模式或开发模式
- 保持网页游戏世界观和 NPC 交互逻辑一致

结论：
- NanoPi 是交付载体，不是产品边界。
- 产品边界是“网页游戏化的开发/教育系统”。

## 3. 账号体系

账号体系要从“公开注册”改成“运营发放”。

明确要求：
1. 保留登录页。
2. 不提供公开注册。
3. 账号密码由运营侧发放。
4. 一个 NanoPi 硬件默认对应一套主账号凭据或一套出厂绑定身份。
5. 后续需要支持一批量下发账号，当前口头目标大约是 500 个账号密码。
6. 但这轮先记方向，不在本轮随便生成真实商业账号数据。

额外约束：
- “一块板对应一套账号”是交付策略，不代表产品只能在板子上使用。
- 同一套产品仍应支持在电脑浏览器访问，具体是否允许同账号跨设备登录，后续做成可配置策略。

## 4. 两大模式

## 4.1 教育模式

教育模式是小白用户的主入口之一。

目标：
- 通过与 NPC 交互接任务
- 完成任务
- 学会基础开发流程
- 形成从器件、接线、环境、代码、调试到完成作品的一条龙体验

教育模式应强调：
1. 用户先接任务，再在任务里学。
2. NPC 负责发任务、收任务、解释下一步。
3. 平台负责把买器件、接线、环境准备、代码生成、调试串成闭环。
4. 不要把教育模式做成复杂后台管理页。
5. 点击“教育模式”后应直接进入教育模式对应的游戏界面。

## 4.2 开发模式

开发模式继承当前 AI 协作平台主线。

目标：
1. NPC 作为固定长期身份，负责某一块工作。
2. NPC 可以切换电脑、切换线程、切换模型。
3. 但 NPC 自己的固定知识库必须保留，不能因为执行壳变化而断档。
4. 平台继续推动 requirement -> minimal ack -> final reply -> next requirement。
5. 后续扩展到硬件调试、编译、烧录、串口、仿真、资料整理等场景。

结论：
- 当前平台主线，不是要被推翻。
- 它是未来开发模式的核心引擎。

## 5. NanoPi 终端形态

NanoPi 交付版本需要做成受控终端，而不是普通 Linux 电脑。

硬要求：
1. 上电就是登录界面或首页。
2. 不让用户碰系统桌面。
3. 不暴露 shell。
4. 不暴露文件管理器。
5. 不让用户直接操作预装开发环境。
6. 前台看到的是网页游戏登录壳和游戏世界。
7. 后台编译、烧录、串口、AI 调用都通过受控服务暴露。

但必须同时记住：
- 这是 NanoPi 部署形态要求。
- 不是说整套产品只能运行在 NanoPi 上。

## 6. 技术与商业载体

未来卖的不是单点软件，而是一整套组合：
1. NanoPi 或类似 Linux 开发板
2. 预配置系统镜像
3. 设备绑定账号体系
4. 可在 NanoPi 和电脑浏览器运行的网页游戏前台
5. 教育模式任务流
6. 开发模式 NPC 协作流
7. 后台 AI / 编译 / 烧录 / 串口服务

商业上更接近：

**硬件载体 + 账号服务 + 网页游戏前台 + 教学/开发平台**

## 7. AI 厂商路线

平台未来一定不能只接 Codex。

长期方向：
1. Codex
2. Claude
3. 千问
4. GLM
5. OpenClaw
6. 其他可接 CLI / API 的模型提供方

当前结论：
- 这不是本轮立刻开干的最高优先级。
- 先把平台骨架、NPC 体系、登录形态、模式切分、设备边界和开发模式内核打稳。
- 再按 provider abstraction 扩模型供应商。

## 8. 前端长期方向

未来前端因为版权问题，要逐步切到我们自己开发的 Unity 3D 游戏界面。

当前判断：
1. 这是长期方向。
2. 现在不优先做。
3. 当前先把产品路线、双模式、账号体系、设备隔离和协作内核打稳。

## 9. 对当前代码主线的影响

从现在开始，当前平台主线应被视为：

**未来开发模式的内核。**

要继续保留并强化：
1. 真实线程识别
2. 真实 NPC 席位
3. requirement 自动推进
4. minimal ack / final reply 闭环
5. NPC 固定知识库
6. 平台持续发指令、线程持续回执
7. 农场/游戏界面承载协作信息

同时开始逐步对齐未来产品：
1. 登录前置
2. 无公开注册
3. 账号与设备绑定
4. 教育模式入口
5. 开发模式入口
6. NanoPi 受控终端部署
7. 同一套网页游戏支持电脑浏览器运行

## 10. 近期优先级

### P0：必须先记住并持续对齐
1. 项目未来不是纯软件平台，而是 NanoPi 等硬件承载的产品方向。
2. 必须保留教育模式和开发模式双入口。
3. 必须取消公开注册，改成运营发号。
4. 必须让 NanoPi 上电后进入受控登录或首页。
5. 必须继续保留并强化 NPC 固定身份与协作闭环。
6. 必须把“同一套网页游戏既能在 NanoPi 跑，也能在电脑浏览器跑”作为硬约束。

### P1：最早落地的产品骨架
1. 登录页改成无注册版。
2. 设备账号模型设计。
3. 模式选择页：教育模式 / 开发模式。
4. 教育模式基础任务流。
5. 开发模式复用当前平台内核。

### P2：设备化落地
1. NanoPi kiosk 启动链。
2. 前后端隔离。
3. 受控 API 能力边界。
4. 编译/烧录/串口服务进程隔离。

### P3：多模型扩展
1. Codex 稳定化。
2. Claude 接入。
3. 千问 / GLM / OpenClaw 等接入。

### P4：长期演进
1. Unity 3D 自研前端。
2. 教育内容体系扩展。
3. 硬件 SKU 与账号服务商业化。

## 11. 给后续线程的硬约束

后续任何线程接手时，都必须遵守：
1. 不要再把项目理解成纯软件 AI 协作平台。
2. 不要把 NanoPi 误解成唯一运行环境。
3. 不要把电脑浏览器入口做没。
4. 不要重新加入公开注册。
5. 不要让 NanoPi 退化成“用户可直接碰 Linux 系统”。
6. 不要把教育模式和开发模式混成一个重后台入口。
7. 当前农场/游戏 UI 的工作，未来是为了服务这两个模式，不是脱离产品路线单独美化。

## 12. 当前轮结论

这轮最重要的不是实现多少，而是把总方向锁死：

**未来总方向已经明确切换为：同一套网页游戏前台 + NanoPi 硬件交付形态 + 电脑浏览器运行入口 + 教育模式 + 开发模式 + NPC 协作开发内核。**

后续所有平台工作，都必须围绕这个方向收敛。
