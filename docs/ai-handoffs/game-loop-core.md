## 负责范围

只负责真正可玩的经营闭环，不负责 HUD 美术、Git 协议、联机架构。
后续一切实现统一收敛到 `projects/[id]` 主线，不再继续扩第二套基地页或并行壳层。
后续所有开发统一以当前搬进来的开源农场游戏为唯一主世界、唯一空间基底、唯一视觉母体；所有经营主循环、建筑、HUD、任务链、AI / 电脑 / 审批 / 交付，都只能基于这张游戏地图适配实现，不能脱离这张地图另起炉灶。
后续主循环验证与截图也必须落在这张游戏地图的真实场景里，不再制作脱离地图的抽象验证页。

本轮重点：

- 前 5 分钟循环
- 中期成长循环
- 建筑与产线解锁顺序
- 每日操作节奏
- 任务田块、AI 养成、电脑产线、审批门槛、交付链闭环

## 已完成

1. 把农场原型从“独立按钮”改成可走通的闭环：`收需求 -> 播种任务苗 -> 照料并派 AI -> 开电脑产线 -> 审批放行 -> 收割工程包 -> 交付`
2. 新增经营状态模型 `loopState`，记录天数、阶段、交付次数、解锁进度、每日节奏和库存
3. 重写任务田块语义，让田块真正承载需求播种、AI 接手、电脑执行、审批、收获
4. 重写 AI / 电脑 / 田块 / 任务卡片文案，让状态能反映“谁在做什么、卡在哪一环”
5. 在项目总览抽屉中加入：
   - 前 5 分钟循环
   - 当前节奏
   - 中期成长
   - 当前卡点
   - 资源取舍
6. 加入中期解锁顺序：
   - 1 次交付：第 2 块田
   - 2 次交付：第 2 位 AI
   - 3 次交付：第 2 台电脑
   - 4 次交付：第 4 块田与更高产线等级
7. 为中期成长补上真实压力：
   - 多田块并行会争抢 AI 口粮、算力、审批票、士气
   - 高风险线和中期产线会提高算力与审批成本
   - 当日交付目标未达成会累计压力并压低士气
8. 把这些资源与压力接进真实动作链，而不是只停留在说明板文案
9. 删除这次收尾过程中新增/使用的临时验证开关：
   - 去掉页面里的 `autostart` / `debug` 查询开关
   - 验证完成后删除临时验证页
