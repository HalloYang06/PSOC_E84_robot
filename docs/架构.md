## 当前总链路

```text
M33 CAN 总线采集原始传感/电机数据
  -> M55 小模型推理和语音转文字
  -> M55 编号/结果码/置信度回到 M33
  -> M33 绑定时间戳、安全状态和输出端关节状态
  -> NanoPi 通过 CAN/ROS2 汇总并解析编号语义
  -> NanoPi 上传服务器
  -> 服务器融合语音、摄像头、输出端 joint、电机诊断、小模型结果、profile/限位
  -> VLA 输出高层任务、分段目标或轨迹候选
  -> NanoPi/仿真主机验证或转换为 JointTrajectory
  -> M33 最终安全裁决
  -> 电机
```

仿真主机链路：

```text
Linux 仿真主机 <-> Wi-Fi/LAN ROS2 DDS <-> NanoPi <-> CAN <-> M33
```

无线 ROS2 只用于状态同步、可视化、MuJoCo/RViz、dry-run、规划和数据采集；急停、力矩/电流内环、fresh 判定和高频助力闭环必须留在 M33 本地安全路径。

## 当前职责分工

| 模块 | 负责什么 | 不负责什么 |
|---|---|---|
| M33 | CAN 原始电机反馈和传感汇总、M55 结果接收、时间戳和安全状态绑定、限位/限速/限流/急停、最终运动裁决、电机控制 | 不运行 VLA，不把安全权交给外部 |
| M55 | 4 路 EMG/IMU 等小模型推理、意图/疲劳/共收缩/异常建议、语音采集和语音转文字 | 不直接控制电机，不放宽 M33 限位，不成为运动许可 |
| NanoPi | ROS2 主控、M33 CAN bridge、摄像头采集、M55/M33 结果编号语义解析、服务器上传、接收 VLA 高层任务并转成可审核轨迹候选 | 不绕过 M33 直控电机，不把 `bench_armed` 当正式穿戴许可 |
| Linux 仿真主机 | MuJoCo/RViz/运动规划、无线 ROS2 接 NanoPi、仿真验证、轨迹候选生成、rosbag/JSONL 数据和标注 | 不做真机实时安全闭环，不直接发 CAN |
| 服务器/总控台 | 汇聚语音、摄像头、输出端 joint、电机诊断、M55 小模型结果、profile/限位、历史数据，运行 VLA 和数据/模型管理 | 不做高频实时控制，不直接发 CAN，不绕过 M33 |

## 关键安全边界

- M33 是最终安全责任核心。
- VLA、服务器、NanoPi、Linux 仿真主机、App、M55 都只能提出请求、预测或建议。
- VLA 不能输出 CAN、电流、力矩、速度或裸电机位置。
- VLA 输出必须是高层任务、分段目标或可验证轨迹候选。
- NanoPi/仿真主机生成的 `JointTrajectory` 仍必须由 M33 检查限位、限速、力矩/电流、急停、通信时效和电机反馈后才能执行。
- 因齿轮、同步轮、减速器、连杆或推杆存在，`motor_id` 不等于机器人/人体 `joint`。上层默认使用经过传动比、方向、零点、限位和回差说明换算后的输出端 joint 状态；原始电机轴数据只作为诊断字段保留。

## 后续 AI 必须遵守

后续任何 AI、脚本或开发者进入本仓库时，必须按以下顺序工作：

1. 先读 `README.md` 和 `docs/REHAB_ARM_SYSTEM_ARCHITECTURE.md`，再读具体任务相关文档。
2. 如果任务涉及 MuJoCo、URDF、VLA、仿真主机或无线 ROS2，必须同时读 `docs/MUJOCO_URDF_GAP_AND_STEP_GUIDE.md` 和 `docs/SIM_HOST_NANOPI_NETWORK_GUIDE.md`。
3. 如果任务涉及患者 ROM、限位、辅助等级、模型阈值或 App/平台配置，必须读 `docs/PATIENT_DEVICE_PROFILE_PROTOCOL_V1.md`。
4. 如果任务涉及 M33/NanoPi CAN 协议，必须读 `docs/PSOC_CAN_PROTOCOL_V1.md`、`docs/MOTOR_PROTOCOLS.md` 和 `docs/M33_SAFETY_INPUT_MAPPING.md`。
5. 不允许复活旧 `CAN Node <-> WebSocket <-> 服务器` 直控思路作为正式路径。
6. 不允许让服务器、VLA、App、Linux 仿真主机或 NanoPi 绕过 M33 直接控制电机。
7. 不允许把 `bench_armed`、`state=ok`、stale motor status 或 VLA 置信度当作正式运动许可。
8. 不允许把 `motor_id` 当作 `joint`；必须保留传动映射、输出端 joint 状态和原始电机诊断的分层。
9. 任何真实运动前必须先 dry-run，并确认 M33 本地安全状态、fresh 电机反馈、限位、急停和现场人工安全条件。
10. 每次改变架构、接口、安全边界或任务流程后，必须更新 `docs/PROJECT_PROGRESS.md`，必要时同步 `docs/TROUBLESHOOTING_AND_LESSONS.md`。

## 旧草案为什么不再采用

旧方案中的通用 `CAN Node <-> Bridge Node <-> WebSocket` 思路可以作为历史参考，但不能作为当前实现依据，原因是：

- 旧文档没有 M33 最终安全裁决边界。
- 旧文档容易让服务器/WebSocket 被误解为可直接发 CAN。
- 旧文档没有区分 `motor_id`、电机轴角和输出端 `joint`。
- 旧文档没有纳入 M55 小模型、语音、VLA、患者 profile、MuJoCo 和无线 ROS2 的安全分工。
- 当前 CAN 基准、ROS topic、M33 协议和正式链路已在新文档中重新定义。

后续如需修改架构，请优先更新 `docs/REHAB_ARM_SYSTEM_ARCHITECTURE.md`，再同步 README 和相关接口文档。
