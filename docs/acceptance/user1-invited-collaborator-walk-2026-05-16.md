## P0 问题

### P0-1: 协作者主页面信息过载，第一屏不像普通用户入口

受邀协作者进入项目后，第一屏同时暴露：

- API PID
- 本机端口
- Runner 心跳
- 线程 watcher 上手说明
- 工位/NPC/电脑/Skill
- 待审、待收口、异常
- token 消耗
- 协作消息数
- 工位归属和工位长缺失
- Watch 命令复制

这些信息对开发者有用，但对新协作者来说过重。用户最先需要知道的是：

1. 我现在处于什么角色。
2. 我能做什么。
3. 哪些事情需要我处理。
4. 哪些动作危险，不能随便点。
5. 下一步最推荐点哪里。

建议：

- 给协作者默认一个“项目状态摘要”视图。
- 把 API PID、本机端口、Watcher 命令等工程信息折叠到“高级诊断”。
- 第一屏只保留角色、当前任务、待处理事项、可安全进入的工作台。

### P0-2: 复制功能在当前 HTTP 环境失败

点击“复制 Claude Code 提示词”后页面显示：

```text
复制失败：Cannot read properties of undefined (reading 'writeText')
```

这说明当前环境下 `navigator.clipboard.writeText` 不可用或未做兜底。HTTP 部署下，剪贴板和摄像头等浏览器能力通常会受限。本轮还捕获到媒体能力警告：

```text
navigator.mediaDevices not supported by this browser. Webcam access will not be available.
Try hosting the page over HTTPS, because some browsers disable webcam access when insecure HTTP is being used.
```

影响：

- 复制提示词、Watch 命令、接入命令会失败。
- 机器人/数据采集场景如果涉及摄像头，会不可用。
- 普通用户不知道是自己点错、浏览器限制，还是系统坏了。

建议：

- 复制失败时展示可手动选中的文本。
- 提示“当前 HTTP 环境不支持自动复制，请手动复制”。
- 生产/演示环境尽快上 HTTPS。
- 对所有复制按钮做统一兜底。

### P0-3: 核心状态互相打架，用户无法判断系统是否可用

NPC 工作台和主页面同时出现这些状态：

- `6/6 线程`
- `线程就绪 0`
- `NPC 自动化 6`
- `待收口 16`
- `当前异常 6`
- `Boss 线程未绑定`
- `桌面 3/6`
- `电脑 1/5 Runner 心跳`

工程师可以猜测含义，但普通用户会困惑：到底是 6 个线程可用，还是 0 个就绪？自动化已开是否代表能工作？异常 6 是否阻塞当前任务？

建议：

- 增加一个统一状态结论，例如：
  - `可继续协作`
  - `部分可用，缺工位归属`
  - `需要先处理待收口`
  - `Runner 离线，无法接单`
- 每个红/黄状态旁边给出“为什么”和“下一步”。
- 合并相互关联的线程指标，避免同时出现多个看似冲突的数字。

### P0-4: 真实入口存在 404

直接访问以下路由返回 404：

- `http://106.55.62.122:3001/projects/72a1cb1d-d8a8-422f-8d87-4ed071f71dbe/robot-site`
- `http://106.55.62.122:3001/projects/72a1cb1d-d8a8-422f-8d87-4ed071f71dbe/skills`

页面上存在“机器人现场”“能力工坊/能力包仓库”等入口，但真实路由命名和可访问路径不一致。用户从刷新、收藏、外部链接或历史记录进入时可能迷路。

建议：

- 为旧路径加重定向。
- 统一 `robot-site`、`robotics`、`能力工坊`、`能力包仓库`、`skill-forge`、`skills` 的命名和路由。
- 404 页可以保留，但应提供明确“去当前真实入口”的按钮，并自动带回原项目。

## P1 问题

### P1-1: 项目列表中按钮和链接语义不统一

`进入项目主页面` 视觉上像按钮，但 DOM 中是链接。项目列表其他入口多为按钮。人类能点，但交互语义不统一，也影响可访问性和自动化测试。

建议：

- 视觉按钮可以保留，但语义上统一为链接或按钮。
- 如果是导航，用 `<a>` 并明确 hover/焦点状态。
- 如果是动作，用 `<button>` 并给提交反馈。

### P1-2: 项目列表邀请能力对协作者不够直观

协作者账号进入“邀请”页签后，页面提示“只有项目负责人、管理员或维护者可以邀请”，但表单仍显示在可操作区域中，且项目下拉只出现自己有权限邀请的测试项目。

优点是权限隔离看起来有效。问题是用户容易误以为自己可以邀请医疗项目成员，或不知道为什么医疗项目不在下拉里。

建议：

- 在“邀请”页签顶部显示“你当前只能邀请这些项目”。
- 对协作者不可邀请的项目给出只读说明，而不是完全消失。

### P1-3: 电脑接入信息中英混杂，行动判断成本高

电脑接入面板中有：

```text
The runner is actively heartbeating and can poll platform work.
The runner is not online, so queued platform work will not be picked up.
Codex Desktop UI automation
106.55.62.122 / Ubuntu 24.04
```

用户能看出在线/离线，但不知道是否需要处理、谁处理、会不会影响当前项目。

建议：

- 统一中文状态：
  - `可接单`
  - `离线，无法接单`
  - `最近心跳 14 秒前`
  - `需要在对应电脑启动 Runner`
- 给每台电脑一个“下一步”提示。
- 把英文原始说明放进诊断详情。

### P1-4: 协作消息池需要更强的“待我处理”入口

协作消息面板显示：

- 85 条消息
- 最终回复 4 条
- 2 条需人审
- 待收口 16
- 最新消息列表
- AI 协作链路说明

信息很完整，但用户最关心的是“现在要我做什么”。目前“需要人工处理”夹在信息流中，不够突出。

建议：

- 顶部固定一个“待我处理”区域。
- 明确区分：
  - 当前必须处理
  - 可稍后查看
  - 历史归档
- 对每条待处理事项给出推荐动作：催办、延长等待、手动收口、去 NPC 工作台。

### P1-5: Git 回退安全边界较好，但动作类型仍会吓到协作者

Git 回退面板已经强调“不直接 reset”“必须人审”。这是优点。但 `创建检查点`、`差异预览` 标成“已接真实表单”，协作者不容易判断是否会修改仓库。

建议：

- 将动作分成：
  - `只读查看`
  - `登记请求`
  - `会改项目状态`
  - `危险动作，需 owner 确认`
- 对协作者角色默认只允许打开“差异预览”和“申请说明”，禁止提交。

### P1-6: 硬件/设备调试入口边界还不够强

设备调试面板中 `扫描 USB`、`配置收发格式`、`波形视图` 都显示为“已接真实表单”。在医疗机械臂项目中，用户需要更明确知道：

- 这是否会访问真实设备。
- 是否只读。
- 是否会发指令。
- 是否需要现场负责人确认。

建议：

- “扫描 USB”标为只读。
- “配置收发格式”标为项目配置变更。
- “波形视图”标为只读展示。
- 真机动作入口必须单独用红色或强审标识隔离。

### P1-7: 公司层空状态解释不足

公司层显示 0 位工位长，但项目已有 4 个逻辑工位和 6 个 NPC。用户会疑惑为什么公司层没有任何人。

建议：

- 空状态直接提示：
  - `已有 4 个工位，但都未指定工位长`
  - `先回主页面将 NPC 分配到工位，再指定工位长`
- 提供直达按钮“去分配 NPC 和工位长”。

## P2 问题

### P2-1: Next 路由偶发 RSC payload 获取失败

本轮捕获过多次：

```text
Failed to fetch RSC payload ... Falling back to browser navigation.
```

页面最终能 fallback 到整页导航，但会造成闪烁、变慢或用户误以为点击无效。

建议：

- 检查跨页面导航、RSC 请求和缓存配置。
- 对用户只显示正常 loading，不要让失败表现成无响应。

### P2-2: 开发阶段提示仍出现在正式体验里

多个页面底部出现：

```text
当前阶段：先固定功能入口和工作台风格。所有业务操作先从右侧按钮点击打开，暂不启用可视化场景物件交互。
```

这对内部开发有用，但会削弱用户信任。

建议：

- 生产/演示环境隐藏这类开发提示。
- 如果需要保留，放进“版本说明”或“开发者模式”。

### P2-3: 命名体系仍不统一

同类能力出现多个名字：

- 能力工坊
- 能力包仓库
- Skill
- skills
- skill-forge
- 机器人现场
- robot-site
- robotics

建议：

- 面向用户统一中文名。
- 面向路由统一英文名。
- 页面标题、入口、面板和 404 重定向都使用同一套映射。

### P2-4: 数据工场和 AI 实验室内容完整，但仍偏“展示型”

数据工场和 AI 实验室结构已经比主页面更清晰，但很多按钮像“看异常”“派给 NPC”“回工作台确认”，用户不确定点击后是否会真实派单、是否只读、是否需要审核。

建议：

- 给每个按钮加轻量动作标签：
  - `只读`
  - `登记`
  - `派单`
  - `需人审`
- 点击前给一个小确认或动作预览，尤其是派给 NPC 的按钮。

## 修复建议顺序

### 第一优先级

1. 修复复制失败，并给手动复制兜底。
2. 上 HTTPS 或至少明确 HTTP 限制。
3. 为 `/robot-site` 和 `/skills` 加重定向或修正入口。
4. 给项目主页面增加“协作者简洁视图”，默认隐藏运维字段。
5. 统一线程/Runner/NPC 状态，给出一句话可用性结论。

### 第二优先级

1. 重做电脑接入状态文案，改成中文行动导向。
2. 协作消息池增加“待我处理”顶置区。
3. 公司层空状态给出直接修复路径。
4. Git 回退和设备调试动作卡增加“只读/登记/会改项目/危险”分类。

### 第三优先级

1. 统一能力、机器人、Skill、工坊相关命名。
2. 隐藏开发阶段提示。
3. 优化 Next RSC fallback 导航体验。
4. 给数据工场和 AI 实验室按钮补动作标签。

## 本轮未执行项

以下动作因风险边界未执行：

- 真实硬件动作。
- ROS 写操作。
- 设备上电、电机动作、固件写入。
- 模型发布和最终验收。
- Git 回退登记提交、reset、revert、delete。
- 审核通过或拒绝。
- 删除/解绑工位、NPC、电脑。

## 结论

平台的“多用户邀请进入项目、多 NPC 工作台、专业工作台、电脑接入、协作消息、Git 安全边界”已经能形成完整体验骨架。最大问题不是入口缺失，而是普通用户进入项目后的判断成本过高。

下一轮应优先把协作者入口收敛成“项目经理式摘要”，把工程诊断信息折叠，把危险动作标签化，把复制和路由 404 修掉。这样新用户才能从“我看到了很多东西”变成“我知道下一步该做什么”。
