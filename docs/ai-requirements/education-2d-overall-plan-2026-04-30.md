# 教育版 2D RPG 嵌入式开发入口总体方案 - 2026-04-30

## 1. 产品定位

教育版 2D 是一个独立于现有开发板入口的游戏化学习入口。用户进入游戏前先选择 AI 模型、配置 API Key，然后进入一个 2D RPG 大世界。玩家通过接任务、找 NPC、完成小游戏、编译、仿真、接线、调参、烧录和验收，逐步完成真实嵌入式项目。

第一条主线项目是“平衡小车”。它不是传统课程，也不是纯聊天问答，而是“AI 导演 + NPC 分工 + 游戏任务 + 真实硬件套件”的嵌入式开发体验。

一句话定义：

```text
让学生像玩 RPG 一样，在 AI 和 NPC 的带领下做出一台真实可运行的平衡小车。
```

## 2. 边界声明

本方案不改动当前平台已有“开发板”入口，不占用另一个 AI 正在开发的开发板工作面。

建议新增独立入口与命名：

```text
/education-2d
/education-2d/setup
/education-2d/world
/education-2d/projects/balance-car
```

实现时应建立独立模块、独立路由、独立资源目录和独立文档，不复用现有开发板入口的页面状态、导航按钮或主流程。

## 3. 进入游戏前置流程

用户进入教育版 2D 前必须完成三件事：

1. 选择 AI 服务商或本地模型通道。
2. 输入 API Key 或选择本机已配置的密钥。
3. 选择体验模式。

推荐配置项：

```text
AI 服务商: OpenAI / Anthropic / 通义 / DeepSeek / 本地 OpenAI-compatible endpoint / 自定义
模型用途: 剧情导师模型 / 代码生成模型 / 视觉分析模型 / 轻量 NPC 对话模型
API Key: 用户输入，本地加密保存，默认不上传云端
体验模式: 学生模式 / 教师演示模式 / 高阶玩家模式
硬件模式: 仿真优先 / 实物优先 / 仅剧情体验
视觉扩展: 不启用 / 启用摄像头拍照分析
```

第一版可以只做一个主模型配置，但数据结构要预留多模型角色：

```text
story_model: 负责剧情、任务拆分、NPC 对话
code_model: 负责代码补全、解释、编译错误修复
vision_model: 负责拍照接线检查、实物状态识别
planning_model: 负责后续自由需求生成支线任务
```

安全原则：

1. API Key 只用于当前用户的会话和本地设备。
2. 明确提示 token 消耗。
3. 摄像头必须由用户主动授权，每次拍照都需要明确确认。
4. 真实烧录、上电、接线检查必须有人类确认门。

## 4. 游戏世界设定

世界风格应延续海报方向：明亮、科技、奇幻、嵌入式实验室与自然大世界混合。建议做成三分区大地图：

1. 新手村：模型配置、手册馆、任务大厅、基础 NPC。
2. 工坊区：代码工坊、接线台、仿真场、调参塔、烧录站。
3. 野外探索区：精灵、材料、支线入口、自由需求任务。

核心空间映射：

```text
手册馆 -> 读取套件资料、芯片手册、接线图、实验说明
任务大厅 -> 接主线和支线任务
模块工坊 -> 电机、IMU、电源、主控、蓝牙、显示等子模块
仿真接线台 -> 接线益智小游戏
调参塔 -> PID 调试与可视化曲线
烧录站 -> 编译、烧录、串口监视
验收广场 -> 演示效果、结算奖励、解锁新玩法
```

NPC 不是装饰，应绑定真实教育或工程职责：

```text
总导师 NPC: 解释目标、拆任务、控制节奏
手册管理员: 从套件资料生成任务上下文
主控工程师: 负责主控初始化与工程结构
传感器工程师: 负责 IMU 读取和姿态解算
电机工程师: 负责电机驱动与 PWM
控制算法师: 负责 PID 调试
仿真师: 负责虚拟接线和仿真验证
烧录师: 负责编译、烧录、串口日志
验收官: 负责任务验收和奖励发放
商店 NPC: 后续支线的器件清单与购买建议
```

## 5. 第一主线：平衡小车

主线分为三轮大任务。

第一轮：模块关键逻辑代码

玩家依次拜访多个模块 NPC，通过选择题、拖拽排序、代码填空、逻辑连线等方式完成关键模块理解。AI 可在玩家卡住时解释，也可在低龄模式下自动补齐复杂代码。

模块建议：

```text
工程初始化
IMU 读取
姿态角计算
电机 PWM 输出
编码器测速
PID 控制
安全停止
串口调试输出
```

第二轮：仿真接线

玩家在接线台完成虚拟接线益智游戏。系统校验每根线的端口、方向、电源电压和风险等级。通过后再允许进入实物接线指引。

第三轮：PID 调试与实物烧录

玩家在调参塔调整 Kp、Ki、Kd，观察小车倾角、速度、震荡和恢复时间。可以选择“我自己调”或“AI 辅助调”。每轮调参后由 AI 自动编译，烧录必须找烧录师 NPC 完成人类确认流程。

## 6. AI 任务编排机制

游戏启动项目时，AI 先读取“项目手册包”，再生成本次游戏任务图。

项目手册包包括：

```text
套件介绍
BOM
主控资料
传感器资料
电机驱动资料
接线图
示例工程
编译命令
烧录命令
常见错误
安全注意事项
验收标准
```

AI 输出任务图：

```text
主线章节
NPC 分工
每个 NPC 的对话目标
玩家任务
小游戏类型
代码片段
验收标准
奖励
失败重试提示
下一步解锁条件
```

建议保存为结构化 JSON，避免只靠聊天上下文驱动游戏。

## 7. 奖励与成长

奖励不能只做视觉反馈，应绑定学习和开发能力。

奖励类型：

```text
经验值: 完成任务获得
技能点: 解锁更高阶玩法
芯片徽章: 完成模块获得
精灵伙伴: 代表某类能力，如传感器精灵、控制精灵
工坊设施: 解锁仿真器、调参器、串口电视
项目图鉴: 保存玩家做过的项目
```

等级解锁：

```text
Lv1: 平衡小车主线
Lv2: 可查看更完整代码
Lv3: 可发起简单支线需求
Lv4: 可生成器件清单和购买建议
Lv5: 可启用摄像头接线检查
Lv6: 可让 AI 规划完整自定义项目
```

## 8. 支线与自由需求

当玩家等级足够后，可向总导师 NPC 提需求。AI 自动生成支线任务，包括：

```text
需求澄清
功能拆解
器件清单
购买建议和淘宝搜索关键词
接线图
仿真流程
代码任务
调试任务
验收标准
```

谨慎点：文档中可以保留“淘宝链接”能力，但第一版建议生成“淘宝搜索关键词 + 规格筛选规则”，不要自动承诺具体链接长期有效。真实商用时如果要放链接，应建立商品库、佣金合规、价格更新和缺货替换机制。

## 9. OpenClaw / 摄像头扩展

高级体验可配置 OpenClaw 或类似本机 Agent 能力，使 AI 能调用摄像头拍照，分析接线、板卡状态和实物位置。

第一版边界：

1. 只允许用户主动点击拍照。
2. AI 只给风险提示，不直接命令用户上电。
3. 接线分析结果必须显示置信度。
4. 对不确定结果给出“请人工核对”的建议。

可分析内容：

```text
杜邦线是否插错
电源正负是否明显反接
模块型号是否与任务匹配
小车姿态是否异常
烧录线是否连接
串口日志照片或截图
```

## 10. Linux 开发板载体

项目最终载体是一块类似树莓派的 Linux 开发板。系统层应预装：

```text
嵌入式工具链
常见板卡 SDK
Agent 环境
OpenClaw 环境
本地资料库
接线图库
示例工程
串口工具
编译烧录服务
离线任务缓存
本地 Web 游戏壳
```

开发板不是普通课程盒子，而是“游戏入口 + AI Agent + 嵌入式工具链 + 硬件套件资料库”的一体化学习终端。

## 11. 商用规划

可售卖形态：

```text
教育版 2D 软件授权
Linux 教育开发板
平衡小车硬件套件
学校机房部署版
教师控制台
课程包和项目包
视觉检查高级包
AI token 服务包
```

商用合规重点：

1. 素材必须自研、采购授权或使用明确可商用资产。
2. AI 生成素材要保留提示词、生成日期、模型、授权说明和人工审核记录。
3. 硬件接线、烧录、上电必须有安全免责声明和教师/成人确认流程。
4. 学生账号、摄像头照片、API Key、学习记录需要隐私保护。
5. 淘宝链接或商品推荐要标注价格和库存可能变化。

## 12. MVP 范围

第一版只做一个足够完整的闭环：

```text
模型/API 配置页
2D 新手村地图
8 个核心 NPC
平衡小车主线 3 轮任务
代码选择题/填空小游戏
仿真接线小游戏
PID 调参界面
编译结果展示
烧录师 NPC 人类确认流程
奖励和等级提升
项目完成证书
```

不做：

```text
多人联机
完整开放世界
真实商城下单
自动上电
AI 无确认烧录
复杂战斗系统
通用所有硬件项目
```

## 13. 技术建议

浏览器版本建议采用 Hybrid Phaser + React：

```text
Phaser: 地图、玩家移动、NPC、碰撞、触发区、精灵
React: 模型配置、HUD、任务面板、小游戏 UI、PID 曲线、确认弹窗
后端/本地服务: 手册解析、任务生成、编译、烧录、串口、摄像头
```

后续 Linux 开发板可使用本地 Web 服务承载游戏，浏览器访问本机地址即可启动。

## 15. 海报视觉锚点补充

2026-04-30 补充参考海报后，教育版 2D 的第一视觉应以 “A Agent” 海报为母版，而不是普通明亮卡通教学游戏。

海报确定的产品结构：

```text
A Agent: 一台盒子，接入你的 AI 协作团队
左侧入口: 教育版 / AI 游戏化学习
右侧入口: 开发版 / 多电脑 / 多模型 / 多线程 AI 协作
底部世界: 平台开发平台 / 以战养战
硬件载体: 黑色小型 Linux 盒子，带屏幕、USB-C、LAN、HDMI、DEBUG
地图语言: 暗色体素大地图 + 发光路径 + 职业 NPC 据点
```

教育版入口要和开发版入口并排出现在同一个品牌首页，但实现边界仍然独立。教育版不复用开发版入口内部逻辑，只共享品牌、硬件盒子和视觉语言。

教育版视觉方向修正为：

```text
dark premium sci-fi poster mood
isometric voxel RPG world
holographic green education panel
black hardware box with neon RGB edge light
tiny pixel/voxel engineer characters
glowing task paths between NPC stations
high-end commercial product render + game map hybrid
```

教育版卡片在首页上的信息应短而强：

```text
教育版
AI 游戏化学习
通过 NPC 接任务
从零学习编程 / 嵌入式 / AI
完成项目实践，解锁新技能
进入教育版
```

第一版教育版地图可以继承海报底部的职业据点结构，但要换成教育主线用途：

```text
产品经理 NPC -> 总导师 / 需求拆解
前端工程师 NPC -> 游戏 UI / 显示面板任务
后端工程师 NPC -> 编译服务 / 日志任务
嵌入式工程师 NPC -> 主控 / 传感器 / 电机任务
测试工程师 NPC -> 验收 / 安全检查
设计师 NPC -> 成就 / 项目展示 / 作品证书
```

海报里的“最多 30 个线程自动化协作”属于开发版卖点。教育版第一版不展示这个能力，避免让学生入口显得过重。教育版可以只显示“1 个 AI 导师 + 多个 NPC 任务角色”，后续等级提升后再逐步解锁更多 AI 分工。

## 16. 首页双入口交互约束

未来实现首页时，建议采用同屏双入口：

```text
左侧教育版: 绿色 hologram 面板，强调学习、NPC、项目实践
右侧开发版: 蓝色 hologram 面板，强调多模型、多电脑、多线程协作
中部/底部: A Agent 硬件盒子，作为两个版本共同载体
底部: 体素大地图，展示职业 NPC 和协作世界
```

点击“进入教育版”后进入 `/education-2d/setup`，要求模型/API 配置。点击“进入开发版”继续走另一个 AI 正在开发的开发版入口，本方案不修改。

教育版用户路径：

```text
A Agent 首页 -> 进入教育版 -> 选择 AI 模型 -> 输入 API Key -> 进入星桥工坊镇 -> 接平衡小车任务
```

开发版用户路径：

```text
A Agent 首页 -> 进入开发版 -> 多模型/多电脑/多线程协作配置
```
