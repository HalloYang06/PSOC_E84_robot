# AI 协作平台闭环矩阵 - 2026-05-10

## 北极星

平台不是聊天壳，也不是任务列表，更不是 Codex / Claude Code 的替代品。它要让用户用一句项目目标，经过 Boss NPC 拆解，进入多工位、多 NPC、多电脑协作，并且每一步都能看到：谁发起、谁执行、谁审核、谁回执、从哪里索引、回到哪里治理。

最终处理过程必须留在绑定的 Codex / Claude Code / Runner 线程中展示和执行。平台只显示派单摘要、最小回执、审核决定、最终结果和可回源索引，避免把长日志、完整推理过程或命令细节搬进工作台。

当前结构必须保持：

`项目 -> 工位 -> NPC -> 线程`

`/projects/{id}/2d-upgrade` 是项目主页面，负责资源治理。  
`/projects/{id}/workbench` 是多 NPC 协作工作台，负责对话、审核、执行摘要、回执。

## 闭环标准

一个能力只有满足下面 6 个点，才算闭环：

1. 用户知道入口在哪里。
2. 平台能预演或提示影响面。
3. 执行对象明确到工位 / NPC / 线程 / Runner。
4. 需要人审时，审核按钮出现在对应 NPC 对话消息后面。
5. 执行后有 ack / progress / done / reject 回执。
6. 所有索引 chip 都能回到资源治理页面或原始详情。

## P0 收口项

1. Workbench 保持聊天优先：审核、回执、CLI 摘要都在 NPC 对话流里，运行详情默认折叠。
2. 主页面资源治理保持唯一来源：电脑、Runner、NPC、线程、Skill、知识库、工位都从主页面创建或修改。
3. Git 回退必须从“手填 ref”升级成“可选版本索引”。
4. Git 回退登记后必须同步给 Boss NPC 和受影响工位长，要求回执“已对齐 / 阻塞 / 需人工”。
5. 用 YueSpeak 走一遍软件项目链路，再用机器人场景模拟任务走一遍高风险链路。

## YueSpeak 验收路径

1. 在 `/2d-upgrade` 确认 GitHub、电脑、NPC、线程、Skill、知识库路径。
2. 在 workbench 打开 Boss、Backend、Frontend、QA 四个 NPC。
3. 用户只给 Boss 一句话目标。
4. Boss 输出分工、线程需求、Skill 与知识库路径。
5. Boss 派给执行 NPC；跨工位需要工位长；待审消息在对话流按钮通过。
6. 执行 NPC 回 ack / progress / done。
7. Git 面板选择一个版本做回退预演，登记后 NPC 收到对齐请求。

## 机器人开发验收路径

1. 创建 App、ROS/Linux、硬件、VLA、QA/Safety 工位。
2. 每个工位至少一个工位长 NPC。
3. 用户提出“机械臂抓取联调”目标。
4. Boss 拆成跨电脑任务，明确本地路径只由 Runner 自己决定。
5. 硬件、上电、串口、机械臂运动相关消息强制人工审核。
6. Git 回退和实验记录必须能通知相关工位长并收回执。
