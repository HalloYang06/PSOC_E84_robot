# AI 必读需求表

本文档是所有 NPC / AI 线程开工前必须读取的固定需求表。它不是聊天日志，而是协作契约：谁提出、谁承接、做什么、什么时候必须停下来等人审核、完成后回给谁。

固定 Skill：`ai-required-requirement-ledger`

平台派给 AI 的 `agent_command` 会自动附带 `AI_REQUIRED_REQUIREMENT_LEDGER_V1` 协议块。目标 AI 必须先读协议块，再执行正文。

## 开工前固定动作

1. 先读本表，确认自己是否是“被提需求者”。
2. 如果需求标记为“需要人工审核”，只能阅读、分析、列方案、给最小回执，不能继续自动执行高风险动作。
3. 如果需求允许自动推进，也必须先回最小回执，再做当前一轮工作。
4. 完成后必须回写给“提需求者”和平台最终回复池；如果需要下游 AI 接棒，必须新增下一条需求，而不是私下连续喊话。
5. 不确定是否该继续消耗 token 时，默认停在最小回执，等待人工确认。

## 人工审核边界

以下情况必须直接转人工审核，不能自动推进：

- 删除、覆盖、回退、发布、提交、推送、迁移数据、执行硬件动作。
- 会明显消耗 token / 费用 / 时间的持续自动化。
- 涉及其他电脑本地路径、硬件环境、串口设备、Unity、嵌入式或外部工具。
- 多个 AI 对同一文件或同一目标给出冲突方案。
- 需求不清楚、验收标准不明确、提需求者和被提需求者不匹配。

## 需求表字段

| 字段 | 必填 | 说明 |
| --- | --- | --- |
| 需求 ID | 是 | 平台 requirement / collaboration message / 人工补录 ID |
| 项目 | 是 | 所属项目，禁止跨项目串数据 |
| 提需求者 | 是 | 人、NPC、AI 线程、电脑主角都可以是提需求者 |
| 被提需求者 | 是 | 明确目标 NPC / AI / 人 / 工位 |
| 需求正文 | 是 | 做什么、参考什么、交付什么 |
| 自动化许可 | 是 | `off` 只做一次；`heartbeat` 可按心跳继续；`manual_review` 等人审 |
| 心跳间隔 | 否 | 自动化打开时的最小轮询间隔 |
| 人工审核规则 | 是 | 哪些动作必须停下来让人确认 |
| 最小回执 | 是 | 被提需求者先回“我接到、我怎么做、是否要人审” |
| 最终回复 | 是 | 完成后给提需求者和最终回复池的交付内容 |
| 下一步需求 | 否 | 如果要接力给其他 AI，必须生成下一条需求 |
| 状态 | 是 | `queued` / `acked` / `needs_review` / `running` / `completed` / `blocked` |

## 10+ 线程协作规则

- 需求不是口头传话，必须落到平台需求表或协作消息池。
- AI 之间可以提需求，但被提需求者必须能看到“谁提的、为什么、交付给谁”。
- 不允许所有线程同时抢同一条需求；需要一个当前负责人。
- 如果某个 AI 完成了自己的部分，要把结果回给原提需求者；如果下一步属于其他 AI，新增下一条需求并注明依赖关系。
- 平台首页必须把 `needs_review` 直接展示给人，直到人给出继续、驳回或补充要求。

## 机器人 / 嵌入式项目补充规则

- 默认先只读或仿真，不直接触碰真实硬件。
- 上电、烧录、串口写入、驱动安装、固件发布必须人审。
- 只读前端页面、截图验收、DOM 检查、平台内部证据链展示可以自动推进；它们不等同于真实硬件动作。
- 不同电脑路径和工具链不同，AI 只能描述需要的环境，不要假设远端路径与本机一致。
- 远端电脑如果没有本地项目，应先按 GitHub 拉代码启动 skill 自行选择本地路径。

## 多电脑 GitHub 协作规则

- 项目如果绑定了 GitHub 仓库，跨电脑协作优先使用 GitHub 仓库、目标分支和相对路径。
- 每台电脑的本地项目路径由该电脑的 Runner / AI 自己决定，禁止把一台电脑的绝对路径当成所有电脑通用路径。
- GitHub 账号绑定只说明“使用哪个身份、凭据从哪里来”，不保存明文 token。
- AI 不能在消息正文、需求表、最终回复或知识库里粘贴 `ghp_...`、`github_pat_...`、私钥等明文凭据。
- `clone`、`status`、`diff`、只读资料整理可以按任务权限执行；`push`、`pull`、`reset`、`revert`、删除分支、发布版本、跨账号访问必须先人审。
- 如果仓库地址、账号或凭据来源缺失，AI 应先回最小回执并请求人补齐，而不是猜测远端仓库或本地路径。

## 纯软件项目补充规则

- 可以更积极地自动推进，但仍要尊重 Git 边界、测试、截图和最终回复。
- 修改代码前先说明写入范围；多个 AI 负责不同文件或模块，避免抢同一处。
- 完成后必须说明验证命令、截图或测试结果。
