# AI 协作平台全链路审计 - 2026-05-10

## 审计结论

当前平台已经能支撑一部分 YueSpeak 协作验证，但还不能算“只靠平台就放心开发项目”。主要问题不是缺一个按钮，而是几个闭环还不完整：

- Workbench 对话正在接近可用，但还需要用真实派单验证审核消息、线程回执、NPC 回执是否都在同一对话流里自然出现。
- `/2d-upgrade` 主页面已经承担资源治理，但 Git 回退仍停留在“手填 ref + 登记请求 + Runner 只读预检”，还没有“可索引版本列表 -> 选择版本 -> 预演 -> 登记 -> 通知相关 NPC -> NPC 确认对齐”的闭环。
- YueSpeak 可以继续作为软件项目验证场景，但每次必须走平台路径，不要绕过前端直接改项目。
- 机器人/机械臂场景需要把 Git 回退、硬件风险审批、实验记录和跨电脑 Runner 状态做成一级能力，否则多电脑协作会不可信。

## 用户真实开发链路

1. 主页面治理项目资源：
   - 绑定 GitHub 仓库。
   - 绑定/扫描电脑 Runner。
   - 创建工位、指定工位长。
   - 创建 NPC、绑定用户已创建的 Codex/Claude Code 线程。
   - 分配 Skill 和 GitHub 相对路径知识库。

2. Boss NPC 生成项目方案：
   - 根据一句话目标和仓库文档拆出工位/NPC/Skill/验收口。
   - 明确每个线程需要读哪些 GitHub 相对路径。
   - 生成项目运行契约，所有 NPC 遵守同一套路径、回执、风险规则。

3. Workbench 执行协作：
   - 用户和每个 NPC 在独立对话窗口交流。
   - NPC 间消息也同步到对应 NPC 对话流。
   - 需要人审的消息作为“系统/审核”消息出现在对话流里，正文后面直接有通过/免审/打回。
   - 同工位 NPC 按职责互相找人。
   - 跨工位请求必须走目标工位长。
   - 线程回执只显示摘要，完整日志放抽屉或 handoff 文档。

4. 版本与回退：
   - 用户从主页面 Git 面板看到可回退版本索引。
   - 选择一个版本后生成回退预演。
   - 预演展示影响的分支、未完成任务、相关 NPC、风险提示。
   - 登记回退后，不直接执行破坏性操作，而是下发给相关 Runner/NPC 做只读预检。
   - 平台在 workbench 给相关 NPC 同步一条“Git 回退对齐请求”。
   - NPC 回执“已对齐/阻塞/需要人工处理”。

## Git 回退现状

已存在：

- 后端 `/api/git/projects/{project_id}/rollback-preview`。
- 后端 `/api/git/projects/{project_id}/rollback`。
- 后端 Git activity read model。
- 主页面 Git 面板有回退预演和登记表单。
- 登记后会写审计，并尝试给 Runner 下发 `git.preflight` 只读预检。

不足：

- 用户必须手填 `target_ref`，没有版本索引列表。
- 没有显示 commit 时间线、标签、分支、最近成功验收点。
- 没有把可回退版本和当前项目任务/NPC/线程关联起来。
- 登记回退后没有明确通知相关 NPC。
- Workbench 没有“Git 回退消息”类型的自然同步。
- 没有“NPC 已经对齐到目标版本”的回执闭环。
- 没有把回退风险和硬件/机器人实验状态联动。

## Git 回退应补的产品形态

### 主页面 Git 面板

新增“版本索引”：

- 当前 HEAD。
- 默认分支。
- 开发分支。
- 最近 20 个 commit。
- 最近登记过的同步点/回退点。
- 可选 tags。
- 每个版本显示：短 sha、标题、作者、时间、来源分支、是否有审计记录、是否有关联任务。

新增“选择回退版本”：

- 用户点某个版本。
- 自动填入目标版本。
- 展示影响摘要。
- 再点“生成预演”。

### 回退预演

预演必须展示：

- 目标版本。
- 当前分支上下文。
- 可合并分支数量。
- 阻塞分支数量。
- 未完成任务。
- 受影响 NPC/工位。
- 是否存在硬件/机器人高风险任务。
- 下一步建议。

### 登记回退

登记后必须：

- 写 Git activity。
- 下发 Runner 只读预检。
- 在 collaboration message 里给 Boss NPC 和受影响工位长/NPC 同步一条消息。
- 消息正文包含目标版本、原因、影响范围、要求回执格式。

### NPC 回执

相关 NPC 必须回：

- 已收到回退请求。
- 当前线程/工作目录版本状态。
- 是否能对齐目标版本。
- 是否有未提交改动。
- 是否需要人工处理。

## YueSpeak 下一轮验收路线

1. 从 `/2d-upgrade` 检查 YueSpeak 项目资源：
   - GitHub 仓库是否绑定。
   - 电脑 Runner 是否在线。
   - Boss、Backend Data、Frontend Miniapp、QA Acceptance 是否绑定线程。
   - Skill 和知识库路径是否能从 chip 回到主页面。

2. 在 `workbench` 真实打开多个 NPC：
   - 用户给 Boss 发目标。
   - Boss 派给 Backend/Frontend/QA。
   - 审核消息是否在对话流里并带按钮。
   - 通过后目标 NPC 是否收到。
   - 目标 NPC 是否回执。

3. Git 回退验证：
   - 当前先验证已有回退预演/登记表单。
   - 记录“必须手填 ref”的缺口。
   - 下一步实现版本索引后，再用 YueSpeak 仓库做真实选择版本测试。

4. 机器人/机械臂泛化验证：
   - 用一个模拟任务验证高风险关键词强审。
   - 确认 Git 回退请求会通知 ROS/Linux、嵌入式、QA/Safety 工位长。
   - 确认不同电脑的本地路径不进入跨电脑知识库约定，只使用 GitHub 相对路径。
