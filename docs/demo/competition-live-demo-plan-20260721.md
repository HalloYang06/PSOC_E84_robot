# 灵动康复机械臂现场演示方案

日期：2026-07-21

目标：在 5 分钟内稳定呈现一套可理解、可观察、可降级、可重复的智能康复机械臂系统，突出多模态交互、边缘视觉、数字孪生、肌电意图、训练规划和安全裁决，而不是堆叠页面或宣称尚未完成的能力。

## 一句话定位

这是一个面向上肢康复的多模态智能助力系统：小智负责自然语言入口，NanoPi 双目视觉和 RK3576 NPU 负责目标证据，ROS 2 与 MuJoCo 负责候选轨迹和数字孪生，M55 负责肌电意图建议，M33 负责最终安全裁决，App 与云平台负责训练计划和全过程留痕。

## 现场必须讲清楚的闭环

```text
用户说“我口渴了”
  -> L：小智语音转文本，平台做语义解析与模式分类
  -> V：双目摄像头识别水杯和末端，输出置信度、深度与相机坐标证据
  -> A：生成取物任务候选，MuJoCo 先预演接近过程
  -> Safety：NanoPi 前置检查，M33 最终裁决
  -> Action：空载机械臂执行已验证的两点接近轨迹并回零
  -> Evidence：平台记录语义、视觉、轨迹、状态和日志
```

当前尚未完成手眼标定，因此不能把相机 XYZ 直接宣称为机械臂 base 坐标。明天的真实运动采用经过台架验证的关节轨迹；视觉坐标用于目标存在、距离变化和 MuJoCo 候选证据。答辩时主动说明这道边界，会比伪装成完整端到端 VLA 更有工程可信度。

## 四级演示档位

四个档位讲同一个故事。出现故障时只降低实时程度，不临时换项目、不现场修代码。

### L1 全实时闭环，首选

- 小智新语音实时进入平台。
- NanoPi 双摄实时上传水杯识别帧。
- 平台显示 L、V、A 状态变化。
- MuJoCo 执行同一候选轨迹。
- 真机 4/5/6 号电机执行两点接近、停顿和回零。
- 平台日志留下本次 session 证据。

启用条件：云端、NanoPi、Linux 主机、CAN、M33 状态、4/5/6 反馈和空载回零全部通过。

### L2 混合闭环，现场推荐保底

- 小智、平台和 NanoPi 视觉保持实时。
- MuJoCo 实时执行。
- 真机轨迹由操作员在 A 阶段单独触发已验证轨迹，不依赖视觉坐标自动转换。
- 平台仍显示完整 VLA 过程，但解说必须称为“感知驱动任务、受控轨迹执行”，不称为已完成手眼闭环。

使用场景：自动触发不稳定、云端事件延迟、真机路径与平台事件难以严格同步。

### L3 数字孪生闭环

- 小智和平台实时。
- 视觉实时或使用刚刚采到的最后一帧。
- MuJoCo 实时运动。
- 真机保持静止，平台显示安全门未放行或操作员未确认。

使用场景：CAN/M33/电机反馈不满足条件。这个档位仍能体现系统安全性，不要在台上修 CAN。

### L4 离线证据闭环

- 播放本地已同步三画面视频：实物、平台、MuJoCo 同时存在。
- 使用平台功能导览录像补充 EMG、App 和日志。
- 讲解使用过去时：“这是系统最近一次完整联调记录”。

使用场景：现场网络、NanoPi 与 Linux 主机均不可用。离线文件必须在本机和 U 盘各保存一份。

## 五分钟现场导演本

### 0:00-0:25 产品与痛点

画面：机械臂实物居中，右侧控制台停在系统总览，不滚动页面。

讲解：

> 上肢康复训练长期存在三个问题：训练动作重复、患者意图难以识别、传统设备交互僵硬。我们的方案把语音、双目视觉、肌电意图、数字孪生和安全控制放进同一个可观察闭环，让设备既理解患者需求，也保留确定性的安全边界。

评委应看到：实物、PSoC E84、NanoPi、双摄像头、平台在线状态。

### 0:25-0:50 架构亮相

画面：平台总览或一张固定架构图。

讲解：

> 系统不是让大模型直接控制电机。M55 负责语音和肌电模型建议，NanoPi 的 RK3576 NPU 负责视觉，ROS 2 和 MuJoCo 负责规划与预演，M33 始终是最终安全裁决者。App 和云平台负责训练计划与审计。

只讲五个角色，不展开协议字段。

### 0:50-2:20 核心 VLA-Lite 取水演示

0:50：操作员先确认平台处于 L 页面或模式路由页。

0:55：演示者面对小智清晰说：

> 小智，我口渴了。

1:00-1:12：平台出现新 transcript、`fetch_object` 或取物模式、水杯目标。不要提前点下一页。

讲解：

> L 层把自然语言“口渴”解析成取水任务，不是简单关键词直接驱动电机。

1:12：切到 V 页面。镜头中水杯放在固定高对比位置，保持 3 秒不动。

评委应看到：

- 左右相机不是同一画面；
- 检测框来自后端图像，不是浏览器伪造框；
- 置信度、深度或相机坐标随新帧更新；
- 画面新鲜度持续变化。

1:25：把水杯平移一次，距离不必太大。平台显示目标位置或距离证据变化。

1:35：MuJoCo 开始执行第一个接近点，真机随后执行第一个点。

1:48：第一个点停顿 2-3 秒。演示者再次移动水杯少量距离，表示重新定位。

1:55：执行第二个目标点，重点让 5/6 号电机产生清楚但平稳的等高度追踪效果。

2:08：轨迹回到逻辑零点。

讲解：

> V 层持续提供目标和末端证据，A 层生成分段接近候选。比赛演示的真机部分采用预验证关节轨迹，视觉坐标到机械臂坐标的手眼标定是下一阶段，我们没有把相机坐标直接冒充机械臂坐标。

### 2:20-2:50 危险物品拒绝

画面：回到 L/A 联动页，机械臂已回零。

演示者说：

> 小智，帮我拿剪刀。

平台应显示危险物品分类、拒绝原因和“未下发动作”。真机与 MuJoCo均保持静止。

讲解：

> 多模态系统不仅要会执行，还必须会拒绝。危险目标不会进入轨迹候选，M33 也不会因为云端指令绕过本地安全边界。

### 2:50-3:40 肌电意图与助力

画面：平台肌电页面，3D 肌肉模型占主视觉，波形和动作可信度位于侧栏。

演示动作：放松、弯肘意图、肩前抬意图各保持 2-3 秒。如果实时模型未复验，只展示实时肌电波形和离线模型评估，不声称当前分类来自实时 M55。

讲解：

> 四路肌电传感器形成 300 毫秒窗口，每 100 毫秒更新一次意图。当前三类 Int8 模型准确率 98.56%，大小仅 22.2 KB，部署目标是 M55。AI 只判断高层意图，助力方向、速度、限流和关节边界仍由 M33 的确定性控制策略决定。

### 3:40-4:20 App 与个体化训练计划

画面：手机 App 的训练计划或 AI 草案页，平台同时停在审计/训练记录。

操作：展示用户填写目标，生成下一训练草案，再显示“草案”和审计来源。不在 BLE 未复验时点击真实下发。

讲解：

> App 不是遥控器。它管理患者档案、训练计划和每日反馈，AI 根据历史训练生成下一周期草案。草案仍需同步、M33 接受、preflight 后才能开始训练；小智语义链和 App 训练规划通过 client type、purpose 和 scope 隔离，记录统一展示但控制链不混用。

### 4:20-5:00 总结与可扩展性

画面：平台日志或系统拓扑。

讲解：

> 本项目的核心价值不是单个模型，而是端云协同的可解释闭环：语言决定任务，视觉提供空间证据，肌电理解患者意图，数字孪生提前验证，M33完成最终安全裁决，App沉淀长期训练数据。当前原型已完成关键链路验证，下一步是手眼标定、完整六自由度映射和患者个体化助力策略。

最后停在系统拓扑或日志证据页，不停在报错页。

## 现场页面与实物布局

### 实物区

- 机械臂空载、无人穿戴，底座固定。

## 今晚必须完成的验收

每项必须记录 `通过/失败/降级档位`，任何一项失败都直接决定明天使用哪个档位。

### 1. 云端

```powershell
Invoke-WebRequest -UseBasicParsing http://106.55.62.122:8011/health
Invoke-WebRequest -UseBasicParsing http://106.55.62.122:3001/projects/e201f41c-25a6-46e1-baf8-be6dcb83284c/rehab-arm-control
```

通过标准：两者 HTTP 200，浏览器能登录，L/V/A/肌电/日志页面均能切换。

当前证据：2026-07-20 两个地址均返回 200。

### 2. 局域网设备

```powershell
ping <NanoPi实际IP>
ping 192.168.3.34
```

通过标准：连续 20 次无明显丢包。不要依赖历史 IP，先从路由器或 `arp -a` 确认 NanoPi 实际地址。

当前证据：2026-07-21 本机位于 `192.168.111.0/24`，不在设备使用的 `192.168.3.0/24`，因此已知 NanoPi 地址和 `192.168.3.34` 不可达不能证明设备未上电。接入设备路由器后必须重新确认。

### 3. NanoPi 视觉

```bash
uname -r
systemctl is-active rehab-rknpu-load.service rehab-vla-vision.service
systemctl is-enabled rehab-rknpu-load.service rehab-vla-vision.service
lsmod | grep rknpu
journalctl -u rehab-vla-vision.service -n 80 --no-pager
```

通过标准：

- 两个服务 active/enabled；
- 不改内核，现有匹配模块正常加载；
- 左右相机是两个真实设备；
- 连续 60 秒有新帧；
- 平台帧龄优先小于 1 秒，至少不持续增长；
- 日志无重复 `async upload failed`；
- 水瓶连续 10 秒大多数帧能检测，不以“每帧 100%”作为不现实门槛。

### 4. MuJoCo 与 ROS 2

在 Linux 主机确认：

```bash
ros2 topic list -t | grep -E "joint_trajectory|joint_states|rehab_arm"
ros2 topic hz /sim/medical_arm/joint_states
```

通过标准：MuJoCo 可视化打开，发布逻辑零点后 `/sim/medical_arm/joint_states` 回到六个零，轨迹运动无突然跳变。

### 5. CAN 与 M33

只做状态检查，不临场改驱动或内核：

```bash
ip -details -statistics link show can0
timeout 3 candump can0
```

通过标准：

- `can0` 为 UP、非 LOOPBACK、非 listen-only、ERROR-ACTIVE；
- RX 中存在 M33 `0x322` 新鲜状态；
- 4/5/6 号电机反馈持续存在；
- tx/rx error、bus-off、restart 不持续增长；
- 3 号电机不参与明天轨迹，不因其缺失阻塞 4/5/6 台架演示。

### 6. 真机轨迹

先空载单独执行一次，不录视频、不说话、不启动平台导演。

## 上场前 30 分钟流程

1. 所有设备接入同一独立路由器，关闭会自动切换的 Wi-Fi。
2. 上电顺序：路由器、Linux 主机、NanoPi、M33/M55、传感节点、电机。
3. 检查云端 200、设备 ping、NanoPi 服务、MuJoCo、CAN、M33 状态。
4. 机械臂回到逻辑零点，确认 3 号电机不参与轨迹。
5. 水瓶放入标记位置，观察 10 秒识别稳定性。
6. 完整跑一次 L2，不做临场代码修改。
7. 浏览器重新预加载七个页面并保持登录。
8. 将演示档位写在纸上：`L1/L2/L3/L4`，由安全观察员在上场前决定。
9. 清除桌面通知，接通电源，关闭休眠、屏保和系统更新。
10. 保持离线视频已打开并暂停在第一帧，切换失败时 5 秒内可播放。

## 三人分工

### 主讲人

- 只负责讲解、说两条语音和移动一次水瓶。
- 不操作命令行，不解释临时故障。

### 平台操作员

- 提前加载页面，按导演本切换 L、V、A、肌电、App 和日志。
- 负责启动一键脚本和 L2 手动 action 触发。
- 发现异常直接按降级表切换，不现场调代码。

### 安全观察员

- 站在机械臂电源与停止装置旁。
- 只观察回零、运动方向、线缆和人员距离。
- 有权直接取消真机部分，主讲人继续 L3 剧本。

如果只有两个人，平台操作员兼任安全观察员，真机运行时不切页面，由主讲人移动水瓶。

## 评委高概率问题

### 这是真正的 VLA 吗？

回答：

> 当前是面向嵌入式原型的可解释 VLA-Lite，不是一个端到端大模型直接输出电机电流。我们把语言任务解析、视觉 grounding、动作候选、数字孪生和安全裁决模块化，既能体现 V、L、A 协作，也符合康复设备对确定性和可审计性的要求。

### 为什么不用深度相机？

回答：

> 我们使用 6 cm 基线双目摄像头，完成了标定接口、点匹配和稠密深度回退，并把推理迁移到 RK3576 NPU。双目方案成本低、结构可定制。当前仍在补手眼标定，所以相机坐标只作为证据，不直接冒充机械臂坐标。

### 98.56% 的肌电准确率可靠吗？

回答：

> 数据按 trial 分组拆分，避免同一次采集泄漏到训练和测试。Int8 模型在 2018 个测试窗口上达到 98.56%。我们同时记录了当前第 21 特征来自 trial index 的局限，正式版本会移除该特征并做跨人、跨日验证。

### AI 出错怎么办？

回答：

> AI 只提供意图和候选。危险语义、低质量视觉、陈旧反馈、未知关节和超限都会阻断候选。最终动作必须经过 NanoPi gate 和 M33 本地裁决，云端和 App 不能直接发 CAN 或释放急停。

### 项目创新点是什么？

回答：

> 第一，PSoC E84 双核把智能推理与确定性控制分离；第二，低成本双目加 RK3576 NPU形成边缘视觉；第三，语言、视觉、肌电和训练计划共用同一平台但权限隔离；第四，MuJoCo 数字孪生和全过程证据让康复训练可解释、可复盘。

## 明天成功的判定

现场成功不等于所有模块都必须实时在线。满足以下五点即可形成完整国赛叙事：

1. 评委能在 30 秒内理解项目解决什么问题。
2. 至少一次展示真实语言进入模式分类。
3. 至少一次展示真实视觉帧或最近一次完整视觉证据。
4. 至少一次展示 MuJoCo 或真机动作，且动作前后状态可解释。
5. 至少一次展示危险拒绝、安全边界或审计证据。

最佳结果是 L1；稳定且诚实的 L2 通常比临场故障频发的 L1 更有竞争力。今晚验收结束后冻结代码、模型、相机位置、轨迹和页面版本，明天只执行，不开发。
