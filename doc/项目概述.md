# 康复机械臂项目概述

更新日期：2026-06-08

## 1. 总体目标

项目目标是构建一套医疗康复外骨骼机械臂系统。它不是单个电机 demo，而是一个分层系统：真实运动必须经过 M33 安全裁决；端侧 AI、服务器/VLA、App 和 MuJoCo 只提供辅助判断、状态展示、标注、仿真或 dry-run 候选。

核心路径：

```text
JointTrajectory -> NanoPi -> M33 -> motor
```

只读/仿真 shadow 路径：

```text
M33 motor status 0x330~0x333
-> NanoPi /joint_states
-> wireless ROS2
-> Linux sim host
-> MuJoCo /sim/medical_arm/joint_states
```

M55 模型结果路径：

```text
C8T6/EMG/语音/电机上下文 -> M33 -> M55
M55 -> M33 -> CAN 0x323 -> NanoPi -> /rehab_arm/model_state
```

`/rehab_arm/model_state` 永远是建议或语义状态，不是运动许可。


职责：

- 作为 M33/M55 共享内存 consumer 时只 attach，不 format 共享块。

重要提醒：

- M55 的 `confidence` 或 `result_code` 不能被上层解释成 `motion_allowed=true`。
- `wifi` 当前不是 Git 仓库，修改前必须先决定版本管理方式。

## 3. 当前已验证状态

- NanoPi `can0` 为 `ERROR-ACTIVE`，`berr-counter tx/rx` 为 `0`。
- `rehab-arm-nanopi-readonly.service` 已验证 active/enabled。
- Linux 仿真主机 `rehab-arm-sim-host-shadow.service` 已验证 active/enabled。
- 3 号电机常态 fresh，能从 M33 `0x330` 进入 NanoPi `/joint_states`，再通过无线 ROS2 进入 MuJoCo shadow。
- 临时打开 4/5/6 active-report 后，四个装机关节能一起进入 MuJoCo shadow；验证后必须关闭 active-report。
- 最后安全检查 `timeout 2 candump -L can0,320:7FF` 为空。

## 4. 当前不能夸大的内容

- 不能说 6DOF 真机闭环运动已经完成；当前验证重点是只读状态和 shadow。
- 不能说真实 4 路 EMG 模型已经训练完成。
- 不能说 VLA 已经能安全控制真机。
- 不能把 7 号外部电机当作正式机械臂关节。
- 不能把无线 ROS2 当作高频安全闭环。
- 不能把 M55、App、服务器或 MuJoCo 的输出当成真实运动许可。

## 5. 下一阶段建议

1. 逐个标定 3/4/5/6 的方向、零点、输出侧比例、软/硬限位和患者 ROM。
2. 明确 4/5/6 active-report 是否由 M33 启动逻辑安全开启，而不是依赖 NanoPi 调试命令。
3. 接线并确认 1/2 腕部 4015 电机协议和关节对应关系。
4. 把 C8T6 四路 EMG 接入 `C8T6 -> M33 -> M55 -> M33 -> 0x323 -> NanoPi`。
5. 把 M55 当前验证模型替换成正式 int8 TFLite Micro EMG/疲劳/意图模型。
6. 服务器/VLA 继续只输出高层任务或 dry-run 候选，经 MuJoCo 和 M33 安全边界后才能进入真实运动主线。
