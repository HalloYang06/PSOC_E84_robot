# AI协作平台上市前审查报告

版本：V0.3 审查更新版  
审查日期：2026-04-19  
审查范围：前端、后端、协作主链、权限与安全、Git 工作流、Runner 与多电脑协作、部署治理

---

## 1. 当前结论

和上一版相比，这个工程已经明显从“能演示”推进到了“可持续内测、可继续产品化”的阶段。

截至 2026 年 4 月 19 日，本轮确认结果如下：

1. 后端回归测试通过：`python -m pytest tests -q` 为 `53 passed`
2. 前端生产构建通过：`npm run build:web` 成功
3. 多电脑节点、AI provider、thread workstation 已经是正式可编辑的产品模型
4. 项目创建、消息写入、需求写入、审批动作、handoff、Git 高风险入口都已接入显式权限校验
5. invitation 的 read model 已不再暴露 token
6. 启动时自动建表、自动灌演示数据，已经从默认行为改成显式测试/开发开关

结论也因此更新为：

**上一版报告里列出的高优先级问题，已经只剩 2 项还应保留为“上市前仍需继续收口”的事项；另外 3 项已修复，2 项应降级为持续产品化事项。**

---

## 2. 原问题还剩几个

上一版阻塞清单里，核心有 5 项：

1. 鉴权和权限控制未完全收口
2. invitation token 暴露
3. 高风险审批链不够硬
4. 默认开发密钥 / bootstrap 身份回退
5. 启动即建表和灌样板数据

现在的状态是：

### 2.1 已修复：3 项

1. invitation token 暴露：已修复  
   invitation 的序列化结果里已移除 `token`，列表、详情、accept 返回都不再泄漏 token。

2. 高风险审批链不够硬：已修复  
   审批创建只允许 `pending`，直接 patch 状态会被拒绝，`approve / reject / request-changes` 会绑定当前真实 principal 并写入审计链。

3. 启动即建表和灌样板数据：已修复默认风险  
   自动建表和自动 seed 不再是默认启动行为，测试环境通过 `tests/conftest.py` 显式开启，生产态还有额外守门。

### 2.2 部分收口但仍需继续跟进：2 项

1. 鉴权和权限控制未完全收口：大幅改善，但建议继续做全模块复核  
   本轮已经补上了 `messages`、`requirements`、`approvals`、`handoffs`、`git` 等关键写入口，主链风险大幅下降；同时新增了 `knowledge` 读取权限、`runner next-task` 和 `git merge-readiness` 的回归覆盖，并把遗漏较多的写接口合同测试收束成了共享审计用例。  
   但如果目标是“正式上市”，仍建议继续对所有写接口做一次全量权限矩阵复核，确保没有边缘模块漏网。

2. 默认开发密钥 / bootstrap 身份回退：风险已明显下降，但还没完全到最终形态  
   `bootstrap` 已经改成显式配置开关，默认关闭；生产态禁止开启。  
   自动建表/seed 也已从默认行为移走。  
   但 session secret 仍保留本地开发兜底语义，虽然生产态会拦截不安全配置，最终仍建议切到完全外部化密钥管理。

### 2.3 应降级为产品化持续事项：2 项

1. Runner / 多电脑调度仍未形成真正执行闭环
2. Git 工作流仍以工作台和请求入口为主，还不是完整的 clone / branch / PR / merge 产品链

这两项很重要，但更适合作为“下一阶段产品化建设重点”，而不是继续和权限/安全问题混在同一层级里。

## 3. 本轮已确认修复的点

### 3.1 权限主链

以下关键写链路已经补上显式权限：

1. 项目创建后，创建者自动成为 owner
2. `messages` 写入口已按项目作用域鉴权
3. `requirements` 写入口已统一走项目写权限校验
4. `approvals` 已区分普通更新和高风险动作
5. `handoffs` 的 create / accept / assign / message 已补权限
6. `git` 的 `sync-github` / `rollback` 已要求项目高权限
7. 核心写接口的共享权限审计测试已补齐，能更快发现漏网的 POST / PATCH / DELETE 路由

### 3.2 邀请链

1. invitation token 不再出现在 API 读模型中
2. accept 流程没有被破坏
3. 前端成员页也不再依赖 token 展示

### 3.3 合同回归

1. `knowledge` 列表已经有成员可见性和项目过滤回归
2. `git merge-readiness` 已有分支态与 blocker 聚合回归
3. `runner next-task` 已有身份边界回归，避免把状态枚举写死成旧值

### 3.4 启动与部署边界

1. `database_auto_create` 默认关闭
2. `database_auto_seed` 默认关闭
3. `allow_bootstrap_auth` 默认关闭
4. 测试环境通过 `tests/conftest.py` 显式开启测试所需行为
5. 生产态会阻止危险启动配置

---

## 4. 现在还该保留的上市前问题

如果按“真正上市、可对外承诺”的标准来看，我建议当前还保留以下 2 个重点问题：

### 4.1 全模块权限矩阵还需要一次总复核

虽然主链已经基本收住，但平台模块很多，仍建议继续做一次“所有 POST / PATCH / DELETE 路由”的总盘点，确认：

1. 是否全部要求真实身份
2. 是否都落在正确的项目作用域
3. 是否高风险动作都要求 privileged 角色
4. 是否不存在只靠 payload 声明身份的边缘接口

这已经不是“明显缺洞”的状态，而是上市前的最后一轮收口检查。

### 4.2 密钥治理还需要切到正式模式

当前生产态已经会拒绝不安全配置，但还没完全进入成熟产品的密钥治理形态。后续建议继续补：

1. `.env.example` 与真实部署变量对齐
2. 正式区分开发密钥、测试密钥、生产密钥
3. 引入 migration-first 数据库治理，而不是只做到“默认不自动建表”

---

## 5. 已不应继续按原强度保留的问题

以下内容已经不应继续写成“当前硬阻塞”：

1. invitation token 暴露
2. 审批状态可被客户端直接伪造
3. 默认启动就自动灌演示数据
4. “多电脑 / 多 AI / 多工位只是配置味”
5. “前端当前无法稳定 build”

其中第 4、5 项尤其需要更新表述，因为当前工程已经具备真实的协作控制台和稳定构建结果。
