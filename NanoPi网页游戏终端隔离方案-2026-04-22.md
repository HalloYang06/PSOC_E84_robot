# NanoPi 网页游戏终端隔离方案 - 2026-04-22

版本：v0.1  
日期：2026-04-22  
用途：解决 NanoPi 上电即进入网页游戏，同时不让用户接触预装开发环境和系统后台的问题

---

## 1. 目标

这份方案要解决的不是单纯“自动打开一个网页”，而是做出一种更接近游戏机或专用终端的体验：

1. NanoPi 一上电就进入网页游戏
2. 用户看不到 Linux 桌面、终端、文件系统
3. 用户不能碰我们预装好的开发环境
4. 游戏界面和后台开发环境隔离
5. 后台仍然能完成编译、烧录、串口读取、AI 调用等能力
6. 即使浏览器崩溃或异常退出，也能自动恢复到游戏界面

一句话定义：

**把 NanoPi 做成一台只能进入网页游戏壳的专用开发终端，而不是一台普通 Linux 电脑。**

---

## 2. 整体思路

推荐采用“三层架构”：

1. 启动层
2. 前台交互层
3. 后台服务层

其工作方式是：

`上电 -> 自动进入受限会话 -> 自动启动全屏网页壳 -> 用户只操作网页 -> 网页通过 API 调用后台服务`

这套模式下，用户永远不直接接触真实系统环境，只能通过你定义好的网页交互来使用能力。

---

## 3. 最推荐的架构方案

### 3.1 启动层：受控开机

NanoPi 启动后不进入普通桌面，不出现系统登录界面，不给用户选择系统功能。

应采用：

1. 自动登录受限账号
2. 自动进入 kiosk 会话
3. 自动全屏打开网页游戏
4. 自动隐藏系统级界面元素

理想体验是：

1. 上电
2. 看到启动 Logo
3. 进入配网页面或游戏首页
4. 后面始终留在你的产品界面里

### 3.2 前台交互层：网页游戏壳

用户可见的只有这一层。

前台可以是：

1. Chromium kiosk 模式
2. 极简 WebView 壳
3. Wayland/X11 下的单应用全屏窗口

建议优先：

**Chromium kiosk + 极简桌面或无桌面会话**

因为实现难度和维护成本最低。

前台应具备这些限制：

1. 无地址栏
2. 无标签页
3. 无开发者工具
4. 无下载入口
5. 无系统任务栏
6. 无文件管理入口
7. 尽可能屏蔽常见快捷键

这样用户感知上就是一个游戏客户端，而不是浏览器。

### 3.3 后台服务层：隐藏开发能力

真正的开发环境、编译环境、烧录工具、AI CLI、串口服务、文件处理服务都放在后台，不暴露给用户。

后台应由以下对象承载：

1. `systemd` 服务
2. root 或专用 service user
3. 容器
4. 本地 API 服务

前台网页只能请求这些能力：

1. 编译
2. 烧录
3. 串口查看
4. 仿真启动
5. AI 生成代码
6. 查询项目状态

但用户不能直接打开 shell、不能直接读写系统文件、不能直接运行命令。

---

## 4. 账号与权限设计

这是整套隔离里最关键的一层。

### 4.1 至少分成两类身份

#### A. 使用者账号

例如：`player`

职责：

1. 自动登录
2. 运行前台网页壳
3. 保存极少量前台会话数据

限制：

1. 不是管理员
2. 没有 sudo
3. 不给正常 shell 使用权
4. 不允许访问后台服务目录
5. 不能改系统启动项

#### B. 系统服务账号

例如：`devsvc`

职责：

1. 跑编译服务
2. 跑烧录服务
3. 跑串口服务
4. 跑 AI CLI 代理
5. 跑日志与状态服务

特点：

1. 不直接给用户登录
2. 只通过 systemd 或容器运行
3. 权限精细化拆分

### 4.2 不建议直接给用户开发者桌面

即使存在“开发者模式”，也不建议让开发者模式等于“进入 Linux 桌面”。

更好的方式是：

1. 用户模式也是网页壳
2. 开发者模式也是网页壳
3. 两者只是权限不同、菜单不同、页面不同

也就是说，开发者模式可以看到更多日志、设备状态、项目页、协作页，但仍然不碰底层系统桌面。

---

## 5. 前后台隔离方案

你昨晚提到的关键点是：

**游戏界面和后台要做隔离。**

这可以拆成 4 层隔离。

### 5.1 界面隔离

用户只看到网页游戏和网页管理界面，看不到：

1. 终端
2. 文件夹
3. 包管理器
4. 系统设置
5. 编译器入口

### 5.2 进程隔离

前台浏览器和后台服务不是同一个进程权限域。

建议：

1. 浏览器进程归 `player`
2. 编译/烧录/串口等服务归 `devsvc` 或 root 控制
3. 通过本地 API 或消息队列通信

### 5.3 文件隔离

不同层的文件应分开放置：

1. 系统文件
2. 预装工具链
3. 项目缓存
4. 用户数据
5. 日志

用户前台只能接触到与项目交互相关的数据副本，不能直接看到真实工具链目录。

### 5.4 网络隔离

前台网页不能任意访问系统本地所有服务。

建议：

1. 只开放必要 API
2. API 做鉴权和白名单
3. 不暴露 Docker socket
4. 不暴露 shell 执行接口
5. 不开放任意命令执行能力

---

## 6. 最适合的技术落地方式

从实现难度、维护性和你当前方向来看，最实际的是下面这套。

### 6.1 显示层

建议：

1. 精简 Linux 系统
2. Wayland 或极简 X11
3. 开机自启 Chromium kiosk

启动参数思路：

1. 全屏
2. 禁止恢复标签页
3. 隐藏崩溃提示
4. 禁止弹出系统默认浏览器交互
5. 默认加载本地欢迎页或远端游戏页

### 6.2 后台服务层

建议做成多个本地服务：

1. `game-shell-service`
2. `build-service`
3. `flash-service`
4. `serial-service`
5. `ai-gateway-service`
6. `device-state-service`

每个服务只负责一种能力，不要做成一个超大后端。

### 6.3 容器层

如果你们后续希望更好维护，可以把高风险后台能力放到容器里：

1. AI CLI 容器
2. 编译环境容器
3. 项目运行容器
4. 日志与数据处理容器

这样好处是：

1. 易更新
2. 易回滚
3. 易限制权限
4. 不容易把整个系统拖坏

## 7. 开机即进入网页游戏的推荐流程

推荐流程如下：

### 7.1 首次开机

1. 上电
2. 自动启动设备初始化服务
3. 如果未配网，则进入“配网页”
4. 配网完成后写入设备配置
5. 自动重定向到登录页或首页

### 7.2 日常开机

1. 上电
2. 系统自动启动网络
3. 自动登录受限账号
4. 自动启动 kiosk 浏览器
5. 浏览器自动打开目标网址
6. 进入网页游戏首页

### 7.3 异常恢复

1. 如果浏览器退出，则自动拉起
2. 如果网页崩溃，则回到默认首页
3. 如果网络断开，则切回断网提示页或本地离线页
4. 如果后台服务异常，则前台显示受控错误页，不暴露系统细节

## 8. 使用者模式与开发者模式

你们已经想到两种模式，这个方向是对的，但应统一在同一网页壳里完成。

### 8.1 使用者模式

适用对象：

1. 普通用户
2. 学生
3. 小白玩家

可见内容：

1. 游戏世界
2. 项目任务
3. 引导教程
4. 提交与回执
5. 简化设备操作

不可见内容：

1. 系统日志详情
2. 编译环境管理
3. 设备内部目录
4. 运维工具
5. 后台服务控制台

### 8.2 开发者模式

适用对象：

1. 高阶用户
2. 资源玩家
3. 内部测试者

可见内容：

1. 更多项目管理页
2. 更多设备状态页
3. 编译结果与日志摘要
4. AI 协作入口
5. 其他电脑协作管理页

注意：

开发者模式也最好只开放“网页后台功能”，不要直接开放完整 Linux 桌面。

这样可以继续保持系统一致性和安全边界。

---

## 9. 如何避免用户破坏预装环境

这是产品能不能长期稳定卖出去的关键。

推荐至少做这 6 件事：

### 9.1 根文件系统尽量只读

系统主分区尽量只读化，减少误操作和环境污染。

### 9.2 用户数据独立挂载

把用户项目缓存、设置、日志放到单独可写分区或目录。

### 9.3 工具链目录只给服务访问

预装 SDK、编译器、AI CLI、脚本目录不暴露给前台账号。

### 9.4 统一通过 API 操作

编译、烧录、串口、仿真、AI 调用都走受控 API，而不是开放终端。

### 9.5 做恢复机制

提供一键恢复默认状态、重建缓存、重置前台配置等机制。

### 9.6 自动自检

开机后检测：

1. 浏览器是否可拉起
2. 后台服务是否在线
3. 本地配置是否损坏
4. 当前设备是否需要进入修复页

---

## 10. 与现有平台的关系

这套 NanoPi 隔离方案，最适合成为你现有平台的一个硬件终端形态，而不是独立于平台之外另做一套逻辑。

更合理的关系是：

1. NanoPi 负责成为专用入口设备
2. 网页游戏负责成为用户前台
3. 后台服务负责本地能力与设备执行
4. 远端平台负责账号、项目、任务、AI 协作和结果回执

也就是说：

**NanoPi 是终端，网页游戏是壳，后台服务是执行层，协作平台是大脑。**

---

## 11. 最小可行版本建议

如果只做一个第一版，建议不要一上来做太复杂的系统改造，而是先做一个能验证体验的 MVP。

### 第一版最小能力

1. NanoPi 开机自动进全屏网页
2. 使用者模式默认进入游戏首页
3. 无桌面、无终端、无文件管理器
4. 后台仅开放几个核心 API
5. 浏览器崩溃自动重开
6. 提供配网页和错误恢复页

### 第一版先不做太重的部分

1. 先不做完整多容器编排
2. 先不做太复杂的离线模式
3. 先不做复杂本地 AI 推理
4. 先不做完整桌面级开发者模式

先把“像游戏机一样上电即用”做扎实。

---

## 12. 一句话结论

想让 NanoPi 一上电就进入网页游戏，并且不让用户碰你们搭好的系统环境，最合适的方式不是给用户一台普通 Linux 机器，而是：

**做一台受限账号自动登录、前台只跑全屏网页壳、后台开发环境隐藏在服务层或容器层、所有能力都通过 API 间接暴露的专用终端。**

如果后续继续推进，下一步最值得补的是：

1. NanoPi 启动流程图
2. 使用者模式 / 开发者模式权限表
3. 后台服务拆分清单
4. 恢复与升级机制方案
## 补充说明（2026-04-22）

本文档描述的是 NanoPi 部署形态下的受控终端方案，不代表整套产品只能运行在 NanoPi 上。

必须同时记住：
1. 同一套网页游戏前台也要支持在普通电脑浏览器运行。
2. NanoPi 方案解决的是硬件交付、开机即用和系统隔离问题。
3. 电脑端仍然是正式产品入口之一。
