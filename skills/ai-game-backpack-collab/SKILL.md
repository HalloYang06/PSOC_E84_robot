# AI Game Backpack Collab

当游戏页右侧协作面板开始变成“所有东西都往里塞”的长滚动混合表单时，必须使用这份 skill。

目标不是继续堆功能，而是把游戏内协作能力收成一个短路径、可切换、可验证的背包式工作台，并持续验证前后端链路是否真的打通。

## 产品目标

背包协作面板固定拆成 6 个分舱：

1. `电脑接入`
2. `NPC 创建`
3. `机房`
4. `信息交流`
5. `Git 合作`
6. `Skill 库`

不要再把这些功能混成一整页滚动内容。

## 分舱边界

### 1. 电脑接入

这里只做：

- 新增电脑
- 配对令牌
- 在线状态
- 线程扫描
- 扫描前置条件说明

不要把 NPC 创建逻辑塞进这里。

### 2. NPC 创建

这里只做 AI NPC / 席位创建。

固定流程：

1. 选择在线电脑
2. 选择这台电脑上的真实线程
3. 命名 NPC
4. 填职位和职责
5. 装配 skill
6. 配置 Git 边界
7. 配置形象 / 场景 / 坐标 / 说明

### 3. 机房

这里是查看和编辑区：

- 电脑页
- NPC 页
- 单个 NPC 属性页

NPC 至少要能看和改：

- 名字
- 职位
- 电脑
- 线程
- 场景
- 坐标
- 形象 key
- `skill_loadout`
- `git_boundary`
- 说明
- 当前状态

### 4. 信息交流

这里只保留最小工作流：

- 最小报告中心
- 按人 / 按线程看
- 收件箱
- 发送命令
- 按需求看回执

发命令时必须指定给某个 AI / NPC，不能发给模糊的“总 Codex”。

### 5. Git 合作

这里是 AI 之间通过 Git 协作的中间桥。

至少包含：

- Git 配置
- 维护员派单
- 分给 AI
- 分给人
- 版本活动
- 平台自维护状态
- Git 边界查看

规则：

- AI 之间主要通过 `Git + handoff + requirement flow` 协作
- AI 通过间接访问 Git 查找发给自己的任务
- 写需求时必须指明“需要哪个 AI 干什么”

### 6. Skill 库

这里是所有 NPC 共用的 skill 来源。

要求：

- 支持检索 skill
- 支持新增自定义 skill
- 支持删除自定义 skill
- 支持把 skill 装备给 NPC
- 装配数量不限

## 工作流规则

每次推进都按这个顺序：

1. 先确认 6 个分舱边界没被破坏
2. 再补当前最缺的一条前后端链路
3. 自己截图验证
4. 自己做后端验证
5. 清掉验证用 demo 和临时脏数据
6. 更新 handoff 文档

## 验证要求

### 前端

至少执行：

- `npm run build:web`
- 页面级验证
- 截图保存到 `artifacts/`

重点验证：

- 用户能否找到入口
- 是否能完整完成一次操作
- 页面文案是否清晰
- 右侧面板是否仍然过载

### 后端

至少执行：

- `python -m pytest tests -q`

必要时补：

- 接口直测
- 数据库回查
- 真实链路写入确认

重点验证：

- 电脑接入是否真实落库
- 线程扫描是否真实回传
- NPC 创建是否真实写入
- 命令 / 回执 / Git 需求是否真实进入后端

### 清理要求

验证结束后必须清掉：

- 临时 demo 账号
- 临时项目成员
- 临时需求单
- 临时消息
- 临时 skill
- 临时数据库记录

## Git 协作要求

这个面板里的协作核心是 Git，不是聊天。

要求：

- 平台项目自己的 Git 要真实接入
- NPC 要能看到自己负责的 Git 边界
- Git 合作页要成为 AI 之间协作的桥
- 需求要能明确分派给人或 AI

## 手动接入要求

最终要能给用户一份手动接入操作指南，至少覆盖：

- 电脑怎么接入
- 线程怎么扫描
- NPC 怎么创建
- Skill 怎么装配
- Git 边界怎么配置
- 怎么给指定 AI / NPC 发命令

## 禁止事项

- 不要回到 dashboard 风格分叉页
- 不要用假线程、假在线电脑糊页面
- 不要只改前端文案不跑后端验证
- 不要保留一次性 demo 残留
- 不要把不同分舱的职责重新混回一起

## 交付要求

每次应用这份 skill 后，在 handoff 文档里写清：

- 改了哪几个分舱
- 接进了哪些真实后端数据
- 做了哪些截图验证
- 做了哪些后端验证
- 清掉了哪些临时 demo 或脏数据
