# UTF-8 Guardrail

当项目里包含大量中文，并且已经出现过乱码、问号占位、整文件误改坏、Markdown frontmatter 损坏或 JSX 被错误替换时，必须使用这份 skill。

## 目标

统一 6 件事：

1. 所有中文源码、文档、配置统一保持 UTF-8。
2. 中文文案统一为正常可读的简体中文。
3. 避免 PowerShell 或 shell 误重写整文件导致编码损坏。
4. 每轮改动后立刻做语法、构建和页面级验证。
5. 验证完成后清掉临时 demo、临时账号、临时数据。
6. 在 handoff 文档里记录编码修复和验证结果。

## 强制规则

### 1. 手工源码修改只用 `apply_patch`

源码、文档、JSON、脚本等人工修改统一用 `apply_patch`。

不要用这些方式直接重写源文件：

- `Set-Content`
- `Out-File`
- `>`、`>>`
- 用脚本整文件替换中文内容
- 逐行拼接重写源码

### 2. 读取时优先保留原样

检查文件时优先使用只读方式：

- `Get-Content -LiteralPath`
- `Select-String`
- `rg`

不要为了改一处中文，先把整个文件重新编码一遍。

### 3. 文案先统一，再继续叠功能

如果当前文件还有乱码、问号、半截中文、异常引号：

1. 先修语法。
2. 再修编码和中文。
3. 再继续加功能。

不要在已知乱码文件上继续堆新功能。

### 4. 中文文案统一风格

统一要求：

- 使用正常简体中文
- 不保留 `???`、`乱码占位`、半中半英碎片
- 同一字段不要混用中文、英文和乱码

优先修这些位置：

- 标题
- 按钮
- 标签
- 提示语
- 表单占位
- 状态词
- handoff 标题

### 5. 优先最小改动

如果文件只是局部乱码，就只修局部。

只有在以下情况才允许整文件重建：

- JSX/TSX 结构已经明显损坏
- frontmatter 已经损坏
- 多处乱码导致继续局部修补成本更高

### 6. 验证后必须清理 demo

验证过程中产生的这些内容，验证结束后必须清掉：

- 临时账号
- 临时项目成员关系
- 临时需求单
- 临时消息
- 临时 skill
- 临时数据库记录

只保留真正有用的截图、日志和正式配置。

## 每轮验证要求

### React / Next

至少执行：

- 语法或解析检查
- `npm run build:web`

如果改动涉及登录态、项目页、背包页或地图交互，还要补页面级验证。

### Python / 后端

至少执行：

- `python -m pytest tests -q`

如果新加了接口或数据写入链，还要补直接接口验证或数据库回查。

### 文档 / Skill / Markdown

至少确认：

- frontmatter 完整
- 文件打开后是正常中文
- 没有乱码标点或问号残留

## 交付要求

每轮结束时，在 handoff 文档里记录：

- 哪些文件做了 UTF-8 修复
- 清掉了哪些乱码
- 做了哪些验证
- 清掉了哪些 demo 残留

## 禁止事项

- 不要在中文源文件上使用整文件覆盖式写入
- 不要跳过构建验证
- 不要把 `验证用脏数据` 留在数据库里
- 不要把乱码 handoff 当成后续协作基线
