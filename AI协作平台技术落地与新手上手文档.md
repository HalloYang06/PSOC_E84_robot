# AI 协作平台技术落地与新手上手文档

版本：v0.1  
日期：2026-04-19  
用途：帮助后续 AI 和新加入的开发者快速理解当前工程结构、技术栈选择、功能接入方式，以及如何在不把工程搞复杂的前提下继续扩展平台能力

---

## 1. 这份文档解决什么问题

这份文档不是产品方案，也不是审查报告，而是回答下面这些最实际的问题：

1. 当前项目到底是怎么搭起来的？
2. 后续功能应该往哪里加？
3. 怎样加功能才不会把工程越做越乱？
4. 新来的 AI 或开发者，怎样最快进入状态？
5. 如果继续做嵌入式和机器人专项功能，技术上最稳的路径是什么？

目标只有一个：

**让别人一看就知道怎么在当前工程里加东西，而且不会把项目搞得繁琐、难维护。**

## 2. 当前项目的真实技术栈

从当前仓库结构看，平台已经形成了比较清晰的分层：

### 2.1 仓库结构

```text
D:\ai合作产品
├─ apps
│  ├─ web        # 前端 Next.js
│  └─ api        # 后端 FastAPI
├─ packages
│  └─ shared     # 预留共享包
├─ infra         # Docker / 部署文件
├─ skills        # 项目内技能和规则
└─ 文档与规范
```

这是一个很适合继续扩展的平台结构。

理由：

1. 前后端已经分开。
2. 共享层已经预留。
3. 部署层独立。
4. 项目文档已经非常多，适合团队继续协作。

---

### 2.2 前端技术栈

当前前端在：

`D:\ai合作产品\apps\web`

技术栈：

1. Next.js 14
2. React 18
3. TypeScript
4. CSS Modules

当前脚本：

1. `dev`
2. `build`
3. `start`
4. `lint`

1. Next.js 已经足够支撑平台型 Web 应用。

3. TypeScript 能帮助多人协作和 AI 协作减少理解成本。

4. 微前端

**把现有页面体系做实，而不是继续换前端花样。**

## 3. 当前工程最好的扩展原则

后面加功能时，必须遵守几个规则。

### 3.1 一个功能，先确定属于哪一层

任何新功能先判断它属于：

1. 前端展示层
2. 前端动作层
3. 后端业务层
4. 数据模型层
5. Runner 层
6. 部署配置层

不要还没想清楚就同时在 4 个地方乱改。

举例：

#### 例 1：新增“硬件调试卡”

应该拆成：

1. 后端新增 `lab_debug_cards` 相关模型和 API
2. 前端实验楼增加页面和表单
3. 审批模块增加关联字段
4. 审计模块增加调试卡留痕

而不是：

1. 直接在前端塞一堆静态文本
2. 再在 `actions.ts` 临时拼接口
3. 再让任务页顺便显示一点

---

### 3.2 新功能优先放进现有模块，不轻易新开系统

当前模块已经很多了。

所以新增功能时优先判断：

1. 这是 `tasks` 的扩展吗？
2. 这是 `requirements` 的扩展吗？
3. 这是 `approvals` 的扩展吗？
4. 这是 `lab` 的扩展吗？
5. 这是 `context` / `handoffs` / `knowledge` 的扩展吗？
6. 这是 `projects` / `collaboration` 的扩展吗？

只有在现有模块完全不合适时，才新开模块。

这样能避免：

1. 功能分裂
2. API 命名混乱
3. 多套入口重复

---

### 3.3 页面新增时，不要先做“新世界”

现在最稳的做法是：

1. 继续沿用现有 `apps/web/app/...` 路由结构
2. 每个页面只负责一个明确业务域
3. 样式尽量复用 `game.module.css` 和现有页面模式
4. 先把动作做实，再升级视觉

例如：

如果要加 `实验记录流`

优先新增：

```text
apps/web/app/experiments/page.tsx
apps/web/app/experiments/[id]/page.tsx
```

不要一上来做：

1. 全新前端子应用
2. 单独前端工程
3. 大型拖拽式实验编辑器

---

### 3.4 后端新增能力时，坚持“router / service / schemas / model”四层

这是后续 AI 最容易看懂的结构。

推荐固定模式：

```text
modules/<feature>/
  router.py
  service.py
  schemas.py
```

必要时再加：

```text
repository.py
```

不要把新增逻辑写成：

1. 全塞在 router 里
2. 全塞在 service 里
3. 到处直接写 SQLAlchemy 查询

最稳的理解方式是：

1. `router.py` 负责接口入口和权限门
2. `service.py` 负责业务逻辑
3. `schemas.py` 负责输入输出结构
4. `models/*.py` 负责持久化

## 4. 对新开发者最友好的上手路径

新接触的人不能一上来就看全部文档，否则会晕。

建议固定一个最短上手路径。

### 第一步：先看项目定位

阅读顺序：

1. `D:\ai合作产品\README.md`
2. `D:\ai合作产品\AI协作平台开发文档.md`
3. `D:\ai合作产品\AI协作平台第一版开发入口.md`

目标：

理解这个平台是干什么的，而不是先看代码细节。

---

### 第二步：理解工程结构

新同学只需要先理解：

1. `apps/web` 是前端
2. `apps/api` 是后端
3. `infra` 是部署
4. `packages/shared` 是共享层

不需要第一天就把全部模块背下来。

---

### 第三步：只挑一个业务链来看

推荐新手先看一条最完整的链：

**任务 -> 审批 -> 交接 -> 审计**

前端页面：

1. `/tasks`
2. `/approvals`
3. `/handoffs`
4. `/audit`

后端模块：

1. `tasks`
2. `approvals`
3. `handoffs`
4. `audit`

为什么推荐这条：

1. 最能代表平台的核心价值
2. 同时覆盖人机协作、风险控制和留痕

---

### 第四步：只从一个功能开始改

新开发者或新 AI 进入时，建议只接一种改动：

1. 改一个页面
2. 加一个表单动作
3. 补一个 API
4. 补一个状态流转

不要让新手第一次就做：

1. 前后端一起大改
2. 改 5 个模块
3. 重构消息系统
4. 改游戏化首页

---

## 5. 当前项目最适合继续加的功能，技术上怎么落

下面这些是最值得加的，而且可以在现有工程里比较稳地落下去。

---

### 5.1 硬件调试卡系统

目标：

让硬件问题有结构化记录，不再只靠口头描述。

最小落地方式：

#### 后端

新增模块建议：

```text
apps/api/app/modules/debug_cards/
```

新增模型建议：

```text
DebugCard
DebugMeasurement
```

字段建议：

1. `project_id`
2. `task_id`
3. `board_name`
4. `chip_model`
5. `firmware_version`
6. `symptom`
7. `checklist`
8. `measurement_result`
9. `serial_log`
10. `attachments`
11. `status`

#### 前端

页面建议：

```text
/lab/debug-cards
/lab/debug-cards/[id]
```

复用现有：

1. `lab`
2. `approvals`
3. `audit`

不要新造独立实验系统。

---

### 5.2 模块责任地图

目标：

把“谁负责什么”正式画出来。

最小落地方式：

#### 后端

优先不要新开超大模块，可以先挂到 `projects`：

1. `ProjectModule`
2. `ProjectModuleAssignment`

#### 前端

新增：

```text
/projects/[id]/modules
```

页面显示：

1. 模块名
2. 负责人类
3. 负责 AI
4. 相关目录
5. 风险等级
6. 最近改动

---

### 5.3 整机版本快照

目标：

让回滚不只是代码回滚，而是整机状态回滚。

最小落地方式：

#### 后端

新增模块建议：

```text
modules/releases/
```

或者先挂在 `projects` / `git` 下，但更推荐独立模块。

模型建议：

```text
SystemSnapshot
SnapshotArtifact
```

字段：

1. `project_id`
2. `git_commit`
3. `firmware_version`
4. `ros_version`
5. `config_version`
6. `model_version`
7. `lab_result`
8. `approved_by`

#### 前端

新增：

```text
/projects/[id]/snapshots
```

---

### 5.4 接口契约中心

目标：

减少跨模块反复问接口。

最小落地方式：

#### 后端

新增模块建议：

```text
modules/contracts/
```

模型：

```text
InterfaceContract
ContractVersion
```

#### 前端

新增：

```text
/projects/[id]/contracts
```

每条契约至少显示：

1. owner
2. consumer
3. 协议类型
4. 字段
5. 示例
6. 版本

---

### 5.5 实验记录流

目标：

让实验成为正式对象。

最小落地方式：

#### 后端

新增模块建议：

```text
modules/experiments/
```

模型：

```text
Experiment
ExperimentArtifact
ExperimentResult
```

#### 前端

新增：

```text
/experiments
/experiments/[id]
```

最好与：

1. `lab`
2. `knowledge`
3. `audit`

联动，而不是做成孤岛。

---

### 5.6 任务接力模式

目标：

让跨模块任务有接力链。

最小落地方式：

#### 后端

优先扩展 `tasks`，不要先开新模块。

可新增字段：

1. `parent_task_id`
2. `upstream_task_id`
3. `handoff_required`
4. `relay_stage`

#### 前端

扩展：

```text
/tasks/[id]
```

增加：

1. 上一棒
2. 下一棒
3. 当前交接状态
4. 当前输入输出

这样比重做一个“流程引擎”稳得多。

## 6. 怎样避免把工程搞复杂

这是最重要的一节。

### 6.1 不要同时引入太多新技术

现在已经有：

1. Next.js
2. FastAPI
3. SQLAlchemy
4. Docker

这套足够了。

不要为了某个新功能再引入：

1. 新前端框架
2. 新后端框架
3. 新数据库
4. 复杂事件总线
5. 额外消息队列

除非现有架构真的撑不住。

---

### 6.2 不要为“未来也许会用到”提前建大系统

比如：

1. 插件市场
2. 微服务拆分
3. 多租户重构
4. 复杂的权限 DSL
5. 独立调度引擎

这些都应该等现有功能真正跑起来以后再说。

---

### 6.3 一个功能最好最多改三层

推荐规则：

1. 改一个后端模块
2. 改一个前端页面
3. 必要时补一个共享类型

超过这三个层级，复杂度就会明显上升。

---

### 6.4 所有新能力都要挂回主链

所谓主链，就是：

```text
项目
-> 成员 / AI / Runner
-> 任务
-> 需求 / 审批 / 交接
-> 审计 / 知识 / 实验
```

任何新功能如果不能挂回这条链，最好先不要做。

## 7. 给新 AI 的最短理解说明

如果后面又有新的 AI 线程进来，给它最短说明可以是：

```text
这个项目是一个 AI 协作平台。

前端在 apps/web，后端在 apps/api，部署在 infra，共享层在 packages/shared。

继续开发时优先沿现有模块扩展，不轻易新开系统。

新增功能优先服务：
1. 项目
2. 任务
3. AI 工位
4. Runner
5. 需求
6. 审批
7. 交接
8. 审计
9. 实验

如果要加嵌入式 / 机器人专项能力，优先做：
硬件调试卡、模块责任地图、整机版本快照、接口契约中心、实验记录流、任务接力模式。
```

这样比让新 AI 直接读全部文档更容易进入状态。

## 9. 最终建议

这个项目后面继续做，一定要记住三句话：

### 9.1 先让别人看得懂，再让功能变复杂

一个新 AI 或新人如果看不懂现在的结构，后面只会越加越乱。

### 9.2 先把主链做实，再做花活

嵌入式和机器人平台最重要的是：

1. 任务
2. 审批
3. 交接
4. 审计
5. 实验

不是先做最炫的页面。

### 9.3 新功能一定要为真实研发服务

后面优先级最高的，不是“更酷的 AI”，而是：

1. 更稳的调试
2. 更清楚的责任边界
3. 更可靠的实验记录
4. 更方便的版本回滚
5. 更低的沟通成本

一句话总结：

**这个项目最好的技术路线，不是不断换技术，而是在当前结构上稳稳地长出更适合嵌入式和机器人研发的能力。**
