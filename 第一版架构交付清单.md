# 第一版架构交付清单

版本：v0.1  
范围：仅面向第一版 AI 合作平台 MVP（以 `AI协作平台第一版开发入口.md` 为准）  
目标：跑通核心闭环，不做过度设计

---

## 1) 第一版系统模块边界

第一版必须跑通闭环：

```text
创建项目
  -> 添加 AI
  -> 注册 Runner
  -> 创建任务
  -> 指派 AI
  -> 创建任务分支
  -> Runner 执行（受限命令）
  -> 查看日志和 diff
  -> 记录 token
  -> 审查
  -> 人类确认
  -> 合并或回滚
```

对应模块边界如下（只列第一版必须）：

1. Web 前端（管理与可视化）：研发基地首页 + CRUD 管理页 + diff/log + 成本/上下文 + 人类确认。
2. API 后端（业务真相中心）：项目/AI/Runner/任务/Git/审计/确认/usage/context/handoff。
3. Runner（执行节点）：注册/心跳/领取任务/工作区/受限命令执行/日志与结果回传。
4. 数据存储：PostgreSQL（必需）；对象存储（可选，先用本地或 DB 存摘要）。
5. Git 集成：创建任务分支、读取 diff、回滚入口（第一版可先依赖 Git CLI 或 GitHub，镜像服务可选）。
6. 安全边界：H0-H4 人类确认点 + 不暴露密钥 + 禁止硬件自动操作 + 全量审计日志。

## 2) 推荐代码目录结构（第一版）

第一版建议先落地三件套：`apps/web`、`apps/api`、`apps/runner`。其余包先预留。

```text
apps/
  web/                       前端（Next.js）
    app/                     路由页（base/projects/agents/tasks/...）
    components/              复用组件（BaseTopBar/WorkstationCard/...）
    lib/                     API client、类型、工具
  api/                       后端（FastAPI）
    app/
      main.py
      core/                  settings/errors/permissions/audit helper
      db/
        models/
        migrations/
      modules/
        auth/
        projects/
        agents/
        runners/
        tasks/
        git/
        usage/
        context/
        handoffs/
        approvals/
        audit/
  runner/                    Runner（Python）
    runner/
      main.py
      client/                调平台 API
      workspace/             工作区管理（任务目录、缓存、清理）
      executor/              受限命令执行（白名单）
      logs/                  日志采集与上传
      adapters/              预留 Agent Adapter（第一版可空壳）
packages/
  shared/                    共享类型（可后置）
infra/
  docker-compose.yml         基础服务（Postgres/Redis 可先上）
docs/                        文档（已在仓库根，后续可整理）
```

原则：

1. 第一版不强制 monorepo 完整治理，先能跑通。
2. 目录边界要清晰，跨目录修改必须走需求单或人类确认。

## 3) 第一版 API 模块清单（路径级别）

说明：第一版 API 追求“够用 + 可扩展”，不追求一次到位。

### 3.1 基础

1. `GET /api/health`
2. `GET /api/version`

### 3.2 Auth（第一版可极简）

第一版可选方案：

1. 方案 A：单管理员模式（本地固定账号/密码），后续再扩展 RBAC。
2. 方案 B：无登录但带“管理口令”（仅局域网），所有写操作仍记录审计日志。

接口占位：

1. `POST /api/auth/login`
2. `POST /api/auth/logout`
3. `GET /api/auth/me`

### 3.3 Projects

1. `GET /api/projects`
2. `POST /api/projects`
3. `GET /api/projects/{id}`
4. `PATCH /api/projects/{id}`

可选（第一版做不做由 AI-PM 排序）：

1. `POST /api/projects/{id}/sync-git`（对接 GitHub/Gitea）

### 3.4 Agents（AI 成员）

1. `GET /api/agents`
2. `POST /api/agents`
3. `GET /api/agents/{id}`
4. `PATCH /api/agents/{id}`
5. `POST /api/agents/{id}/enable`
6. `POST /api/agents/{id}/disable`

字段必须支持（第一版重点）：

```text
access_mode = manual_codex_thread
thread_name / thread_url（可选）
permission_level（L0-L5）
token_budget（可选）
```

### 3.5 Runners

1. `POST /api/runners/register`
2. `POST /api/runners/heartbeat`
3. `GET /api/runners`
4. `GET /api/runners/{id}`
5. `PATCH /api/runners/{id}`（例如暂停、标签、能力修正）

### 3.6 Tasks

1. `GET /api/tasks`
2. `POST /api/tasks`
3. `GET /api/tasks/{id}`
4. `PATCH /api/tasks/{id}`
5. `POST /api/tasks/{id}/plan`（第一版可选：记录计划文本）
6. `POST /api/tasks/{id}/approve-plan`（第一版可选：人类确认执行）
7. `POST /api/tasks/{id}/run`（分配给 Runner/或标记为手动线程执行）
8. `POST /api/tasks/{id}/cancel`

任务事件（建议第一版就做，便于追溯）：

1. `GET /api/tasks/{id}/events`

### 3.7 Git（第一版只做必要能力）

1. `POST /api/git/branches`（创建任务分支：project + base_branch + branch_name）
2. `GET /api/git/commits`（按 project/branch 查询）
3. `GET /api/git/diff`（按 project/branch 或 commit 范围）
4. `POST /api/git/rollback`（回滚到 commit/tag，必须人类确认）

### 3.8 Logs / Artifacts（第一版可简化）

1. `POST /api/tasks/{id}/logs`（Runner 上传日志摘要；完整日志可后置）
2. `GET /api/tasks/{id}/logs`

### 3.9 Usage（token/成本）

1. `POST /api/usage/logs`（Runner/人类/AI 手动上报 usage）
2. `GET /api/usage/summary`（按项目/AI/日期）

说明：第一版允许手动上报或估算，不强依赖 OpenAI API。

### 3.10 Context Health（上下文健康）

1. `GET /api/tasks/{id}/context-health`
2. `POST /api/tasks/{id}/summarize-context`（第一版可只存“摘要文本”）

### 3.11 Handoffs（交接包）

1. `POST /api/tasks/{id}/create-handoff`
2. `GET /api/tasks/{id}/handoffs`
3. `GET /api/tasks/{id}/handoffs/{handoff_id}`

### 3.12 Approvals（人类确认）

1. `GET /api/approvals`
2. `POST /api/approvals`（创建确认请求：H0-H4）
3. `POST /api/approvals/{id}/approve`
4. `POST /api/approvals/{id}/reject`

### 3.13 Audit（审计日志）

1. `GET /api/audit/logs`

要求：所有关键写操作都写入审计。

## 4) 第一版最小数据表清单（建议）

第一版推荐最小表集（字段按“够用”原则）：

1. `users`（可选，第一版极简登录时需要）
2. `projects`
3. `repositories`（可选，若 Git 信息直接挂在 projects 上，可先不拆）
4. `agents`
5. `runners`
6. `runner_capabilities`（可选，能力可 JSON 存 runners）
7. `tasks`
8. `task_events`（建议必做）
9. `task_logs`（可选，日志也可作为 task_events 的一种类型）
10. `git_branches`（可选，若 Git 信息直接按任务存，也可先不拆）
11. `approvals`（必做，保证硬件/回滚/合并等流程有记录）
12. `usage_logs`（必做，token/成本）
13. `context_health_snapshots`（可选，第一版可只存 tasks 上的简化字段）
14. `handoff_packages`（必做，交接包）
15. `audit_logs`（必做）

建议 `tasks` 至少包含：

```text
project_id
title
module
priority
status
assignee_agent_id（或 assignees JSON）
branch_name
acceptance_criteria（JSON/文本）
execution_mode（manual_thread / runner）
human_approval_required（bool）
created_at / updated_at
```

建议 `agents` 至少包含：

```text
name
role
access_mode（manual_codex_thread / future）
thread_name / thread_url（可选）
host_runner_id（可选）
permission_level（L0-L5）
allowed_read_paths（JSON）
allowed_write_paths（JSON）
token_budget_daily（可选）
status（active/disabled）
```

## 5) 第一版明确不做的内容（禁止发散）

第一版不做：

```text
多租户/多组织/复杂 RBAC
企业 SSO/LDAP/OIDC
插件市场（只预留 provider/adapters 扩展位）
复杂自动需求路由/知识图谱/跨项目记忆
AI 自动模型竞标/自动模型路由
复杂报表系统（只做每日战报雏形）
完整 Kubernetes 部署（Docker Compose 足够）

AI 自动合并 main
AI 自动烧录/自动控制真实硬件
硬件在环自动化
商业化计费体系
```

## 6) 前端、后端、Runner 的接口边界说明

### 6.1 前端边界（Web）

前端负责：

1. 研发基地首页可视化（项目/AI/任务/Git/硬件确认/成本/告警）。
2. 管理类 CRUD（项目、AI、Runner、任务）。
3. 审查视图（diff、日志、上下文健康、交接包、审批）。
4. 危险操作必须二次确认，并展示 H0-H4 风险等级。

前端不负责：

1. 调用大模型（第一版不做自动模型调用）。
2. 执行命令或直接操作 Git（必须通过后端/Runner）。
3. 保存任何明文 token/密钥（禁止 `NEXT_PUBLIC_*`）。

### 6.2 后端边界（API）

后端负责：

1. 业务真相：项目/AI/Runner/任务/状态流转/审批/审计/成本。
2. 提供 Git 能力入口（创建分支、diff、回滚），并加人类确认点。
3. 接收 Runner 上报（心跳、日志、结果、usage）。
4. 强制执行安全规则（硬件操作默认阻断、确认点记录）。
5. 统一错误码和响应格式。

后端不负责：

1. 直接控制真实硬件。
2. 把密钥返回到前端。
3. 在第一版中强制实现复杂企业权限体系。

### 6.3 Runner 边界（执行节点）

Runner 负责：

1. 主动注册与心跳。
2. 领取任务、创建工作区、拉取代码（如需要）。
3. 执行受限命令（白名单）。
4. 回传日志摘要/结果/测试状态。
5. 任务结束清理临时凭证与工作区（按策略）。

Runner 不负责：

1. 自动控制真实硬件（默认 `ALLOW_HARDWARE_ACCESS=false`）。
2. 自主扩大权限或执行危险命令（必须被拦截）。
3. 输出任何明文 token/密钥到日志。
