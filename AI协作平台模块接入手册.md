# AI 协作平台模块接入手册

版本：v0.1  
日期：2026-04-19  
用途：说明后续开发时，应该如何在当前项目中扩展模块能力，避免重复造轮子、避免模块边界混乱

---

## 1. 这份手册解决什么问题

后面无论是人还是 AI，要继续开发这个平台，最容易遇到的问题不是“不会写代码”，而是：

1. 不知道功能该加到哪个模块
2. 不知道前后端该怎么对应
3. 不知道哪些地方能复用，哪些地方不能乱改
4. 不知道怎样扩展才不会把工程搞乱

这份手册的目标就是：

**告诉后续开发者，新增能力时优先接在哪个模块、页面、数据结构和流程上。**

## 3. projects 模块怎么扩展

### 3.1 这个模块负责什么

`projects` 是平台的顶层业务域，适合放：

1. 项目基本信息
2. 仓库信息
3. 主分支 / 开发分支
4. 项目配置
5. 项目级成员
6. 项目级协作规则

### 3.2 适合放进 projects 的新能力

适合继续加：

1. 模块责任地图
2. 整机版本快照入口
3. 项目模板
4. 项目级 requirement policy
5. 项目级 collaboration config

### 3.3 不适合放进 projects 的内容

不要塞进来：

1. 任务具体执行逻辑
2. 审批细节
3. 实验记录详情
4. 上下文健康记录

这些应该去各自模块。

### 3.4 推荐扩展方式

如果要加“模块责任地图”：

#### 后端

优先在 `projects` 扩：

1. 新增 `ProjectModule`
2. 新增 `ProjectModuleAssignment`

#### 前端

新增：

```text
/projects/[id]/modules
```

#### 与其他模块联动

1. `tasks` 关联 `module_id`
2. `agents` 显示负责模块

## 4. tasks 模块怎么扩展

### 4.1 这个模块负责什么

`tasks` 是平台最核心的业务模块，适合放：

1. 任务 CRUD
2. 任务状态流转
3. 任务负责人
4. 任务分支
5. 任务日志
6. 任务事件
7. 任务审批入口
8. 任务交接入口

### 4.2 适合继续放进 tasks 的新能力

适合继续扩展：

1. 任务接力模式
2. 子任务链
3. 任务执行策略
4. 任务验收项
5. 任务依赖关系

### 4.3 推荐扩展方式

如果要加“任务接力模式”：

#### 后端

优先扩展 `Task` 自身：

1. `parent_task_id`
2. `upstream_task_id`
3. `relay_stage`
4. `handoff_required`

#### 前端

扩展：

```text
/tasks/[id]
```

在详情页中展示：

1. 上一棒
2. 下一棒
3. 当前输入
4. 当前输出
5. 当前接手人

### 4.4 不建议做的事

不建议第一时间新开：

1. 独立工作流引擎
2. BPM 系统
3. 可视化拖拽流程编排器

现阶段直接在 `tasks` 扩展就够了。

## 5. approvals 模块怎么扩展

### 5.1 这个模块负责什么

`approvals` 负责：

1. 审批单
2. 审批状态
3. 审批备注
4. 审批责任人
5. 和高风险动作的联动

### 5.2 适合继续扩展的能力

1. 更严格的人类审批签名
2. 审批模板
3. 审批等级
4. 双人确认
5. 审批 SLA

### 5.3 推荐扩展方式

如果要加“硬件专项审批模板”：

#### 后端

不要新开系统，优先：

1. 在 `Approval` 上加 `approval_template`
2. 或新增 `ApprovalTemplate`

#### 前端

继续用：

```text
/approvals
```

增加：

1. 模板选择
2. 风险说明
3. 必填清单

### 5.4 与实验楼的关系

实验楼不应该自己发明一套审批。

原则：

1. `lab` 发起
2. `approvals` 审核
3. `audit` 留痕

## 6. handoffs 模块怎么扩展

### 6.1 这个模块负责什么

`handoffs` 负责：

1. 交接包
2. 接手确认
3. 交接目标人 / AI
4. 交接状态
5. 交接历史

### 6.2 适合继续扩展的能力

1. 自动生成交接包
2. 交接模板
3. 指定接手候选人
4. 交接版本
5. 接手评分

### 6.3 推荐扩展方式

如果要加“基于上下文红区自动建议交接”：

#### 后端

优先从 `context` 触发，最终落在 `handoffs`。

不要把交接逻辑塞回 `tasks`。

#### 前端

继续强化：

```text
/handoffs
```

### 6.4 与 context 的边界

规则：

1. `context` 负责检测和摘要
2. `handoffs` 负责交接动作

不要混成一个模块。

## 7. context 模块怎么扩展

### 7.1 这个模块负责什么

`context` 负责：

1. 上下文健康记录
2. token 占用
3. 上下文等级
4. 摘要
5. 建议动作

### 7.2 适合继续扩展的能力

1. 自动阈值提醒
2. 上下文趋势图
3. 自动摘要策略
4. 建议切换模型
5. 建议接手人

### 7.3 推荐扩展方式

如果要加“上下文趋势图”：

#### 后端

扩展 `ContextHealthRecord`

#### 前端

继续扩展：

```text
/context-health
```

### 7.4 不建议做的事

不要让 `context` 自己承担：

1. 审批
2. 交接确认
3. 知识沉淀最终存储

它只负责“判断和建议”。

## 8. knowledge 模块怎么扩展

### 8.1 这个模块负责什么

`knowledge` 负责：

1. 沉淀知识条目
2. 来源追踪
3. 与需求 / 任务 / 交接的关联
4. 长期知识管理

### 8.2 适合继续扩展的能力

1. 接口契约中心
2. ADR 决策记录
3. 芯片手册知识页
4. 原理图说明页
5. 硬件问题 FAQ

### 8.3 推荐扩展方式

如果要加“接口契约中心”：

优先不要新做独立系统，可以先放在 `knowledge` 下的子分类里。

等规模变大，再拆独立 `contracts` 模块。

为什么：

1. 第一阶段最需要的是沉淀
2. 不一定一上来就要完整契约管理平台

### 8.4 什么时候单独拆 contracts 模块

当接口契约开始需要：

1. 版本控制
2. owner / consumer 管理
3. 结构化 schema
4. 自动校验

这时再拆更合理。

## 9. lab 模块怎么扩展

### 9.1 这个模块负责什么

`lab` 负责：

1. 实验楼页面
2. 真机相关任务说明
3. 高风险动作入口
4. 与审批门岗联动

### 9.2 适合继续扩展的能力

1. 硬件调试卡
2. 实验记录流
3. 板卡状态
4. 串口日志入口
5. 测量结果录入
6. 真机验证清单

### 9.3 推荐扩展方式

如果要加“硬件调试卡系统”：

#### 第一阶段

先挂在 `lab` 下最稳：

```text
/lab/debug-cards
/lab/debug-cards/[id]
```

#### 第二阶段

如果调试卡越来越复杂，再独立成 `debug_cards` 模块。

### 9.4 不建议做的事

不建议让 `lab` 同时承担：

1. 审批最终存储
2. 审计系统
3. 设备注册系统

这些都应该通过模块联动实现。

## 10. messages / discussions 怎么扩展

### 10.1 当前最需要的原则

后面一定要把消息系统收口。

建议目标：

1. 项目留言
2. 任务留言
3. 需求留言
4. 审批留言
5. 交接留言

都走统一消息模型。

### 10.2 推荐扩展方式

不要再继续同时扩：

1. `messages`
2. `collaboration messages`
3. `requirement messages`

建议后续逐步统一到：

**通用消息模型 + 不同实体挂载**

### 10.3 discussions 页该做什么

`/discussions` 应该成为：

1. 项目总讨论区
2. 可以切到任务 / 需求 / 审批 / 交接分视图
3. 统一查看正式记录

不要让它成为另一套“本机草稿聊天室”。

## 11. auth / members / collaborators 怎么扩展

### 11.1 auth 负责什么

1. 注册
2. 登录
3. 会话
4. 邀请
5. 项目成员

### 11.2 collaborators 负责什么

1. 协作者工位总览
2. 电脑节点
3. AI provider
4. 线程工位

### 11.3 推荐未来的清晰边界

1. `login`：身份进入平台
2. `members`：项目成员和邀请
3. `collaborators`：成员 + AI + 电脑 + 线程的协作编排

这样新用户容易理解。

### 11.4 推荐扩展方式

如果要加“多电脑绑定”：

1. 用户身份仍在 `auth`
2. 电脑与工位关系落在 `collaboration`
3. 页面呈现在 `collaborators`

不要把电脑信息塞回登录页。

## 14. 推荐给其他 AI 的最短接入提示

后面如果别的 AI 接手，你可以直接把这段给它：

```text
当前工程：
- 前端：apps/web（Next.js 14）
- 后端：apps/api（FastAPI）
- 共享层：packages/shared
- 部署：infra

加功能时优先扩现有模块，不轻易新开系统。

推荐映射：
- 项目级配置 -> projects
- 任务与接力 -> tasks
- 审批与高风险动作 -> approvals
- 交接动作 -> handoffs
- 上下文统计 -> context
- 知识沉淀 / 接口契约 -> knowledge
- 硬件调试 / 实验 -> lab
- 多电脑 / 多 provider / 线程工位 -> collaboration + collaborators
- Runner 执行能力 -> runners
- 留言和讨论 -> messages / discussions
```

这样它会更快进入状态。

## 15. 最终建议

继续开发这个平台时，最稳的方式不是一直新建模块，而是：

1. 先找到最接近的现有业务域
2. 先扩展现有页面和模块
3. 等这个能力真的长大，再拆独立系统

要永远记住：

### 15.1 不是所有功能都值得单独成系统

很多能力第一阶段只是“某个模块的子功能”。

### 15.2 页面只是表象，主链才是关键

主链永远是：

```text
项目
-> 成员 / AI / Runner
-> 任务
-> 需求 / 审批 / 交接
-> 审计 / 知识 / 实验
```

### 15.3 对嵌入式和机器人最有价值的，不是更多页面，而是更稳的协作结构

真正值得优先做的，永远是：

1. 硬件调试卡
2. 模块责任地图
3. 整机版本快照
4. 接口契约中心
5. 实验记录流
6. 任务接力模式

一句话总结：

**这个平台后面最好的扩展方式，不是越分越散，而是在现有模块边界上长出更适合机器人和嵌入式研发的能力。**
