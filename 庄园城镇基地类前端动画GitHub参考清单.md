# 庄园 / 城镇 / 基地类网页游戏前端参考清单

## 1. 这份清单的用途

这份清单不是为了找“好看截图”，而是为了给当前平台确定一条真正像网页游戏的前端实现路线。

当前项目不是普通后台，也不是纯娱乐农场游戏，而是：

- 单人公司视角的项目经营平台
- 多项目切换
- 多电脑客户端接入
- 多 AI 工位协作
- 任务、需求、审批、交接、实验、Git 交付联动

所以最适合的方向不是“照抄 QQ 农场”，而是：

**用成熟网页游戏的主界面组织方式，承载真实开发和协作功能。**

核心要求是：

1. 地图就是主界面
2. 大部分操作不跳页
3. 邮箱、聊天、设置、任务、资源、状态都在游戏界面内
4. 表单和详情只作为弹层、抽屉或二级面板出现

---

## 2. 当前前端应该学什么，不该学什么

### 应该学

- 部落冲突 / 养成类页游的主界面布局
- 城镇经营 / 基地经营游戏的功能组织
- 大地图 + 顶部资源条 + 左右侧功能入口 + 底部工具栏
- 点击建筑后弹出功能面板，而不是切页
- 用建筑、区域、地块表达功能模块

### 不该学

- 把后台页面直接塞进地图下面
- 用一堆说明文字占屏幕
- 一屏里塞满卡片和表单
- 把地图当装饰图，真正操作还在页面下半段
- 把旧页面原样保留，只在上面加个“农场壳”

## 3. 最值得参考的开源方向

## 3.1 地图与交互底座

### Phaser

- GitHub: [phaserjs/phaser](https://github.com/phaserjs/phaser)
- 适合用来做：
  - 地图场景
  - 小人移动
  - 镜头跟随
  - 建筑触发区
  - 交互范围检测

结论：
- 这是当前项目最适合的场景引擎底座。
- 后面“项目地图”“项目广场”“车间地图”都可以共用这层。

### Phaser + React 混合模板

- GitHub: [blopa/top-down-react-phaser-game-template](https://github.com/blopa/top-down-react-phaser-game-template)
- 适合借：
  - 顶视角结构
  - Phaser 与 React 面板的分层
  - 游戏场景里触发 UI 面板

结论：
- 非常适合我们这种“地图是游戏层，表单是 React 层”的产品。

### Farmhand

- GitHub: [CoherentLabs/GameUIComponents](https://github.com/CoherentLabs/GameUIComponents)
- 适合借：
  - 游戏化按钮
  - 面板
  - HUD 结构
  - 工具栏风格

结论：
- 适合用来约束“像游戏的 UI 组件层”，尤其是顶部、底部、浮层按钮风格。

## 4. 参考“部落冲突”后，当前产品应该怎么组织

部落冲突这种成熟游戏有几个关键点，当前平台应该借的是这些：

### 4.1 主界面只有一个核心场景

不是一堆页面切来切去，而是：

- 登录后进入主城 / 主基地 / 主农场
- 所有关键操作都从这个主界面出发
- 需要深入编辑时才开弹层或进二级页

映射到当前平台：

- 登录后：项目广场
- 选中项目后：进入该项目自己的地图

### 4.2 顶部必须是短资源条，不是大段说明

部落冲突式界面不会让一屏上来先读说明文案。

顶部应该只有高频指标：

- Token
- AI 数
- 在线电脑数
- 活跃任务
- 阻塞
- 待审批
- 新邮件 / 新邀请

### 4.3 左右两侧是功能入口，不是说明书

建议这样映射：

左侧：
- 邮箱
- 任务引导
- 新邀请
- 当前主线

右侧：
- 聊天
- 系统状态
- 角色属性
- 最近回执

### 4.4 底部是工具栏，不是传统导航页签

底部适合放：

- 建造 / 布置
- 任务
- AI
- 电脑
- 仓库
- 设置
- 地图切换

### 4.5 建筑点击后，不跳到长页面

正确方式：

- 点击需求信箱 -> 打开邮箱弹层
- 点击任务田块 -> 打开任务面板
- 点击 AI 工位站 -> 打开工位管理面板
- 点击电脑车间 -> 打开 Runner / 电脑控制面板
- 点击留言小院 -> 打开聊天与留言面板
- 点击审批门岗 -> 打开审批弹层
- 点击交付工坊 -> 打开 Git 操作面板

不是：

- 点一下建筑
- 滚动到下面
- 再看一整页表单

## 5. 当前平台功能如何映射到游戏模块

### 地图主区

- 项目大门：项目概览、成员、邀请、切换项目
- 需求信箱：需求、澄清、需求到任务
- 任务田块：任务创建、状态推进、派工
- AI 工位站：AI 提供方、模型、工位
- 电脑车间：电脑节点、Runner、命令中转
- 留言小院：讨论、聊天、交接、上下文
- 审批门岗：高风险动作审批
- 交付工坊：Git、分支、同步、回滚
- 实验楼：硬件实验、真机验证、记录
- 档案仓：审计、知识、复盘

### 游戏 HUD

- 等级：项目成熟度或个人经营等级
- 经验：最近推进量 / 交付量
- Token：当天或当周消耗
- AI 数：当前可用工位数量
- 在线电脑：当前在线执行节点
- 声望：可映射为交付成功数或通过审批率

注意：
- 不能乱造 RPG 数值
- 所有“属性”都必须对应真实业务信号

## 6. 现在最该做的不是贴图，而是重构交互层

当前项目前端下一阶段应按这个顺序落地：

1. 登录后主页改成“项目广场”
2. 单项目页改成完整主界面
3. 删除地图下方大段内容堆叠
4. 把核心功能改成游戏内弹层
5. 再接入小人移动和建筑交互
6. 最后补真实动画与素材

也就是说：

**先做网页游戏的信息架构，再做网页游戏的美术与动画。**

---

## 7. 对当前代码改造的硬规则

后面所有前端改造都要遵守：

1. 不再让地图下面堆大段表单
2. 不再让项目页先显示长说明文案
3. 不再让“设置 / 邮箱 / 聊天 / 任务 / 工位”分散成主流程外的多个大页面
4. 所有主操作优先放进游戏界面内完成
5. 二级页面只保留给复杂配置、详细审计、长文本编辑
6. 旧页面继续兼容，但必须降级为二级入口

---

## 8. 当前最有价值的实现路线

如果从产品和工程都不跑偏的角度看，当前最正确的路线是：

**把 AI 协作平台做成“一个可经营、可切换项目、可调度 AI 和电脑的网页游戏式研发基地”。**

不是：

- 后台换皮
- 地图贴卡片
- 把表单塞进农场

而是：

- 游戏界面承载主要操作
- 地图就是导航
- 建筑就是模块
- HUD 就是业务状态
- 工具栏就是常用动作
