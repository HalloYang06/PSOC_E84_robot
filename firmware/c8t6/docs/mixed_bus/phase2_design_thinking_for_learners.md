# 设计思路（面向学习者）：为什么 v1 选“电机CANopen + F103私有协议”

## 1. 先定目标，再选复杂度
这版的核心目标不是“协议最统一”，而是“尽快稳定联调”。  
你的系统里同时有：
- 5 个电机（可统一走 CANopen）
- 一个资源紧张的 F103 传感节点
- 一个 PSoC 主控（可做协议汇聚）
- 一个 NanoPi/ROS 上层

如果第一版就要求每个节点都跑完整 CANopen，工程风险会明显增大。  
所以 v1 采用分层收敛策略：
- 电机侧：统一 CANopen，便于维护
- 传感侧（F103）：私有轻量帧，保证资源和时序可控
- 主控（PSoC）：承担协议汇聚和统一语义出口

## 2. 为什么硬件过滤很关键
学习者最容易忽略的一点是：  
“CAN 总线上能收到”不等于“CPU 扛得住持续处理中断”。

5 个电机高频运行时，F103 如果把无关报文也收进来，主循环和采样都会被打断。  
所以我们在 bxCAN 过滤器上直接只放行 `0x7C0`，其余在硬件层丢弃。  
这个设计通常比软件层 if/else 过滤更稳、更省 CPU。

## 3. 为什么 ACK 里要带 seq
在现场网络中，命令重发是常态，不是异常。  
`cmd_id + seq` 的意义是让 F103 能识别“同一条命令的重发”，直接回放上次 ACK，避免副作用重复执行。  
这属于“幂等设计”，是工业总线里非常实用的习惯。

## 4. 为什么先不做分片长包
v1 的传感与控制参数都能装进 8 字节，先不用分片。  
好处是：
- 调试简单，抓包一眼看懂
- 代码路径短，故障面小
- 适合快速闭环验证

等你们系统稳定后，再在 v2 加入分片和更复杂诊断负载会更稳妥。

## 5. 给学习者的实践顺序建议
1. 先用抓包工具确认总线上全是经典 CAN 帧。  
2. 再验证 F103 只接收 `0x7C0`（中断率明显下降）。  
3. 再测命令 ACK：正常、重发、错误参数三条路径。  
4. 最后上长时间运行（30分钟+）看错误计数是否可解释。

## 6. 你可以这样理解这套架构
- PSoC：总线调度中枢
- 电机：标准执行器（CANopen）
- F103：高频传感器协处理节点（轻协议）
- ROS：策略与记录层

这不是“退而求其次”，而是“按系统风险分配复杂度”的工程化做法。
