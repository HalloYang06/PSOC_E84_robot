# M55 LVGL 中文字库资源预算与整改路线

日期：2026-07-17

## 1. 当前结论

当前主要问题不是 Flash 不够，而是字体覆盖策略不稳定：

- `applications/rehab_wifi_font.c` 只包含固定 UI 使用的一小组中文字符，约 113 个显式中文字形。
- 它回退到 `lv_font_simsun_16_cjk`，后者约覆盖 1000 个常用 CJK 字符，仍不能覆盖任意小智动态回复。
- 当前两个静态字体合计约 226 KiB，主要位于只读代码区，对永久 RAM 影响很小。
- M55 应用 XIP 区为 8 MiB；当前固件加载量约 1.66 MiB，静态增加 0.5-1.0 MiB 字库仍有空间。
- 内部 RAM 和 `gfx_mem` 更紧张，不应把完整 TTF、glyph cache 或字体位图搬入图形内存。

因此建议使用“源代码字符清单 + 可复现静态子集 + 缺字测试”，而不是继续手工猜字符，也不建议直接启用完整动态 TTF。

## 2. 当前内存边界

链接脚本关键区域：

| 区域 | 地址/大小 | 字库约束 |
| --- | --- | --- |
| `m55_nvm` | `0x60580000`, 8 MiB | 静态 const 字体的首选位置 |
| `m55_trailer` | `0x60D80000`, 256 KiB | 不得侵入 |
| `m55_data_INTERNAL` | 256 KiB | 不放大字库和大 glyph cache |
| `m55_data_secondary` | 1.375 MiB | 运行堆、线程和模型共享压力较大 |
| `gfx_mem` | 3 MiB | 已承载双显示 buffer、framebuffer/VG-Lite 资源，不放字体 |

当前 `LV_USE_FREETYPE=0`；`LV_USE_TINY_TTF=1`，但 `LV_TINY_TTF_FILE_SUPPORT=0`，glyph cache 配置为 256。当前 UI 实际仍使用编译期 LVGL C 字体。不要因为 tiny_ttf 编译存在，就认定运行时完整中文字库已经可用。

## 3. 为什么不能直接上完整 TTF

- 动态 glyph raster 和 cache 会消耗 RAM，并可能造成堆碎片。
- 从 SMIF0/FAL 动态读取字体会与现有文件系统、Wi-Fi 配置写入和 GPU lifecycle 产生新的并发边界。
- VG-Lite、LCD 和字体 cache 的组合负载必须重新验证，风险大于静态字体。
- 任意小智长回复不适合完整显示在小尺寸 LCD；UI 应显示受控摘要，完整内容保留在 App。

## 4. 推荐覆盖等级

### 等级 A：固定 UI 精确子集

扫描所有会进入 LVGL label 的固定字符串，生成确定性 manifest。适合菜单、状态、错误和配网文案。目标是固定 UI 缺字为 0。

### 等级 B：3500 常用汉字

建议 Noto Sans SC，18 px，2 bpp，预计增加约 300-500 KiB。用于短状态摘要和常见回复，Flash 预算可接受。

### 等级 C：GB2312 6763 字

预计约 600-800 KiB。只有缺字统计证明 3500 字不足时再采用。

不建议直接加入约 2 万 CJK 字符。它会扩大构建、镜像和回归范围，却仍不能覆盖所有 Unicode 文本。

## 5. 实施顺序

1. 增加字符提取工具，只扫描明确进入 LCD 的字符串和一份人工维护的业务补充表。
2. 生成 UTF-8 排序、去重的 `font_symbols.txt`，提交该 manifest。
3. 增加缺字测试：固定 UI 字符必须全部能从主字体或 fallback 查到 glyph。
4. 用同一 Noto Sans SC 文件、固定 `lv_font_conv` 版本和固定参数生成字体。
5. 先生成 18 px、2 bpp 的 3500 字候选，保留现有版本做 A/B 截图。
6. 确认候选覆盖后，合并为一个主要 CJK 字体，并评估删除重复 SimSun fallback，预计可回收约 180 KiB。
7. 只有缺字日志仍超标，才升级到 GB2312 级别。

字体生成文件应由脚本产出，不手工编辑大 C 数组。

## 6. 构建门槛

- 链接后应用末地址必须小于 `0x60D80000`。
- 距离 `m55_trailer` 至少保留 1 MiB 应用余量。
- 单次字体改动造成的 Flash 增量不超过 1.5 MiB。
- `.data/.bss` 不应因静态字体明显增长。
- 不允许把字体 section 放入 `gfx_mem` 或共享 IPC 区。

## 7. 功能和稳定性验证

### 固定画面

- 配网、连接、断开、错误、诊断、小智监听/思考/播放等全部页面截图。
- 对 manifest 中每个字符做 glyph lookup，缺失数必须为 0。
- 检查 baseline、行高、中文/数字/ASCII 混排和换行。

### 动态文本

准备常用康复、网络、语音和错误语料，记录缺失字符，不允许静默显示方框。超出 LCD 覆盖范围的长回复应截断为受控摘要，并在 App 展示全文。

### 压力回归

- 50 次冷启动，无白屏、HardFault 或 SMIF timeout。
- Wi-Fi、小智、NPU/CPU 推理和 LVGL 同时运行 30 分钟。
- 无持续超过 33 ms 的 LVGL flush 退化。
- FAL 写入 Wi-Fi/token 时显示仍能恢复。
- 比较改动前后 heap 最低余量和 LVGL 内存监控。

## 8. 提交拆分

1. 字符 manifest 和提取/缺字测试。
2. 仅增加候选静态字体，不切换默认字体。
3. 仅切换 UI 字体并补截图结果。
4. 验证通过后，单独删除重复 fallback。

不要把字体替换与 LCD 驱动、VG-Lite、FAL、NPU 或语音控制改动放在同一提交中。
