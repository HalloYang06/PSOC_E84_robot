# M55 Ethos-U55 NPU 可行性与验证路线

日期：2026-07-17

## 1. 当前结论

当前固件不能因为定义了 `ETHOSU55` 就认定模型已经在 NPU 上运行。工程同时存在三种不同状态：

1. 芯片和 PDL 中包含 Ethos-U55 驱动。
2. `applications/ifx_deepcraft` 中包含 Vela 生成的模型和另一套推理支持代码。
3. 当前 RT-Thread TFLM package 的 `tensorflow/lite/micro/kernels/ethosu.cc` 中，`Register_ETHOSU()` 仍直接返回 `nullptr`。

因此现在直接打开 NPU 路径，最可能得到初始化失败、算子注册失败或静默回退，不能证明硬件加速生效。

## 2. 现有可利用资源

- 目标硬件：Ethos-U55。
- 驱动基础：`libraries/components/mtb-device-support-pse8xxgp/.../COMPONENT_U55`。
- DeepCraft 适配：`applications/ifx_deepcraft/source`。
- 已生成模型：`AM_LSTM_model_bin`，模型长度 `218096` 字节，arena `18432` 字节。
- PMU 支持：`ethosu_pmu.c`，可用于证明 NPU 实际执行并采集周期/事件。

首个候选应使用已经过 Vela 处理的 AM_LSTM，而不是先改当前 EMG intent 小模型。后者即使能运行，模型太小也可能因调度、复制和 cache 成本得不到收益。

## 3. 启用前必须修复的问题

### 3.1 初始化结果不能被忽略

`applications/ifx_deepcraft/source/va_core/va_ml_model.c` 调用 `mtb_ml_init(NPU_PRIORITY)` 时没有检查结果；`mtb_ml.c` 还需要确认失败时不会增加 `mtb_ml_init_state`。否则上层可能把失败状态当作已经初始化。

验收要求：

- 初始化失败返回明确错误。
- 失败后状态仍为未初始化。
- Shell 诊断能看到 backend、init result、invoke count、timeout count、reset count 和 fallback count。

2026-07-17 已完成本阶段的错误传播与宿主故障注入测试，但不代表 NPU 已启用。`mtb_ml_ethosu_init()` 在 IRQ 注册成功、U55 上电后仍可能于 `ethosu_init()` 内失败；现有驱动没有经过验证的“部分初始化反向清理”契约，直接调用完整 `ethosu_deinit()` 可能操作未完成初始化的驱动对象。因此本次没有猜测性地增加 IRQ/电源/驱动回滚，实验门禁继续保持关闭。后续接入真实 kernel 前，必须先用厂商匹配版本确认部分初始化失败时的合法清理顺序并补硬件故障注入。

### 3.2 不能在语音或 IPC 线程中同步等待 NPU

使用单独 `npu_worker` 和固定大小请求槽：

- 生产者只做有界复制/引用和零等待入队。
- 同一时刻只允许一个 NPU job。
- invoke 必须有有限超时。
- 超时后执行 soft reset，并切换 CPU fallback。
- 不允许在 M33/M55 IPC 消费、音频 capture/TTS、LVGL handler 中等待 NPU。

### 3.3 内存和 cache 属性必须明确

NPU 是独立 bus master，不能假设 CPU cache 或 SMIF0 guard 自动覆盖它。

- command stream、arena、输入和输出必须满足对齐及可访问地址要求。
- NPU 访问前 flush，完成后 invalidate，不能依赖空 weak hook。
- 第一阶段把 NPU 工作集放内部 RAM；不要让 NPU 直接读取 `0x60000000-0x60FFFFFF` 的 SMIF0 XIP 数据。
- 若未来必须读 XIP 权重，需要把 NPU 纳入现有 SMIF0 lifecycle/quiesce 设计，再做并发擦写验证。

## 4. 分阶段实施

### 阶段 A：CPU 基线，不改 backend

固定同一批输入，记录：

- 预热 20 次，正式运行 1000 次。
- 特征提取、invoke 和完整 pipeline 的 P50/P95/P99/max。
- 分类结果、输出 tensor、heap 最低余量和线程最大栈占用。
- audio drop/underrun、IPC fail/drop、LVGL 最大 flush 时间。

这一阶段先补诊断，不改变现有输出和控制行为。

### 阶段 B：初始化和单样本 smoke test

1. 修复 `mtb_ml_init()` 错误传播和状态计数。
2. 接入与当前 TFLM 版本匹配的真实 Ethos-U custom op registration。
3. 只加载 AM_LSTM，运行一份 golden input。
4. PMU 必须出现有效计数，证明不是 CPU 或空实现。
5. 任一错误立即禁用 NPU backend，并回退 CPU。

### 阶段 C：独立 worker 和超时恢复

加入固定队列、单 worker、有界等待、soft reset 和 CPU fallback。先通过故障注入：未初始化、队列满、IRQ 不到、invoke timeout、输出异常。

### 阶段 D：1000 次 A/B

CPU 和 NPU 必须使用完全相同的 golden 输入顺序。每个样本比较类别和量化输出，不能只看平均耗时。

### 阶段 E：组合负载

在 Wi-Fi、小智录音/TTS、M33/M55 IPC 和 LVGL 动画同时工作时运行至少 30 分钟。组合负载通过后，才考虑迁移唤醒模型；EMG intent 最后评估。

## 5. 通过门槛

| 项目 | 门槛 |
| --- | --- |
| 正确性 | 100% golden 样本完成；类别一致；量化输出差异不超过事先定义的 2 LSB |
| 稳定性 | 0 HardFault、0 NPU timeout、0 无法恢复错误 |
| 性能 | 完整 pipeline P99 至少降低 20%，不能只比较单个 invoke 平均值 |
| 实时性 | audio drop/underrun、IPC fail/drop 不高于 CPU 基线 |
| 显示 | 无白屏；LVGL flush 不出现持续超过 33 ms 的退化 |
| 内存 | heap/stack 最低余量满足现有安全门槛，arena 不使用运行时反复分配 |
| 回退 | NPU 初始化或 invoke 失败后，CPU backend 能继续给出结果 |

如果 AM_LSTM 都不能达到这些门槛，不应继续把更多模型迁到 NPU。
