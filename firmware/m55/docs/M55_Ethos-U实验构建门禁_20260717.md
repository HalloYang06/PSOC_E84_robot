# M55 Ethos-U 实验构建门禁

## 当前结论

`M55_ETHOSU_EXPERIMENTAL` 默认值为 `0`。默认固件继续使用现有 Edge Impulse CPU 后端，不改变源文件选择、编译宏和链接库。

目前显式设置 `M55_ETHOSU_EXPERIMENTAL=1` 会在 SCons 配置阶段失败。这是有意的 fail-closed 门禁，不代表 NPU 已经可用。

## NO-GO 原因

1. 官方小智 Edge Impulse INT8 模型不包含 `ethos-u` 自定义算子。
2. 当前 intent INT8 模型不包含 `ethos-u` 自定义算子。
3. 现有 AM_LSTM 是唯一已确认由 Vela 3.11.0 生成且包含单个 `ethos-u` v1 自定义算子的候选模型。
4. 当前 RT-Thread TFLM package 的 `Register_ETHOSU()` 返回 `nullptr`，无法执行 Vela 模型。
5. NPU 初始化失败传播、单工作线程、超时恢复、cache 一致性和 CPU fallback 尚未完成验证。

因此不能仅通过定义 `ETHOSU55`、加入 U55 驱动或链接另一套官方实现来宣称 NPU 已启用。两套 TFLM/驱动实现混合链接还会引入重复符号和生命周期不一致风险。

## 小步放行顺序

1. 保持本门禁关闭，冻结 CPU golden input、分类输出和 p50/p95/p99 基线。
2. 修复并测试 `mtb_ml_init()` 失败传播，不接入业务推理路径。
3. 适配一个真实 `Register_ETHOSU()`，只用 AM_LSTM 跑单次 golden smoke，并保留 CPU fallback。
4. 增加唯一 NPU worker、有限等待、超时软复位和连续失败熔断。
5. 完成至少 1000 次 A/B：分类完成率 100%、输出误差不超过 2、零超时/HardFault，p99 至少改善 20%。
6. 同时确认音频丢帧、IPC 丢包、LVGL flush/TTS underrun 不回退后，才允许把构建门禁改为实验可用。

模型和构建门禁由 `tools/test_m55_ethosu_artifact_gate.py` 持续检查，避免 CPU 模型被误标为 NPU 模型或未经准备的 NPU 路径被意外打开。
