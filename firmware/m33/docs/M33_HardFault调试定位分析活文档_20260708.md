# M33 HardFault 调试定位分析活文档

日期：2026-07-08

用途：主线程逐步调试 M33 HardFault 时，本文件用于同步记录证据链、地址反查、判断逻辑和后续排查方向。目标不是只给结论，而是让后续每一次 fault 都能按同一套方法拆解。

## 1. 当前调试结论摘要

当前已确认的核心结论：

1. 这次 fault 不是普通 RT-Thread Non-secure HardFault handler 直接接住的。
2. `g_rt_hw_fault_dump` 没有写入有效 magic，说明没有顺利进入 RT-Thread 的 C 级 fault dump。
3. Cortex-M33 TrustZone 配置让 HardFault 先进入 Secure 侧 handler。
4. Secure handler 入口断点能命中，说明异常确实被 Secure 接管。
5. 通过 Secure 状态读取 `MSP_NS / PSP_NS / CONTROL_NS`，拿到了真正的 Non-secure 异常栈。
6. 从 Non-secure 异常栈 `+0x18` 取到 stacked PC。
7. stacked PC 反查后落在 RT-Thread SysTick / timer / scheduler 路径，而不是 CAN 驱动本身。

简化链路：

```text
Non-secure RT-Thread app 触发 fault
        |
        v
HardFault 被 TrustZone 路由到 Secure handler
        |
        v
RT-Thread g_rt_hw_fault_dump 没写入
        |
        v
从 Secure handler 读 MSP_NS
        |
        v
解析 Non-secure 异常栈
        |
        v
stacked PC -> addr2line/objdump -> RT-Thread SysTick 调度路径
```

## 2. 为什么先看 `g_rt_hw_fault_dump`

项目里有一个辅助 dump 结构，主线程曾尝试读取：

```text
g_rt_hw_fault_dump
```

预期：

```text
magic = FA17FA17
```

如果 magic 存在，说明 RT-Thread 的 fault C 函数已经执行，并保存了异常栈。这样可以直接从 dump 里取 stacked PC。

但当前结果是：

```text
g_rt_hw_fault_dump 全 0
```

这说明：

```text
不是“RT-Thread 已经捕获 fault 但串口没打印”
而是“异常没有顺利进入 RT-Thread 的 fault dump 路径”
```

这一步非常关键，因为它决定后续不能只盯 RT-Thread handler。

## 3. 为什么 RT-Thread HardFault 断点没用

主线程查到过符号：

```text
HardFault_Handler              = 0x08398292
rt_hw_hard_fault_exception     = 0x0839834c
```

这些地址属于当前 Non-secure RT-Thread app。

给这些地址下硬件断点后没有命中，反而出现 double fault / lockup 迹象。

这说明：

```text
CPU 没有跳到 0x08398292 这个 Non-secure HardFault_Handler
```

此时应该马上检查：

```text
当前 VTOR 指向哪里
HardFault vector 实际指向哪里
异常是否被 Secure 世界接管
```

## 4. VTOR 与向量表证据

主线程读到当前 Secure VTOR 指向一张 RAM vector table，并发现 HardFault vector 实际不是：

```text
0x08398293    RT-Thread HardFault_Handler | Thumb bit set
```

而是：

```text
0x181042bb    Secure 固件里的 handler | Thumb bit set
```

因此判断：

```text
HardFault 被 Secure handler 接管
```

注意地址最低 bit：

```text
0x08398293 实际代码地址是 0x08398292，最低 bit 表示 Thumb 状态
0x181042bb 实际代码地址是 0x181042ba，最低 bit 表示 Thumb 状态
```

所以后续给 Secure handler 下断点应使用：

```text
bp 0x181042ba 2 hw
```

## 5. TrustZone 路由为什么会这样

配置中出现过：

```text
SCB_AIRCR_BFHFNMINS_VAL = 0
```

含义：

```text
BusFault / HardFault / NMI 目标状态是 Secure state
```

所以即便错误源头在 Non-secure RT-Thread app，HardFault 也会被 Secure handler 接走。

这解释了两个现象：

1. RT-Thread fault dump 没写入。
2. RT-Thread HardFault handler 断点不命中。

当前工程使用预签名 Secure 固件，例如：

```text
tools/edgeprotecttools/cm33_s_signed_fw/proj_cm33_s_signed.hex
```

所以只改 Non-secure 工程里的 `partition_ARMCM33.h` 不一定会改变板子上真实运行的 Secure fault 路由。

## 6. Secure handler 断点证据

主线程在 Secure handler 入口下断点：

```text
bp 0x181042ba 2 hw
```

然后 reset run 后能断住。

这证明：

```text
异常确实进入了 Secure HardFault 入口
```

读到：

```text
LR = 0xffffffa1
```

这个 EXC_RETURN 模式说明和 Secure/Non-secure 异常返回有关，进一步支持“Secure handler 正在处理来自 Non-secure 的 fault”。

## 7. 如何拿真正的 Non-secure 异常栈

因为异常被 Secure handler 接管，所以普通：

```text
reg msp
reg psp
```

读到的是 Secure 侧上下文，不一定是真正 RT-Thread 现场。

要读：

```text
reg msp_ns
reg psp_ns
reg control_ns
```

当前现场读到：

```text
CONTROL_NS = 0
MSP_NS     = 0x240fcf18
```

判断规则：

```text
CONTROL_NS bit[1] = 0 -> Non-secure 使用 MSP_NS
CONTROL_NS bit[1] = 1 -> Non-secure 使用 PSP_NS
```

所以本次真正的 Non-secure 异常栈在：

```text
0x240fcf18
```

## 8. 异常栈怎么拆

Cortex-M 硬件异常栈基本格式固定：

```text
offset  内容
0x00    r0
0x04    r1
0x08    r2
0x0C    r3
0x10    r12
0x14    lr
0x18    pc
0x1C    xpsr
```

因此：

```text
stacked LR   = *(MSP_NS + 0x14)
stacked PC   = *(MSP_NS + 0x18)
stacked xPSR = *(MSP_NS + 0x1C)
```

当前现场解析出：

```text
stacked PC = 0x0839c1e0
stacked LR = 0x0839c555
xPSR       = 0x6100000f
```

其中：

```text
xPSR 低位异常号 = 0x0f = 15
```

Cortex-M 异常号 15 是：

```text
SysTick
```

所以判断：

```text
HardFault 发生时，Non-secure app 正在处理 SysTick 中断
```

## 9. 地址反查结果怎么理解

使用：

```powershell
arm-none-eabi-addr2line.exe -e .\rt-thread.elf -f -C 0x0839c1e0 0x0839c555 0x08398964
```

以及：

```powershell
arm-none-eabi-objdump.exe -d -S .\rt-thread.elf --start-address=0x0839c180 --stop-address=0x0839c600
```

主线程判断当前落点在 RT-Thread tick/timer/scheduler 路径，例如：

```text
_thread_timeout()
rt_schedule_insert_thread()
```

这不是说 RT-Thread 内核一定有 bug，而是说明：

```text
SysTick 扫描/唤醒线程时，访问到了已经损坏的线程/定时器/链表数据
```

常见真实原因：

- 线程栈溢出，踩坏 `rt_thread` 对象。
- 数组越界，踩坏 RT-Thread 内核链表。
- 定时器对象生命周期错误。
- ISR 里调用了非 ISR-safe API。
- rehab 三路数组下标错误。
- CAN 回调里直接做了过重逻辑。
- 内存区安全属性或地址映射异常。

## 10. 为什么不能直接说是 CAN 坏了

现象上可能是：

```text
CAN 发了几帧
然后系统倒了
```

但 fault 现场显示：

```text
正在 SysTick / scheduler 路径
```

所以更合理的解释是：

```text
CAN 或 rehab 逻辑前面写坏了某个内存对象
SysTick 后续调度时刚好踩到坏对象
于是 fault 暴露在调度路径
```

这类问题叫“破坏点”和“崩溃点”分离。

破坏点可能在：

```text
CAN RX 解析
rehab mode apply
三路 assist/resist 状态更新
motor feedback 回调
线程/队列/定时器操作
```

崩溃点显示在：

```text
_thread_timeout()
rt_schedule_insert_thread()
SysTick_Handler()
```

## 11. 结合本项目，优先怀疑点

### 11.1 三路助力/阻力数组下标

当前三路目标是 joint 4/5/6，对应 mask：

```text
0x38
```

危险点：

```c
state[joint_id]
```

如果 `joint_id` 是 4/5/6，而数组长度按 `CONTROL_MOTOR_JOINT_COUNT` 开，必须确认数组至少覆盖这些 index，或者使用 `joint_id - 1` 并做边界检查。

推荐检查：

```c
index = joint_id - 1;
if (index >= CONTROL_MOTOR_JOINT_COUNT)
{
    return -RT_EINVAL;
}
```

### 11.2 mask 和 index 混用

mask 是 bit，不是 index。

例如：

```text
joint 4 -> bit = 1 << 3 = 0x08
joint 5 -> bit = 1 << 4 = 0x10
joint 6 -> bit = 1 << 5 = 0x20
mask = 0x38
```

数组 index 通常是：

```text
joint 4 -> index 3
joint 5 -> index 4
joint 6 -> index 5
```

不能写成：

```c
array[bit]
array[mask]
```

这会非常容易越界。

### 11.3 rehab service 状态结构体

重点查：

```text
active_joint_mask
assist_engaged_mask
assist_state[]
resist_state[]
status copy
worker thread stack
```

### 11.4 线程栈

如果三路模式后调用链变深，可能造成栈溢出。

优先关注：

```text
control thread
rehab worker thread
CAN RX processing thread
BLE thread
main thread
```

如果 fault 暂时难复现，先加大这些线程栈是低风险缓解方式。

### 11.5 ISR / callback 里做重活

CAN RX 或 timer 回调里不应直接做复杂状态机和阻塞操作。

推荐：

```text
ISR/callback 只入队
worker thread 里解析和应用
```

```text

2. RT handler 断点是否命中

```

## 15. 地址拆解练习：看到一个地址先判断什么

主线程后面继续调试时，最常见会拿到几类地址。不要一看到地址就直接 `addr2line`，先判断它属于哪个地址空间。

### 15.1 `0x083xxxxx`

这通常是当前 Non-secure M33 app 的代码地址，也就是 `rt-thread.elf` 里的地址。

例如：

```text
0x0839c1e0
0x0839c555
0x08398964
```

处理方式：

```powershell
arm-none-eabi-addr2line.exe -e .\rt-thread.elf -f -C 0x0839c1e0
```

如果 `addr2line` 能反查到函数，说明它属于当前 ELF。

### 15.2 `0x603xxxxx`

这通常是 M33 app relocation 后在外部 SMIF/QSPI 上的执行地址或镜像地址。

本项目构建时会把原始 app 从：

```text
0x08340400
```

relocate 到：

```text
0x60340400
```

注意：

```text
0x083xxxxx 和 0x603xxxxx 可能是同一段 app 的不同映射/加载视角
```

如果 OpenOCD 烧录时报：

```text
no flash bank found for address 0x60340400
```

优先怀疑烧录命令没有拉起 SMIF bank，而不是代码本身。

### 15.3 `0x181xxxxx`

这通常不是当前 RT-Thread app，而是 Secure 固件、ROM/安全启动相关区域或厂商安全 handler。

例如：

```text
0x181042ba
0x181042bb
```

这类地址用当前 `rt-thread.elf` 通常查不到源码：

```powershell
arm-none-eabi-addr2line.exe -e .\rt-thread.elf -f -C 0x181042ba
```

如果返回 `??`，不是工具坏了，而是这个地址不属于当前 ELF。

本项目里：

```text
0x181042bb = Secure HardFault vector 里的 handler 地址，最低 bit 是 Thumb bit
0x181042ba = 实际可下断点的代码地址
```

### 15.4 `0x240xxxxx`

这通常是 SRAM 地址，可能是：

```text
线程栈
RT-Thread 对象
heap
全局变量
异常栈现场
```

例如：

```text
MSP_NS = 0x240fcf18
```

这种地址不能直接 `addr2line`，因为它不是代码地址。要先判断里面存的 word 是什么。

如果它是异常栈地址，则按：

```text
word0 r0
word1 r1
word2 r2
word3 r3
word4 r12
word5 lr
word6 pc
word7 xpsr
```

读取 word6 才能拿到真正用于 `addr2line` 的 PC。

### 15.5 `0xE000xxxx` / `0xE002xxxx`

这是 Cortex-M 系统控制空间，常用于读 SCB 寄存器。

常见：

```text
0xE000ED08  Secure SCB->VTOR
0xE000ED0C  Secure SCB->AIRCR
0xE000ED24  Secure SHCSR
0xE000ED28  Secure CFSR
0xE002ED08  Non-secure SCB_NS->VTOR 视图
0xE002ED28  Non-secure CFSR 视图
```

这些地址也不能 `addr2line`。它们是寄存器。

## 16. Thumb bit：为什么 vector 里是奇数地址

Cortex-M 只执行 Thumb 指令。函数指针和异常向量表里的 handler 地址通常最低 bit 为 1，用来表示 Thumb state。

例如：

```text
vector 里看到 0x08398293
实际函数入口 0x08398292
```

```text
vector 里看到 0x181042bb
实际函数入口 0x181042ba
```

所以：

```text
addr2line 可以尝试奇数或偶数地址
下断点一般用偶数真实地址
```

OpenOCD 断点：

```text
bp 0x181042ba 2 hw
```

不要写成：

```text
bp 0x181042bb 2 hw
```

## 17. 如何根据 xPSR 判断 fault 发生在哪个异常里

异常栈里的 xPSR 低位包含 IPSR，也就是当前异常号。

本次：

```text
xPSR = 0x6100000f
```

低位：

```text
0x0f = 15
```

Cortex-M 异常号：

```text
15 = SysTick
```

所以：

```text
fault 发生时 CPU 正在 SysTick_Handler 或 SysTick 触发的 RTOS tick 路径里
```

常见异常号速查：

```text
3   HardFault
4   MemManage
5   BusFault
6   UsageFault
11  SVCall
14  PendSV
15  SysTick
16+ 外部中断 IRQ0, IRQ1, ...
```

如果 xPSR 低位是 0，说明 fault 发生在 thread mode，而不是异常中。

## 18. 为什么 SysTick 路径会暴露内存破坏

RT-Thread 的 SysTick 通常会做几类事：

```text
1. 系统 tick 自增
2. 检查 timer list
3. 唤醒超时线程
4. 把线程插入 ready list
5. 触发调度
```

所以它会频繁访问：

```text
rt_thread 对象
rt_timer 对象
链表节点
ready list
timer list
线程状态字段
线程优先级字段
```

如果前面某个模块越界写，把这些对象破坏了，SysTick 会是第一个踩上去的人。

因此：

```text
PC 落在 rt_schedule_insert_thread()
```

不能马上说：

```text
RT-Thread 调度器有 bug
```

更合理的判断是：

```text
调度器正在处理某个已经被写坏的线程对象或链表节点
```

## 19. 如何从崩溃点反推破坏点

### 19.1 先找被调度的是哪个线程

如果 PC 落在：

```text
rt_schedule_insert_thread(thread)
```

要想办法读传入参数里的 `thread` 指针。

根据函数调用 ABI，Cortex-M 函数前几个参数通常在：

```text
r0, r1, r2, r3
```

如果 fault 栈帧里的 `r0` 像一个 `0x240xxxxx` 地址，就可能是 thread 指针。

下一步可以 dump：

```text
mdw <r0地址> 32
```

再对照 RT-Thread 的 `struct rt_thread` 布局，看：

```text
线程名
栈起始地址
栈大小
当前 sp
状态
优先级
链表节点
```

如果线程名乱码、链表 next/prev 是异常地址，说明对象已被踩。

### 19.2 看 fault 前哪个线程最活跃

结合串口日志和业务流程：

```text
刚发了 CAN heartbeat?
刚切 rehab mode?
刚开 motor_report?
刚收到 F103 数据?
刚启动 BLE?
刚启动 sound/mic?
```

哪个动作最接近 fault，哪个模块就是第一怀疑对象。

### 19.3 做二分开关

不要一次性开全部功能。按二分法：

```text
只启动 shell
只启动 CAN
只发 heartbeat
只发 passive
只开 motor feedback
只开 joint4
开 joint4+5
开 joint4+5+6
切 assist
切 resist
```

观察 fault 从哪一步开始出现。

## 20. 三路助力模式相关专项排查表

当前业务上最可疑的新增复杂度是三路助力/阻力。这里列专项检查项。

### 20.1 joint mask 是否正确

三路 4/5/6：

```text
joint4 bit = 0x08
joint5 bit = 0x10
joint6 bit = 0x20
mask       = 0x38
```

检查所有代码里是否有：

```c
if (mask & (1U << joint))
```

如果 joint 是 1-based，这里应该是：

```c
if (mask & (1U << (joint - 1U)))
```

### 20.2 数组长度是否覆盖 joint 6

如果数组按 `CONTROL_MOTOR_JOINT_COUNT` 开，必须确认：

```text
CONTROL_MOTOR_JOINT_COUNT >= 6
```

如果当前只定义 3 个 ROS joints，但物理 joint id 是 4/5/6，就不能直接用 physical joint id 当 index。

危险：

```c
assist_state[joint_id]
resist_state[joint_id]
```

安全：

```c
index = joint_id - 1U;
if (index >= CONTROL_MOTOR_JOINT_COUNT)
{
    return -RT_EINVAL;
}
```

如果 `CONTROL_MOTOR_JOINT_COUNT` 不是物理最大 joint 数，而是 ROS joint 数，则还需要单独的映射函数，不能简单 `joint_id - 1`。

### 20.3 status 结构体是否被老代码按旧大小拷贝

如果最近给结构体新增字段：

```text
active_joint_mask
assist_engaged_mask
```

要检查是否有旧代码：

```c
memcpy(dst, src, old_size)
memset(status, 0, old_size)
rt_mq_send(..., sizeof(old_struct))
```

结构体大小不一致会导致状态错乱，甚至越界读写。

### 20.4 rehab worker 线程栈

三路模式会增加：

```text
循环次数
策略状态
日志
状态拷贝
安全检查
CAN 输出
```

如果 worker 栈原来只够单路，现在可能不够。

临时验证方法：

```text
把 rehab worker/control thread 栈加大一倍
```

如果 fault 消失，高度怀疑栈溢出。

### 20.5 CAN 回调是否直接调 rehab 状态机

如果 CAN RX callback 里直接做：

```text
解析 mode
申请 mutex
改 rehab 状态
发电机控制
打印大量日志
```

风险较高。

建议：

```text
CAN callback 只入队
rehab worker thread 处理
```

## 23. Case 1 - Secure 接管的 SysTick 调度路径 fault

触发条件：

```text
M33 启动/运行后发生 HardFault，RT-Thread shell 或控制链路不稳定。
```

关键证据：

```text
g_rt_hw_fault_dump = 全 0
RT-Thread HardFault_Handler 断点不命中
Secure HardFault handler 0x181042ba 断点命中
CONTROL_NS = 0
MSP_NS = 0x240fcf18
```

异常栈解析：

```text
stacked PC = 0x0839c1e0
stacked LR = 0x0839c555
xPSR       = 0x6100000f
```

地址含义：

```text
0x0839c1e0 -> RT-Thread tick/timer/scheduler 路径
0x0839c555 -> 同一调度相关调用链
xPSR low = 0x0f -> SysTick
```

判断：

```text
fault 暴露在 SysTick 调度路径，不代表 SysTick 是根因。
更可能是此前某个业务模块写坏了 thread/timer/list 数据结构。
```

当前最该排查：

```text
三路 rehab assist/resist 的 joint index/mask 越界
线程栈溢出
CAN 回调里做重活或错误调用 RTOS API
status 结构体大小/拷贝不一致
```

下一步建议：

```text
先只跑 shell + CAN passive，不开三路输出。
若不 fault，再逐步打开 joint4、joint5、joint6。
每次 fault 都重新取 MSP_NS 异常栈，看 stacked PC 是否变化。
```
