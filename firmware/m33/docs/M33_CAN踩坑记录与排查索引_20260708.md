# M33 CAN 踩坑记录与排查索引

日期：2026-07-08

用途：集中记录本项目 M33 CAN bring-up、NanoPi 通信、HardFault、模式切换、云端 WAIT 相关踩坑。每个坑按“现象、误判风险、根因、定位方法、修复/预防”整理，后续遇到类似问题可快速对照。

## 坑 1：M33 不回 322，看起来像 CAN 没收到

### 现象

```text
NanoPi cansend 0x321
candump 看不到 M33 0x322
模式切换不生效
云端页面仍 WAIT
```

### 误判风险

容易直接判断：

```text
CAN ID 不对
bitrate 不对
NanoPi 脚本错
M33 parser 错
```

### 真正问题

M33 可能已经收到 CAN RX，但在 RX 读取/解析路径里 HardFault，导致没有机会回 322。

### 定位方法

```text
1. halt M33
2. 读 xPSR/CFSR/HFSR
3. 看是否在 HardFault/Lockup
4. 读 CAN FIFO/MRAM/寄存器确认硬件是否已有帧
5. 解析异常栈 stacked PC
```

### 预防

CAN 不回时第一步先确认：

```text
M33 是否活着
CFSR/HFSR 是否为 0
xPSR 是否 Thread state
```

不要只看 candump。

## 坑 2：RT-Thread fault dump 全 0，不代表没有 HardFault

### 现象

```text
g_rt_hw_fault_dump 全 0
RT-Thread 串口没有打印 HardFault
```

### 误判风险

容易判断：

```text
不是 fault
是串口或 shell 问题
```

### 真正问题

Cortex-M33 TrustZone 下，HardFault 可能被 Secure handler 接管。Non-secure RT-Thread 的 `rt_hw_hard_fault_exception()` 没有执行，因此 dump 全 0。

### 定位方法

```text
读 VTOR
读 HardFault vector
比较实际 vector 是否等于 RT-Thread HardFault_Handler
```

本次关键证据：

```text
RT handler = 0x08398293
实际 vector = 0x181042bb
```

### 预防

遇到 dump 无效：

```text
不要停在 RT-Thread 层
立刻查 TrustZone / Secure vector
```

## 坑 3：RT-Thread HardFault 断点不命中

### 现象

在：

```text
0x08398292 HardFault_Handler
0x0839834c rt_hw_hard_fault_exception
```

下断点，不命中。

### 真正问题

当前 HardFault vector 指向 Secure handler：

```text
0x181042bb
```

实际断点应打：

```text
bp 0x181042ba 2 hw
```

### 学到的点

向量表里的地址最低 bit 是 Thumb bit：

```text
0x181042bb -> 实际入口 0x181042ba
```

## 坑 4：只读 MSP/PSP 拿不到真实现场

### 现象

CPU 停在 Secure HardFault，普通：

```text
reg msp
reg psp
```

不是 RT-Thread 业务现场。

### 真正问题

异常源头在 Non-secure app，但 handler 在 Secure state。

### 正确方法

读：

```text
reg msp_ns
reg psp_ns
reg control_ns
```

根据：

```text
CONTROL_NS bit[1]
```

选择 `MSP_NS` 或 `PSP_NS`。

本次：

```text
CONTROL_NS = 0
MSP_NS = 0x240fcf18
```

异常栈：

```text
stacked PC = *(MSP_NS + 0x18)
```

## 坑 5：崩在 SysTick，不代表 SysTick 是根因

### 现象

异常栈解析：

```text
stacked PC = 0x0839c1e0
xPSR = 0x6100000f
```

`xPSR low = 0x0f = SysTick`。

addr2line/objdump 落在：

```text
_thread_timeout()
rt_schedule_insert_thread()
```

### 误判风险

容易判断：

```text
RT-Thread 调度器有 bug
SysTick 配置有问题
```

### 正确理解

SysTick 经常是第一个踩到坏内存的人。真正破坏点可能更早：

```text
CAN RX 解析
rehab 三路模式
数组越界
线程栈溢出
定时器对象被踩
```

### 预防

继续缩小触发条件，找更直接的 PC。

## 坑 6：OpenOCD 能读 MRAM，不代表程序能直接读

### 现象

OpenOCD 可以读 CAN MRAM/FIFO 相关地址，看起来硬件有数据。

但 M33 程序执行到：

```text
0x42850000 + index * 16
```

直接 load 时 HardFault。

### 根因

Non-secure M33 运行态不能用这种固定 MRAM 地址直接读取 CAN RX FIFO 元素。可能涉及：

```text
安全属性
总线窗口权限
外设访问路径限制
MRAM 映射差异
```

### 修复

改为 CANFD FIFO TOP 寄存器读取：

```text
打开 RXFTOP_CTL.F0TPE
连续读 RXFTOP0_DATA
得到 R0/R1/data0/data1
```

### 预防

业务/驱动代码不要硬编码外设 MRAM 地址。优先使用：

```text
PDL 推荐路径
HAL 封装
寄存器窗口
官方 driver 已验证读法
```

## 坑 7：烧录写 0 bytes

### 现象

OpenOCD 输出：

```text
no flash bank found for address 0x60340400
wrote 0 bytes
verify failed
```

### 根因

M33 relocated image 位于 SMIF/QSPI 地址，烧录时没有拉起正确 flash bank。

### 正确烧录关键参数

```text
set DEVICE PSE84xGxS2
set ENABLE_CM55 1
```

成功标志：

```text
Programming Finished
Verified OK
```

### 预防

不要只看 OpenOCD exit code。必须看：

```text
wrote xxx bytes
Verified OK
```

## 坑 8：OpenOCD 末尾 acquisition warning 容易误导

### 现象

OpenOCD 最后出现：

```text
kitprog3: failed to acquire the device
Acquisition in Test Mode FAILED
```

### 判断方法

如果前面已经有：

```text
Programming Finished
Verified OK
```

则通常说明烧录已经成功，末尾 warning 是释放/重连阶段问题。

### 预防

报告时区分：

```text
烧录/verify 是否成功
reset/shutdown 是否有尾部 warning
```

## 坑 9：CAN 322 通了，但云端还是 WAIT

### 现象

CAN 上已经看到：

```text
321 -> 322
0x320 set_mode 被执行
```

但云端页面仍显示 WAIT。

### 误判风险

容易回头怀疑：

```text
M33 CAN 又坏了
0x322 payload 不对
```

### 真正排查方向

分层继续往上查：

```text
NanoPi ROS bridge
sensor_state publisher
uploader
ROS_DOMAIN_ID
云端 API
```

本次发现：

```text
NanoPi ROS 节点运行在 ROS_DOMAIN_ID=42
手动 ros2 topic list 没带 domain 42，所以看不到正确 graph
```

### 预防

查 ROS 前固定带：

```bash
export ROS_DOMAIN_ID=42
export ROS_LOCALHOST_ONLY=0
export ROS_AUTOMATIC_DISCOVERY_RANGE=SUBNET
```

## 坑 10：模式切换不能只看 322 字段

### 现象

0x322 状态字段没有直观显示模式变化，但 CAN 上出现了电机私有协议帧。

### 判断

说明：

```text
0x320 已进入 M33 并执行到 motor chain
只是 0x322 payload 诊断字段还不够清晰
```

### 预防

后续 0x322 必须增加：

```text
current_mode
last_seq
last_result
reject_reason
active_joint_mask
fault_flags
```

避免只能靠猜。

## 坑 11：三路助力的 joint id / mask 容易踩内存

### 风险

三路目标：

```text
joint4 -> 0x08
joint5 -> 0x10
joint6 -> 0x20
mask   -> 0x38
```

危险写法：

```c
array[joint_id]
array[mask]
array[bit]
```

### 预防

明确区分：

```text
physical joint id
ROS joint index
bit mask
array index
```

所有转换都必须有边界检查。

## 坑 12：串口/plink 抓日志可能干扰节奏

### 现象

plink 抓串口可能卡住、COM 被占用、输出不完整。

### 处理原则

关键 fault 现场不要依赖串口。

优先：

```text
OpenOCD halt
读寄存器
读 fault dump
读异常栈
addr2line
```

串口用于辅助，不作为唯一证据。

## 快速排查流程索引

### CAN 不回 322

```text
1. candump 确认 NanoPi 发出 321
2. halt M33，确认 xPSR 是否异常
3. CFSR/HFSR 是否为 0
4. 如果 fault，解析 MSP_NS stacked PC
5. 如果 M33 alive，读 rx_total/parsed/applied
6. 如果 M33 回 322，继续查 NanoPi ROS/uploader
```

### HardFault 没 dump

```text
1. 查 VTOR
2. 查 HardFault vector
3. 对比是否 RT handler
4. 若是 0x181...，断 Secure handler
5. 读 MSP_NS/PSP_NS/CONTROL_NS
6. 解析异常栈
```

### 模式切不了

```text
1. 先确认 321/322 heartbeat
2. 发送 0x320
3. 看 cmd parse/apply 计数
4. 看 reject reason
5. 看电机私有协议帧
6. 看 prearm/feedback/safety gate
```

### 云端 WAIT

```text
1. 先确认 CAN 321/322
2. 确认 NanoPi bridge 进程
3. 带 ROS_DOMAIN_ID=42 查 topic/node
4. echo sensor_state/safety_state
5. 查 uploader 日志
6. 查云端 API
```

## 总结

本项目 CAN 踩坑的主线是：

```text
不要把“没回包”直接等价为“协议错”
先确认 M33 是否活着
再确认异常是否被 Secure 接管
再解析真正的 Non-secure 异常栈
再反查到具体代码行
最后分层验证 M33 -> NanoPi -> ROS -> 云端
```

CAN 鲁棒性最终要靠：

```text
安全读取
严格解析
状态机
ACK/reject reason
超时降级
错误恢复
压力测试
```
