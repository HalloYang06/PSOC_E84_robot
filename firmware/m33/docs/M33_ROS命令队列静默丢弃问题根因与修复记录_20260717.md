# M33 ROS 命令队列静默丢弃问题根因与修复记录

日期：2026-07-17

## 1. 现场现象

NanoPi 发送合法的 `0x320 SET_MODE` 后，M33 诊断出现：

```text
ros_id=2 parsed=2 enq=2 applied=1
stale=0 recheck_reject=0 apply_fail=0
```

新增的 ASSIST 帧已经完成收包、解析和入队，但没有进入执行统计，也没有进入旧命令、安全复核拒绝或执行失败统计。`rehab status` 仍为 PASSIVE，电机没有动作。

该现象不能解释为模式安全门拒绝。安全门拒绝应增加 `recheck_reject`，执行失败应增加 `apply_fail`，命令超时应增加 `stale`。

## 2. 容易误判的原因

PASSIVE/STOP 使用独立 emergency latch，不经过普通 ROS message queue。测试中 PASSIVE 可以正常执行并增加 `applied`，但普通 ASSIST/RESIST/ACTIVE 命令全部经过 `s_ros_cmd_mq`。

因此会形成误导性组合：

```text
PASSIVE 正常
普通命令 enq 增长
ros_cmd 线程存在
电机无动作
```

这看起来像助力状态机或电机准入失败，实际问题发生在 MQ 消费返回值判断。

## 3. 根因

当前工程使用的 RT-Thread `rt_mq_recv()` 返回类型为 `rt_ssize_t`。成功时返回实际复制的消息长度，失败时返回负错误码：

```c
rt_ssize_t rt_mq_recv(...)
{
    return _rt_mq_recv(...);
}
```

`_rt_mq_recv()` 成功路径最终执行：

```c
return len;
```

原 `ctrl_ros_cmd_entry()` 仍按旧式 `rt_err_t` 语义判断：

```c
if (rt_mq_recv(...) != RT_EOK)
{
    continue;
}
```

完整 `control_ros_command_t` 的长度大于 0，因此每次成功接收都会满足 `!= RT_EOK`。消息已经从 MQ 取出并归还到 free list，但消费者立即 `continue`，没有设置 `normal_pending`，也不会进入任何执行结果计数。

## 4. 证据链

1. CAN RX 诊断证明命令完成 `ros_id -> parsed -> enq`。
2. `applied/stale/recheck_reject/apply_fail` 全部不变，证明命令没有进入消费后的分支。
3. GDB 只读检查显示：

```text
s_ros_cmd_mq.entry = 0
s_ros_cmd_mq.max_msgs = 16
s_ros_cmd_mq.msg_queue_head = 0
```

4. 队列为空但没有任何消费结果，符合“已取出后被错误 continue”的唯一数据流。
5. RT-Thread 当前源码明确记录：成功返回值大于 0，返回实际消息长度。

## 5. 修复

提交：

```text
2dfd90e93 fix(control): consume RT-Thread MQ length results
```

消费者现在保存 `rt_ssize_t` 返回值，并且只接受完整结构长度：

```c
recv_size = rt_mq_recv(...);
if (recv_size != (rt_ssize_t)sizeof(deferred_normal))
{
    continue;
}
```

这样同时处理：

- 正常完整消息：进入 `normal_pending` 和后续安全复核。
- timeout/interrupt 等负错误：继续等待。
- 异常短消息：拒绝执行，避免沿用结构体残留字段。

本次没有修改 MQ 深度、线程优先级、CAN 协议、rehab 状态机或电机控制算法。

## 6. 自动验证

测试采用先失败后修复：

- 修复前新增测试因缺少 `rt_ssize_t recv_size` 和长度判断而失败。
- 修复后 ROS MQ、CAN owner、单关节 mask、heartbeat lease、助力故障阻断和语音 precheck 相关测试共 `31 passed`。
- 完整 SCons 交叉编译通过，生成重定位后的 `build/rtthread.hex`。

构建仍有工程原有的未使用变量/函数警告，本次修改没有新增编译错误。

## 7. 安全影响

该缺陷主要造成普通远程命令静默丢失，没有观察到因此产生非预期电机动作。独立 PASSIVE/STOP latch 仍能执行，是正确的安全优先级设计。

但它会让上位机误以为命令已可靠交付，也会掩盖控制线程、状态机和助力策略的真实状态。因此验收不能只看 `enq`，必须同时检查：

```text
ros_id -> parsed -> enq -> applied
```

对于拒绝命令，必须能在 `stale/recheck_reject/apply_fail` 中看到唯一归宿。任何命令都不应在 `enq` 后无统计地消失。

## 8. 板上验证计划

1. 烧录包含 `2dfd90e93` 的 M33 `build/rtthread.hex`。
2. 发送 PASSIVE，确认 emergency latch 路径保持正常。
3. 发送缺失 mask 和多 bit mask，确认 NanoPi 在发送前拒绝。
4. 在 motor5 `fault=0`、反馈新鲜、限流 `0.2 A` 时发送：

```text
04 <seq> 03 10 00 00 00 00
```

5. 确认 `applied` 增长且 `rehab status` 明确进入单关节 ASSIST。
6. 立即发送 PASSIVE，确认模式回退且电流输出为 0。
7. 停止 heartbeat，确认 CAN owner 的 active 模式能够按租约自动回退 PASSIVE。

板上结果完成后，应在本文追加实际计数、模式状态、电流上限和 STOP 结果。

## 9. 板上复测结果

2026-07-17 烧录包含 `2dfd90e93` 的 M33 固件后，先确认：

```text
motor5 fault=0x00
VOICE_PRECHECK result=PASS
assist_max_x1000=200
CAN state=ERROR-ACTIVE
CAN tx/rx error=0, bus-off=0
```

NanoPi 发送单关节 ASSIST：

```text
TX STD 0x320 [8] 04 33 03 10 00 00 00 00
```

M33 诊断结果：

```text
ros_id=1 parsed=1 enq=1 applied=1
stale=0 recheck_reject=0 apply_fail=0
mode=assist source=1 m33_joint=5 fresh=1
current_x1000=-30 limit_x1000=200
MOTOR[5] mode=2 fault=0x00
```

随后立即发送 PASSIVE：

```text
TX STD 0x320 [8] 04 34 00 00 00 00 00 00
```

停止结果：

```text
ros_id=2 parsed=2 enq=2 applied=2
mode=passive current_x1000=0 limit_x1000=0
MOTOR[5] mode=0 fault=0x00
```

结论：`rt_mq_recv()` 长度返回值修复已通过真机 `0x320 + mask=0x10` 的入队、消费、执行和 PASSIVE 回退验证。本次是短时无人工施力测试，电流始终低于 `0.2 A` 上限；人工施力下的助力方向、峰值与停止手感需要单独现场验收，本记录不将其标记为已完成。
