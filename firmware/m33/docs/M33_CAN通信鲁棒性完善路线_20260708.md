# M33 CAN 通信鲁棒性完善路线

日期：2026-07-08

用途：在当前 M33/NanoPi/F103/M55/云端闭环已经打通基础通信后，继续把 CAN 从“实验室能跑”提升到“现场更稳、更可诊断、更安全”的工程路线文档。后续新增问题、指标、测试脚本和协议字段时，可以持续追加到本文件。

## 1. 当前状态判断

当前 CAN 链路已经验证到：

```text
NanoPi can0 -> M33 0x321 heartbeat
M33 -> NanoPi 0x322 status/ack
NanoPi -> M33 0x320 set_mode
M33 执行到 rehab/motor 链路
阻力/被动切换时能看到电机私有协议帧
```

也定位并修复过一个关键底层问题：

```text
Non-secure M33 程序直接读取固定 CAN MRAM 地址 0x42850000 会 fault
修复为通过 CANFD FIFO TOP 寄存器窗口读取 RX FIFO
```

这说明链路“能通”，但还不能算工业级鲁棒。后续重点应从“继续堆功能”转为“通信层抗异常、可恢复、可观测”。

## 2. 工业级 CAN 的目标

工业级 CAN 不是保证永远不出错，而是保证：

```text
错误帧不会导致 HardFault
异常输入会被拒绝
拒绝原因可观察
超时会自动降级
总线错误能恢复
电机不会因通信异常保持危险输出
云端能显示健康状态
```

目标能力：

1. 输入严格校验。
2. RX 和业务处理解耦。
3. heartbeat 和超时降级。
4. 命令 ACK 和拒绝原因。
5. sequence 去重和重发容忍。
6. bus-off/error-passive 监控和恢复。
7. 电机安全门独立于 CAN 命令。
8. 故障现场可记录、可回放、可定位。
9. NanoPi 侧具备压力测试和自动验证脚本。
10. 云端能看到 M33 CAN 健康状态。

## 3. 分层架构建议

推荐分成四层，不要让业务代码直接碰 CAN 寄存器或 MRAM。

```text
硬件/PDL 层
  - CANFD 寄存器
  - FIFO TOP
  - TX/RX status
  - bus error status

M33 CAN driver 安全封装层
  - can_read_fifo_safe()
  - can_send_frame_checked()
  - can_get_bus_health()
  - rx overflow / tx fail 计数

协议解析层
  - 0x321 heartbeat parser
  - 0x320 set_mode parser
  - 0x7C1/F103 or sensor parser
  - DLC/cmd/mode/mask/seq 校验

业务状态机层
  - rehab mode manager
  - safety/prearm
  - motor control
  - cloud/uploader status
```

原则：

```text
业务层不直接访问 CAN 寄存器
业务层不直接读 MRAM
CAN callback 不直接做复杂业务
所有输入先变成受校验的 command object
```

## 4. RX 处理模型

当前最应该补强的是 RX 解耦。

推荐：

```text
CAN ISR / CAN callback
    只负责：
      1. 从 FIFO TOP 安全读取一帧
      2. 填充 can_rx_frame_t
      3. 放入 ring buffer 或 rt_mq
      4. 增加 rx_total/rx_overflow 计数

CAN worker thread
    负责：
      1. 取出 frame
      2. 按 ID 分发
      3. 严格解析
      4. 更新 heartbeat/status
      5. 调用 rehab mode manager
      6. 生成 ACK/status
```

不要在 ISR/callback 中：

```text
拿可能阻塞的 mutex
执行 rehab 状态机
下发电机控制
打印大量日志
等待队列/信号量
做 memcpy 大块数据
```

## 5. 协议严格校验

### 5.1 0x321 heartbeat

建议规则：

```text
ID        = 0x321
DLC       = 1 或 8，需统一定义
Byte0     = sequence 或 heartbeat counter
其它字节  = reserved，建议必须为 0，或明确含义
```

校验失败：

```text
不刷新 heartbeat 时间戳
记录 reject reason
可选择回 0x322 错误状态
```

### 5.2 0x320 set_mode

当前格式：

```text
ID    = 0x320
DLC   = 8
Byte0 = 0x04    # SET_MODE
Byte1 = seq
Byte2 = mode
Byte3..7 reserved
```

mode：

```text
0x00 passive
0x03 assist
0x04 resist
```

必须校验：

```text
DLC == 8
Byte0 == 0x04
mode in {0x00, 0x03, 0x04}
reserved bytes 是否符合约定
heartbeat fresh
prearm 通过
motor feedback fresh
joint mask allowed
```

### 5.3 0x322 status / ack

建议把 0x322 设计成可诊断状态，而不是只表示“我还活着”。

建议字段：

```text
Byte0 = protocol version 或 status marker
Byte1 = last_rx_seq
Byte2 = current_mode
Byte3 = last_result
Byte4 = reject_reason
Byte5 = active_joint_mask
Byte6 = fault_flags low
Byte7 = heartbeat_age 或 counter
```

result：

```text
0x00 OK
0x01 ACCEPTED
0x02 REJECTED
0x03 DUPLICATE_ACK
0x04 DEGRADED
0x05 FAULT
```

reject_reason：

```text
0x00 NONE
0x01 BAD_DLC
0x02 BAD_CMD
0x03 BAD_MODE
0x04 BAD_MASK
0x05 HEARTBEAT_TIMEOUT
0x06 PREARM_FAILED
0x07 MOTOR_FEEDBACK_STALE
0x08 SAFETY_LOCKED
0x09 BUS_ERROR
0x0A INTERNAL_FAULT
```

## 6. Sequence 与重复执行保护

任何会改变状态的命令都应带 sequence。

规则：

```text
同 source + cmd + seq 已执行过：
  不重复执行业务动作
  只重发 ACK

新 seq：
  解析、校验、执行

seq 跳变：
  记录 seq_gap
  根据策略决定是否接受
```

这样 NanoPi 重发不会导致：

```text
重复切模式
重复下发电机命令
重复触发安全状态变化
```

## 7. Heartbeat 与超时降级

建议定义：

```text
heartbeat fresh        <= 300 ms
heartbeat degraded     > 300 ms
heartbeat lost         > 500 ms
communication offline  > 1000 ms
```

动作：

```text
fresh:
  允许接收 assist/resist 请求

degraded:
  不接受新 assist/resist
  保持或转 passive，按安全策略

lost:
  强制 passive
  停止 rehab 输出

offline:
  motor safe stop / idle
  fault_flags 标记 CAN_LINK_LOST
```

状态恢复：

```text
heartbeat 恢复后，不自动进入 assist/resist
必须重新 prearm
必须重新接收 set_mode
```

## 8. Bus error 与自动恢复

需要监控：

```text
TEC
REC
error active/passive
bus off
rx fifo overflow
tx fail
arbitration lost
last error code
```

建议状态机：

```text
CAN_LINK_DOWN
CAN_LINK_HEARTBEAT_ONLY
CAN_LINK_READY
CAN_LINK_ACTIVE
CAN_LINK_DEGRADED
CAN_LINK_FAULT
```

bus-off 策略：

```text
进入 CAN_LINK_FAULT
rehab mode -> passive
motor safe stop
延迟重新初始化 CAN controller
恢复后等待 heartbeat
等待 prearm
不自动恢复助力/阻力
```

## 9. 电机安全门独立于 CAN

CAN 命令永远不能直接等价于电机输出。

模式切换前必须通过：

```text
heartbeat fresh
motor feedback fresh
F103/sensor 数据 fresh
joint mask allowed
prearm passed
mode allowed
torque/current/rpm limits valid
emergency stop not active
no internal fault
```

任何失败：

```text
拒绝命令
保持 passive
必要时 motor stop
0x322 上报 reject_reason
云端显示拒绝原因
```

```text

```

## 11. 压力测试清单

必须不 HardFault：

```text
乱 DLC
乱 Byte0 cmd
乱 mode
乱 reserved bytes
乱 joint mask
重复 seq
seq 跳变
高频 321
高频 320
0x321/0x320 交错发送
can0 down/up
NanoPi 进程 kill/restart
M33 reset
F103 停发
motor feedback stale
M55 不响应
云端 uploader 断网
```

测试通过标准：

```text
M33 不 HardFault
CAN 不 bus-off，或 bus-off 后可恢复
电机进入安全状态
0x322 给出明确 reject/fault 原因
NanoPi 能重新同步
云端状态可解释
```

## 12. 后续实施优先级

### P0：防崩溃

```text
统一 CAN RX 安全读取
清除所有业务层直接寄存器/MRAM 访问
RX queue + worker thread
所有 parser 加 DLC/cmd/mode 边界检查
```

### P1：可诊断

```text
完善 0x322 状态/ACK/reject reason
增加 cmd_control_debug 输出字段
NanoPi 记录 last_322_result
云端显示 reject reason
```

### P2：安全降级

```text
heartbeat timeout -> passive
bus-off -> safe stop
feedback stale -> reject assist/resist
prearm 必须 gate 所有主动模式
```

### P3：压力测试

```text
NanoPi fuzz 脚本
自动化 cansend/candump 验证
M33 fault snapshot 脚本
回归测试文档化
```

## 13. 当前最小落地建议

下一步建议先做这 5 件事：

1. 固化 CAN RX FIFO TOP 读法，禁止回退到固定 MRAM 地址读取。
2. 给 0x320/0x321 parser 加严格 DLC/cmd/mode/reserved 校验。
3. 设计并实现 0x322 的 `result + reject_reason + last_seq` 字段。
4. M33 内部加入 heartbeat timeout 自动 passive。
5. NanoPi 写一个 `can_robustness_check.sh`，自动发正常帧、异常帧、重复 seq、高频帧并统计 322。

## 14. 核心原则

```text
CAN 通信不是“收到就执行”，而是“收到 -> 校验 -> 状态机 -> 安全门 -> 执行 -> ACK”。
```

```text
工业级鲁棒性不是不出错，而是出错可控、可恢复、可诊断。
```
