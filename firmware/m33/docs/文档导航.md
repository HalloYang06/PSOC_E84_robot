# 文档导航

Last updated: 2026-07-06

This repository uses long-lived Git branches to hold different parts of the rehabilitation arm system. Do not read the current checkout as the whole project. Start here, then open the subsystem branch you need.

## Read First

1. `docs/AI_PROJECT_STRUCTURE_GITHUB.md`
   - Stable AI agent entry index copied from the current project structure brief.
   - Explains branch ownership, subsystem routes, document update rules, and the formal/shadow/bench boundary.

2. `docs/PROJECT_LONG_TERM_BRIEFING.md`
   - Long-term project briefing.
   - Explains the system goal, branch map, current truth, known gaps, and next engineering direction.

3. `docs/PROTOCOL_AND_SAFETY_BOUNDARIES.md`
   - Safety and protocol boundary document.
   - Explains which paths may move hardware, which paths are readonly/debug/suggestion-only, and how CAN/BLE/IPC IDs fit together.

4. `docs/小智语音云端Token配置流程.md`
   - XiaoZhi cloud relay token configuration and verification record.
   - Explains platform token generation, M33 chunked delivery to M55, persistent FAL provisioning, WiFi reconnect, and `m55qa_status` success criteria.

5. `docs/CAN_MOTOR_BRINGUP_RETROSPECTIVE_20260605.md`
   - CAN and motor bring-up retrospective.
   - Useful when debugging physical CAN, motor feedback, stale/fresh semantics, and `0x320` safety concerns.

6. `docs/can_busoff_debug_20260608.md`
   - Bus-off note for the current M33 CAN direct-send guard.
   - Useful after power interruptions, bus wiring changes, or repeated direct-send failures.

7. `docs/用户手册.md`
   - Current operator-facing QA commands.
   - Includes M33 rehab bench current-mode commands, motor slot parsing examples, and runtime gain tuning.

8. `docs/康复智能跟随算法.md`
   - Current rehab follow/assist/resist control law, tuning order, and smart-follow learning path.
   - Use this before changing adaptive assist or impedance/admittance logic.

9. `docs/M33_CAN_NANOPI_BRINGUP_20260521.md`
   - Older M33/NanoPi handoff.
   - Still useful for basic heartbeat and bring-up commands, but newer safety boundaries in this index take precedence.

10. `docs/NANOPI_CAN_MANUAL_DEBUG_PROTOCOL.md`
   - NanoPi SocketCAN manual debug protocol.
   - Summarizes manual `cansend`/`candump` tests for NanoPi self-check, M33 heartbeat, F103/C8T6 sensor frames, private motor Get_ID, CANSimple scan, loopback caveats, and bus-off interpretation.

## Current Source Of Truth

When documents disagree, use this order:

1. Hardware evidence from the latest bring-up session.
2. `feature/rehab-arm-ros2-architecture` current briefing/progress/protocol docs.
3. Current M33 code in `applications/control/` and `applications/m33/`.
4. This current-branch documentation index and briefing.
5. Older README/demo docs.

Old demos are useful for learning, but they are not proof of current product readiness.

## Documentation Rules

- Say whether a feature is `mainline`, `shadow-sim`, `dry-run`, `bench-debug`, `offline-demo`, or `side-channel`.
- Do not write that a path controls real hardware unless it passes through M33 safety authority.
- Mark synthetic/demo data explicitly.
- Keep CAN IDs, BLE commands, IPC message types, and ROS topics in protocol docs instead of burying them in chat history.
- Record hardware validation with date, branch/commit, command, observed frame/log, and what was not tested.
- Prefer updating these long-term docs over adding new one-off notes unless the note is a specific debug incident.
