# Nanopi 与 F103 联调调试分析

日期：2026-07-07

## 结论摘要

本次联调最终确认：

- Nanopi-M5 网络已恢复，`192.168.3.36` 可 ping，SSH 22 端口可达。
- Nanopi `can0` 工作在 1 Mbps，状态为 `ERROR-ACTIVE`，总线错误计数为 0。
- F103/C8T6 传感节点已恢复输出，Nanopi 被动抓包可看到稳定的 `0x7C2` 传感帧和周期 `0x7C3` 健康帧。
- ROS `/rehab_arm/sensor_state` 已解析 F103 数据，频率约 50.94 Hz。
- ROS `/rehab_arm/safety_state` 仍保持 PSoC/M33 安全状态闭环，`source=psoc`、`state=ok`、`motion_allowed=false`。
- 云端上传进程在 Nanopi 上运行中，但本轮手动 POST smoke 因命令行 JSON 转义问题返回 422，不能作为云端失败证据。

## 系统边界

本次现场链路包含四个主要节点：

- Infineon M33：康复控制、安全状态、CAN 控制层，负责发送 `0x322` 安全状态。
- Infineon M55：小智语音、云端 WebSocket、WiFi/token 持久化。
- Nanopi-M5：CAN 桥接、ROS 话题、非实时传感上报。
- STM32F103/C8T6：肌电/ADC 传感节点，通过 CAN 输出 `0x7C2` 和 `0x7C3`。

关键 CAN ID：

| ID | 方向 | 含义 |
| --- | --- | --- |
| `0x321` | Nanopi -> M33 | Nanopi 心跳 |
| `0x322` | M33 -> Nanopi | PSoC/M33 安全状态 |
| `0x7C0` | M33/Nanopi -> F103 | F103 控制命令 |
| `0x7C1` | F103 -> 总线 | F103 ACK |
| `0x7C2` | F103 -> 总线 | F103 传感数据，4 路 ADC，小端 `u16` |
| `0x7C3` | F103 -> 总线 | F103 健康状态 |

## 现象与排查过程

### 1. 重新上电后安全状态不恢复

现象：

- Nanopi 可看到自己发出的 `0x321`，但没有 M33 返回的 `0x322`。
- M33 `cmd_control_debug` 显示 `rx_total=0 hb=0`。
- 未初始化时执行 `cmd_m33_prearm_check` 触发 RT-Thread mutex 断言。

根因：

- `applications/main.c` 中 `M33_XIAOZHI_MINIMAL_FRAMEWORK` 为 1。
- `m33_init_framework()` 在启动 M55 IPC 后提前返回，跳过了 `can_driver_init()`。
- 因此重新上电后 CAN/control layer 没有稳定自动初始化。

修复：

- 在最小框架下新增 `can_auto` 延迟线程，启动后延迟 2 秒调用 `can_driver_init()`，失败则重试。
- 给 `cmd_m33_prearm_check` 增加 `s_is_inited` 检查，未初始化时打印诊断，不再直接访问未初始化 mutex。

验证：

- SCons 编译通过。
- OpenOCD 烧录 M33，`Verified OK`。
- 执行 `reboot` 后不手动运行 `cmd_control_init`，M33 自动恢复心跳处理。
- Nanopi 抓到连续 `0x321/0x322`。
- ROS `/rehab_arm/safety_state` 解析为 `source=psoc`、`state=ok`、`motion_allowed=false`。

### 2. F103 一度看起来“不通”

现象：

- 用户断电 Nanopi，仅保留 Infineon 和 F103。
- 此时不能再用 Nanopi 作为 CAN 分析仪。
- M33 串口执行：
  - `f103_ping 5 100` 返回 `ret=0`
  - `cmd_sensor_rate 1 100` 返回 `ret=0`
  - 但 `CTRL_DBG_F103: ack=0 sensor=0 health=0`

判断修正：

- 在 Nanopi 断电时，不能用 Nanopi 抓包判断 F103。
- `hb` 不增长也是正常现象，因为 Nanopi 心跳 `0x321` 已停。
- 这轮只能说明 M33 未收到 F103 返回，不能直接证明 F103 永久不在线。

### 3. Nanopi 上电后 F103 恢复确认

Nanopi 恢复后，重新测试：

网络：

```bash
ping 192.168.3.36
```

结果：

- 3 发 3 收
- 丢包 0%
- SSH 22 端口可达

CAN 状态：

```bash
ip -details -statistics link show can0
```

结果：

- `can0` 为 `UP, LOWER_UP`
- `state ERROR-ACTIVE`
- bitrate `1000000`
- bus error / bus-off 计数为 0

被动抓 F103：

```bash
timeout 4s candump -L can0,7C0:7FC
```

关键证据：

```text
can0 7C2#7C021F022A02FA05
can0 7C2#7A021C022802FA05
can0 7C2#88020F023902EE05
can0 7C3#0100000002006F17
```

说明：

- `0x7C2` 正在连续输出。
- `0x7C3` 健康帧存在。
- F103 在总线上是活的。

ROS 传感话题：

```bash
ros2 topic echo --once --full-length /rehab_arm/sensor_state
ros2 topic hz /rehab_arm/sensor_state
```

关键结果：

- `source=f103_sensor`
- `id_hex=0x7C2`
- `valid=true`
- `physical_source=stm32_f103_emg3_can_0x7c2`
- 示例 ADC：`[633, 543, 561, 1531]`
- 频率约 `50.94 Hz`

安全话题：

```bash
ros2 topic echo --once --full-length /rehab_arm/safety_state
```

关键结果：

- `source=psoc`
- `id_hex=0x322`
- `state=ok`
- `control_mode=bench_armed`
- `motion_allowed=false`

## 排查经验

1. 先确认 Nanopi 是否上电。Nanopi 断电时，`hb` 不增长是正常现象，不能把它误判成 M33 CAN 接收失败。
2. F103 问题优先用 Nanopi 被动抓包确认。只要能看到 `0x7C2` 和 `0x7C3`，F103 本体和总线输出就是活的。
3. M33 侧 `f103_ping` 返回 `ret=0` 只代表命令发送成功，不代表 F103 已回包。
4. 看到 `0x7C2` 后，还要确认 ROS `/rehab_arm/sensor_state`，这样才能证明 Nanopi 桥接和解析链路也通。
5. 安全闭环和传感数据是两条链路。`0x321/0x322` 正常不等于 F103 正常，F103 正常也不等于允许运动。
6. 不建议现场热用 `cmd_can_bitrate 1000000` 恢复 CAN；之前该命令导致 M33 卡住。掉线后优先用复位或 `cmd_control_init can0`。
7. 小智语音不应直接绕过 M33 的 `rehab_service/rehab_mode_manager` 安全检查；语音路径只生成模式请求，实际执行仍要走 M33 安全门禁。

## 后续建议

- 将 F103 被动抓包、ROS sensor echo、ROS sensor hz 三条命令整理成一键脚本，减少现场手动转义错误。
- 云端上报建议用 Nanopi 本地 Python 文件或固定脚本做 POST smoke，不要在 Windows -> SSH -> shell 多层命令里直接嵌 JSON。
- 如果需要证明 M33 也解析到了 F103 数据，需要恢复 KitProg3 COM 口后再执行 `cmd_sensor_show` 和 `cmd_control_debug`。
- 继续保持 `motion_allowed=false` 的安全边界，直到急停、限位、电机反馈新鲜度和校准条件全部确认。
