# 康复智能跟随算法笔记

Last updated: 2026-06-17

Status: bench-debug design note. This document explains the current M33 rehab
current-mode behavior and the next algorithm direction. It is not product
motion permission.

## Current Firmware Behavior

The local M33 rehab service currently owns three current-mode bench strategies:

- `active`: if torque leaves the deadband, output current follows torque sign.
  If only velocity leaves the deadband, it outputs the minimum follow current.
- `assist`: after torque passes the enter threshold, output current is
  proportional to measured torque until torque falls below the exit threshold.
- `resist`: output current opposes velocity, proportional to velocity.

Runtime shell entry:

```text
rehab status
rehab cfg
rehab active 5
rehab assist motor5
rehab resist m33:5
rehab stop
```

Runtime tuning entry:

```text
rehab cfg direction -1
rehab cfg active_gain 1.6
rehab cfg active_max 1.0
rehab cfg assist_gain 2.5
rehab cfg assist_max 1.0
rehab cfg adaptive_enable 0
rehab cfg resist_gain 3.0
rehab cfg resist_max 1.0
rehab cfg resist_dir 1
rehab cfg
```

If the operator feels "force is too small", first check these in order:

1. `rehab status` shows `fresh=1` and the intended `m33_joint`.
2. The current direction is correct. If the motor pushes against the intended
   follow direction, stop and flip `direction`.
3. `current_x1000`, `limit_x1000`, and `sat` show whether the output is already
   capped. If `sat=1`, increasing gain will not increase current.
4. `assist_gain` or `active_gain` is high enough to produce useful current from
   the measured torque.
5. The torque feedback is meaningful under load. If torque barely changes while
   the physical load changes, no assist algorithm can adapt from torque alone.

If the operator feels "resistance is not enough", first check:

1. Velocity feedback changes with hand motion.
2. `resist_dir` makes the current oppose motion instead of helping it.
3. `sat=1` means `resist_max` is now the limiting value.
4. `resist_gain` is high enough for slow movement. Low velocity multiplied by
   a low gain can look like no resistance.

## Why Fixed Gain Feels Weak

The current algorithm is linear and memoryless:

```text
assist_current = direction * sign(torque) * abs(torque) * assist_gain
resist_current = -resist_dir * velocity * resist_gain
```

This is safe and easy to tune, but it does not understand the difference between
gravity, patient intent, friction, a sudden external load, or sensor noise. When
load changes, it only reacts if the measured torque or velocity changes enough.

The most common reasons for poor load following are:

- feedback stale or low-rate compared with the physical motion;
- torque estimate filtered too heavily or not calibrated;
- current cap reached before the user feels enough help;
- direction sign wrong, so the motor partially cancels the intended behavior;
- gravity/friction compensation missing, so the fixed gain spends most current
  overcoming the mechanism rather than helping the user.

## Optimization Directions

### 1. Better Manual Gain Scheduling

Use different gains by joint, direction, and speed range:

```text
low speed:  higher assist_gain, higher static friction compensation
mid speed:  normal assist_gain
high speed: lower gain and stronger current cap
```

This is the safest next step after bench tuning. It still has predictable
bounds, but feels less flat than one gain for all conditions.

### 2. Impedance Or Admittance Control

For human-following rehab control, learn these two concepts early:

- Impedance control: command torque/current from position and velocity error,
  like a virtual spring-damper.
- Admittance control: convert measured human force/torque into a desired motion,
  like a virtual mass-damper.

For this M33 current-mode path, a practical bench version is closer to
admittance-assisted current control:

```text
intent = filtered_torque_after_deadband
desired_assist = Kt * intent + friction_feedforward + gravity_feedforward
resistance = B * velocity
current = clamp(desired_assist - resistance)
```

The important idea is not "make gain bigger". It is to separate intent,
friction/gravity, and damping so each part can be tuned independently.

### 3. Adaptive Assist Gain

The firmware now has an optional adaptive assist gain path for `assist` mode.
It is disabled by default, and it never changes the current cap. Adaptive gain
changes slowly inside hard safety bounds:

```text
load_estimate = low_pass(abs(torque)) + velocity_load_term
target_gain = base_gain + load_gain * load_estimate
effective_gain = rate_limit(target_gain, previous_gain)
current = clamp(sign(torque) * effective_gain * abs(torque), max_current)
```

Rules for safety:

- never adapt `max_current` automatically;
- rate-limit gain changes;
- freeze adaptation when feedback is stale or direction is uncertain;
- log the effective gain, output current, torque, velocity, and saturation flag.

Current bench controls:

```text
rehab cfg adaptive_enable 1
rehab cfg adaptive_base 2.0
rehab cfg adaptive_load 0.5
rehab cfg adaptive_max 4.0
rehab cfg adaptive_step 0.1
```

When this is enabled, `gain_x1000` in `rehab status` is the effective gain.
`sat=1` still means the current cap is limiting the output.

### 3.5 Load/Speed Scheduled Adaptive PID

The firmware also has a bounded adaptive PID outer loop for `assist` and
`resist`. It is disabled by default. This is the algorithm requested for
"PID parameters change automatically according to load and speed".

It does not replace the motor's internal current loop. M33 still sends a target
current through `iq_ref(0x7006)`. The adaptive PID only adds a small, limited
current trim on top of the existing assist/resist feedforward.

The scheduler normalizes measured load and speed:

```text
load_level  = clamp((abs(torque) - pid_load_low)  / (pid_load_high  - pid_load_low),  0, 1)
speed_level = clamp((abs(vel)    - pid_speed_low) / (pid_speed_high - pid_speed_low), 0, 1)

Kp = kp_base + load_level * kp_load + speed_level * kp_speed
Ki = ki_base + load_level * ki_load - speed_level * ki_speed_reduce
Kd = kd_base + speed_level * kd_speed
```

For `assist`:

```text
error = abs(torque) - assist_pid_target
pid_trim = PID(error)
assist_current = clamp(abs(torque) * assist_gain + pid_trim, 0, assist_max)
```

For `resist`:

```text
error = abs(velocity) - resist_pid_target
pid_trim = PID(error)
resist_current = clamp(abs(velocity) * resist_gain + pid_trim, 0, resist_max)
```

Safety rules in this bench implementation:

- `assist_pid_enable` and `resist_pid_enable` default to `0`.
- PID trim has its own `*_pid_trim` limit and still cannot exceed the mode
  current cap.
- Integral state is bounded by `*_pid_i_limit`.
- PID state is reset when parameters change, when the mode changes, and when
  resist enters the velocity deadband.
- `rehab status` exposes `pid_kp_x1000`, `pid_ki_x1000`, `pid_kd_x1000`,
  `pid_load_x1000`, `pid_speed_x1000`, `pid_err_x1000`, and
  `pid_trim_x1000`.

Useful first bench commands:

```text
rehab cfg pid_load_low 0.2
rehab cfg pid_load_high 1.5
rehab cfg pid_speed_low 0.03
rehab cfg pid_speed_high 0.20

rehab cfg assist_pid_target 0.25
rehab cfg assist_pid_kp_base 0.10
rehab cfg assist_pid_kp_load 0.35
rehab cfg assist_pid_ki_base 0.02
rehab cfg assist_pid_ki_load 0.04
rehab cfg assist_pid_kd_speed 0.01
rehab cfg assist_pid_trim 0.30
rehab cfg assist_pid_enable 1

rehab cfg resist_pid_target 0.05
rehab cfg resist_pid_kp_base 0.10
rehab cfg resist_pid_kp_speed 0.50
rehab cfg resist_pid_kd_speed 0.02
rehab cfg resist_pid_trim 0.30
rehab cfg resist_pid_enable 1
```

Tune slowly. If `sat=1`, the hard current cap is limiting the output; increasing
PID gains will not create more current. If `pid_trim_x1000` jumps sign rapidly,
lower `kd_speed` or widen the velocity range.

### 4. Friction And Gravity Compensation

Before adding complex AI, identify the mechanism:

- Coulomb friction: constant current needed to start moving.
- Viscous friction: current proportional to velocity.
- Gravity torque: depends on joint angle.
- Cable/load changes: depends on posture and external load.

Even a simple table by joint angle can make assist feel much smarter than a
large fixed gain.

### 5. EMG Or Model Intent As Advisory Input

M55, EMG, or cloud/VLA results should remain suggestion-only unless routed
through M33 mode arbitration and safety checks. A good first use is to choose a
profile:

```text
low confidence intent  -> lower assist_gain
clear flex intent      -> normal assist_gain
fatigue detected       -> higher assist_gain within cap
spasm or anomaly       -> stop or high resistance only if safety policy allows
```

Do not let model output directly command motor current.

## Learning Path

Study in this order:

1. Discrete-time control basics: sampling period, low-pass filters, saturation,
   deadband, hysteresis, and rate limiters.
2. Motor current control: what current mode means, how current maps to torque,
   and how CAN feedback latency affects stability.
3. PID plus feedforward: especially friction and gravity feedforward.
4. Impedance and admittance control for physical human-robot interaction.
5. System identification: collect logs, fit friction/gravity/load terms, then
   validate on held-out movements.
6. Safety engineering: fault latching, current caps, stale feedback handling,
   stop latency, and operator-visible state.

## Proposed Next Firmware Step

After board-side tuning proves the signs and gains:

1. Add rehab service telemetry fields for output current, measured torque,
   velocity, saturation, and effective gain.
2. Keep adaptive assist only in `assist` mode until bench logs prove it is stable.
3. Keep the hard current cap from `assist_max`.
4. Verify with a bench script:
   - no feedback: stop and latch fault;
   - low torque: no assist;
   - increasing torque: gain rises slowly;
   - torque drops: gain falls slowly;
   - current never exceeds `assist_max`.

The first useful experiment is not a neural network. It is good telemetry plus a
bounded adaptive gain that can be turned off instantly.
