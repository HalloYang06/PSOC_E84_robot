# 助力失控根因与五号电机弯举整改记录

日期：2026-07-18  
适用工程：M33 `Edgi_Talk_M33_Blink_LED`、Android App `ai-`

## 1. 事故结论

现场把 `rehab cfg direction` 从 `-1.0` 临时改为 `+1.0` 后，4/5/6 号关节出现持续加速和大电流。该修改只发生在运行时，没有写入源码，随后已执行 `rehab stop` 并恢复为 `-1.0`。

根因不是简单的“方向写反”，而是助力速度回退算法形成了正反馈：

1. 速度绝对值超过 `0.01 rad/s` 后，算法自动进入 velocity fallback。
2. 输出符号由 `follow_direction * sign(feedback_velocity)` 决定。
3. `direction=-1` 时电流与运动方向相反，使用感受接近阻力。
4. `direction=+1` 时电流与运动方向相同，形成“电流增大 -> 速度增大 -> 同方向电流继续增大”的正反馈。
5. 四档又把最小助力电流同步到约 `2 A`，虽然存在每 20 ms 约 `0.03 A` 的斜率限制，仍会持续爬升到限流值。

因此，不能再通过全局正负号试错修复助力。正确的助力方向必须来自受控轨迹或可信的高层运动意图，反馈速度只用于阻尼、到位判定和安全保护。

## 2. 暴露出的底层缺口

### 2.1 速度保护未实际生效

`CONTROL_REHAB_ASSIST_MAX_VEL_RAD_S` 虽然已有定义，CAN 模式命令也携带最大速度，但原助力执行路径没有消费该值。电流设定点前只检查反馈新鲜度和电机 fault，没有实时速度越限停机。

### 2.2 电流模式缺少位置包络

原助力 current 路径没有每关节机械角度软限位，也没有“到边界只允许向内输出”的约束。出现正反馈后，软件无法在接近机械端点时主动截断。

### 2.3 stop 缺少电机确认

当前私有协议 stop 以 CAN 发送成功作为成功，没有等待 ACK 或闭环状态回读。CAN 正常时能快速停机，但总线拥塞、掉线或电机异常时缺少确认和重试证据。

### 2.4 运行中允许修改方向

Shell 可以在 active/assist/resist 运行期间修改 `follow_direction`，服务会立即替换参数并复位策略。后续应限制为仅 passive 状态可改，并改成逐关节标定参数。

### 2.5 原始坐标与关节坐标混用

私有协议 `cmd_motor_fb` 当前显示的是电机原始位置，而 `control_motor_position_control()` 接收的是关节坐标，并会再次应用零偏、方向和减速比。

- 5 号零偏：`6.010 rad`，方向 `+1`
- 现场原始硬边界：`6.238..7.829 rad`
- 对应关节硬边界：约 `0.228..1.819 rad`

把 `6.238/7.829` 直接传给位置控制 API 会再次加上 `6.010 rad` 零偏，目标严重错误。现有轨迹录制保存 raw feedback、回放却当 joint target 使用，也存在同类风险，不能用于本次弯举。

## 3. 本次五号电机弯举实现

本次没有复用助力 current 策略，也没有修改 MuJoCo/ROS 路径。新增独立的单关节位置闭环状态机：

`IDLE -> MOVE_TOP -> DWELL_TOP -> MOVE_BOTTOM -> DWELL_BOTTOM -> MOVE_TOP`

固定配置：

| 项目 | 数值 |
|---|---:|
| 运动关节 | 仅 5 号，mask `0x10` |
| raw 硬边界 | `6.238..7.829 rad` |
| raw 训练目标 | `6.288..7.779 rad` |
| 端点安全余量 | `50 mrad` |
| 位置到位容差 | `30 mrad` |
| 到位连续样本 | 5 个 |
| 指令速度上限 | `0.12 rad/s` |
| 反馈硬速度上限 | `0.35 rad/s` |
| CSP 电流上限 | `2.0 A` |
| 端点驻留 | `500 ms` |
| 单段超时 | `20 s` |
| 位置命令刷新 | `200 ms` |

原始目标在服务内显式转换为 joint target 后再调用 CSP 位置接口。任何陈旧反馈、电机 fault、raw 位置越界、反馈超速或单段超时都会锁存故障并调用既有 stop 路径。状态改变前后受 `actuation_lock` 和 `mode_generation` 保护，旧线程周期不能在 stop 或新模式之后继续下发位置。

6 号电机本轮不驱动。用户给出的范围暂记录为 raw `3.532..5.101 rad`，对应当前标定 joint 约 `-1.011..0.558 rad`；在独立标定和单关节低速验收前不加入弯举。

## 4. App 到 M33 的控制链

App 只发送固定高层请求，不携带位置、电流或任意轨迹：

```json
{"schema":"rehab_ble_v1","type":"training_request","request_id":104,"profile":"single_joint_curl_j5_v1","joint_mask":16,"ttl_ms":1000}
```

控制流程：

1. App 先通过云端安全门创建训练记录。
2. Android 原生 BLE NUS 插件发送固定 `training_request`。
3. M33 BLE worker 解析请求并交给 mode manager。
4. mode manager 绑定 BLE session generation、App 所有权和 TTL lease。
5. M33 完成反馈/fault/位置/速度预检后返回同 `request_id` 的 `command_result`。
6. App 只有收到 `result=applied` 才进入训练页。
7. 训练页每 500 ms 发送 heartbeat；暂停、结束和离页发送 `stop_request`。
8. 心跳超时或 BLE 断连后，M33 lease 路径触发 stop。

GATT write complete 只表示手机写入特征成功，不代表 M33 已接受动作。本实现以 M33 的 applied ACK 为准。

## 7. 真机验收顺序

当前不能直接宣称稳定运行，也不能在 4/5/6 故障未检查时直接启动训练。

1. 烧录后先保持电机功率关闭，确认 Shell、CAN RX 和 `rehab_svc` 正常。
2. 执行 `m33_ble_status`，再执行一次 `m33_ble_start`；确认无 HardFault，`list_thread` 检查 `ble_work`、HCI 和 `rehab_svc` 栈余量。
3. 手机扫描/连接后只测 heartbeat 和 ACK，不上电运动。
4. 读取 `cmd_motor_fb 5`，要求 fault 为 0、反馈连续、位置在 raw `6.238..7.829 rad` 内、静止速度低于门限。
5. 机构卸载或可靠支撑，人员手持急停，先验证单次向内目标；方向错误立即断电，不允许改全局 direction 试错。
6. 完成一个 `6.288 -> 7.779 -> 6.288 rad` 往返，连续观察 `rehab status` 的 `curl_phase/curl_reps`。
7. 分别注入断连、停心跳、CAN 反馈停止、fault 和轻微越界，确认均停机且不会自动恢复。
8. 通过以上项目后，才允许 App 连续运行多次弯举并记录温升、峰值电流、段时间和最大调度抖动。

## 9. 2026-07-19 五号助力方向与力度实机复盘

### 9.1 本轮有效安全边界

- 五号助力 raw 位置硬限位调整为 `6.000..8.264 rad`。
- 助力进入电流模式前先检查反馈新鲜度、fault 和位置，越界时不允许执行 current prepare。
- 助力工作线程每周期重复检查位置和速度。
- 五号助力只允许正向电流；策略产生负向电流时强制改为 `0 A` 并复位该关节助力斜坡状态。
- 五号电流脉冲 Shell 命令同样受 `6.000..8.264 rad` 限位保护。
- 助力硬停速度由 `1.0 rad/s` 分阶段调整为 `2.0 rad/s`。五号减速比为 `1.0`，该数值就是关节侧速度，约 `114.6 deg/s`。
- 默认助力改为二档，`assist_min=assist_max=1.0 A`；运行时仍可切换 `0.5/1.0/1.5/2.0 A` 四档。

### 9.2 向下推力根因与证据

修复前的三秒实机测试中，人员只用手托住机构、无人穿戴。串口同时记录到：

- 策略输出 `current=-0.03 A`；
- 随后负向速度达到 `-1.052 rad/s`；
- `fault_stage=4` 超速保护触发并自动停机。

该负电流不是力矩分支导致。现场力矩只有约 `-0.067 Nm`，低于 `0.5 Nm` 力矩触发阈值。实际路径是机构受重力先向负方向下落，velocity fallback 把下落误判为用户负向意图并输出负电流；电流斜坡又在换向时短暂保留负值，从而形成明确的向下主动推力。

加入五号负向电流硬阻断后复测：

- 负向运动 `vel=-0.020 rad/s` 时，`current=0 A`；
- 正向运动 `vel=+0.023 rad/s` 时，`current=+0.03 A`；
- 用户确认不再感到主动向下推力；
- 停机后 `mode=0、fault=0`。

### 9.3 正向助力分档验收

所有测试均为无人穿戴、人员手托机构、短时运行并在脚本 `finally` 路径无条件发送 `rehab stop`。

| 测试 | 现场结果 |
|---|---|
| `+0.2 A / 200 ms` 方向脉冲 | `7137 -> 7199 mrad`，确认正电流对应数值增大方向 |
| 一档短时助力 | 实际 `+0.300 A`，速度 `+0.354 rad/s`，力矩 `+0.339 Nm`，位置 `7366 -> 7459 mrad` |
| 二档短时助力 | 上限 `1.0 A`，实际爬升到 `+0.660 A`，速度 `+0.671 rad/s`，力矩 `+0.651 Nm`，位置 `7435 -> 7662 mrad` |

二档测试未出现负电流、越界、超速或电机 fault，用户确认该力度合适，因此没有继续进行 `1.5 A` 和 `2.0 A` 测试。烧录最终镜像后，Shell 查询确认上电默认值为 `level=2, current=1.000 A`。

### 9.4 仍未解决的问题

1. `control_motor_stop()` 当前语义为电机失能，机构没有机械抱闸，停机、复位和烧录期间会在重力作用下自由下落。软件不能把急停改成持续通电保持来替代机械抱闸，硬件仍需抱闸、配重或弹簧平衡方案。
2. 当前助力仍依赖 velocity fallback 识别正向动作；尚未建立基于关节姿态和负载模型的重力补偿，也未完成穿戴负载验收。
3. 高频执行完整 `rehab status` 会把 `rehab_svc` 最大调度抖动推高到约 `117..131 ms`。正式运动调试需要轻量快照或锁存计数，不能在控制期间连续打印完整状态。
4. 本轮只验证了五号单关节。四号、六号的正负方向、位置包络和安全电流仍需分别标定，不能直接把五号结论复制到 `mask=0x38`。
5. 小睿、App、BLE 和 CAN 模式切换链路尚未在本轮最终镜像上重新做端到端验证；本轮结论只覆盖 M33 Shell 到五号电机的底层执行路径。
