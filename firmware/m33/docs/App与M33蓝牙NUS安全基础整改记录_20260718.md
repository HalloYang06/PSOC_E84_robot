# App 与 M33 蓝牙 NUS 安全基础整改记录（2026-07-18）

## 1. 本阶段边界

本阶段只建立 App BLE NUS 的只读传输基础，以及 M33 蓝牙运行门、诊断、断线清理和异步启动状态机。

以下路径没有接入蓝牙：

- 康复模式执行；
- 4、5、6 号关节控制；
- CAN `0x320`；
- MuJoCo 动作规划；
- 电机电流、速度、位置和急停控制；
- M33/M55 共享 IPC。

M33 仍保持：

```c
#define M33_ENABLE_APP_BLE_RUNTIME 0
```

因此当前镜像不会自动启动蓝牙，也不能据此宣称 App 与设备已经完成真机联调。

## 3. HardFault 风险整改

### 3.1 已消除

- Shell 诊断不再解引用可能已经释放的 `cybt_task[]`。
- Shell 诊断不再跨线程读取 live HCI queue wrapper。
- rehab、BLE worker 和 Shell 栈水位扫描在 RT-Thread 临界区内完成查找和有界扫描。
- HCI RX/TX 栈没有可靠生命周期所有权时显示 `unsupported`，不读取悬空指针。
- 系统 heap largest-free 和当前 HCI TX heap 不支持时显示 `unsupported`，不把 0 伪装成真实值。
- GATT 高频事件日志默认编译关闭，事件只做定长计数。
- GATT 断线后清除旧 CCCD、旧数据长度和 pending command，避免重连后继承上一次会话。
- BLE runtime 和 HCI runtime 改为复制快照 API，不向其他线程暴露内部可变指针。
- 蓝牙栈启动返回 `PENDING` 时保持 STARTING，不再提前报告 READY。

### 3.2 尚未解决，禁止据此开启量产运行

1. `cybt_platform_task_deinit()` 仍可能与 HCI queue 使用并发，底层 queue wrapper 生命周期需要单独整改。
2. `cybt_platform_task_get_tx_heap_utilization()` 当前固定返回 0，不能代表真实系统 heap 压力。
3. `APP_BLE` 来源、心跳 TTL 租约和断线停止已实现，但重复/旧 `request_id` 去重和 BLE worker bridge 尚未实现。
4. App 目前只有 BLE 扫描、连接、分帧和 heartbeat 基础，没有完成主动/助力/抗阻按钮联调。
5. 当前没有硬件端口，本阶段没有烧录、蓝牙连接、反复断连或 HardFault 压力测试。

### 3.3 GATT RX worker 已完成的防护

- GATT 回调只验证连接、指针和长度，使用 `rt_mq_send()` 非阻塞入队，不解析、不发 ACK、不调用控制。
- RX MQ 深度固定为 4，消息和 2048 字节 worker 栈均为静态内存，队满返回失败并计数，不覆盖旧消息。
- 重组器支持 20/244 字节分片和同包多帧，单帧上限 256 字节，500 ms 从首个分片起算，后续分片不延长截止时间。
- 断线和新连接通过 generation 隔离；旧半帧、队列中的旧消息和迟到 worker 消息都不能进入新会话。
- `rt_mq_recv()` 按当前 RT-Thread API 的实际消息长度 `> 0` 判断，并再校验长度必须等于完整消息结构。
- 完整帧只进入 `rehab_ble_v1` 严格解析器；解析成功后仍不提交控制，不会进入 CAN、电机或康复控制执行路径。

### 3.4 NUS TX 和协议已完成的防护

- ACK 和 telemetry 不能从 GATT 回调或其他线程直接调用 vendor notify，只有 BLE worker 是发送者。
- 发送使用持久静态 buffer，必须同时收到对应连接和 handle 的 notification complete，以及对应 buffer returned，才能复用。
- 生产者必须携带 `{generation, conn_id}` session token；断线后的旧 ACK、旧 telemetry 和迟到完成事件不能被当作新会话数据。
- 协议解析器不使用 heap，固定 32 个 token，单帧最大 256 字节，只接受 heartbeat、mode_request 和 stop_request。
- mode_request 只接受 active、assist、resist，并要求 `joint_mask=0x38`；被动模式不在 App 协议中。
- 异常 UTF-8、嵌套值、未知或重复字段、数字溢出、尾随字节和越界 TTL 都会被拒绝。

这些防护只说明传输和解析边界已建立，不代表 App 已获得模式控制权。

### 3.5 APP_BLE 所有权与退出路径

- BENCH、CAN、VOICE 的枚举值保持 0、1、2，`APP_BLE` 只追加为 3。
- App 租约与 NanoPi/VOICE legacy 租约分开存放；`0x321` CAN 心跳只能更新 legacy 租约。
- App 心跳必须同时匹配 service source、mode generation 和 BLE session generation，迟到心跳不能解除已锁存停止。
- App 只接受 active、assist、resist，只接受 `joint_mask=0x38`，TTL 范围为 200–2000 ms。
- App 不能抢占活动 CAN、VOICE 或 BENCH；legacy 非 STOP 命令也不能抢占活动 App。
- 模式切换在 service 内以 source + generation 进行准备前、准备失败后和 apply 前三次校验，失效请求不会污染新 owner 的 fault 状态。
- TTL 和断线停止使用 `stop_if_owned(APP_BLE, mode_generation)`；显式 STOP 可停止任意当前模式。
- 底层停机失败后保持 fail-closed，每 100 ms 重试；重复 STOP 只返回 pending/busy，不会绕过节流反复发 CAN 停机帧。

这些 manager API 尚未从 BLE worker 调用，因此当前代码仍不会因 App 帧产生电机动作。

## 4. 当前状态机语义

```text
OFF --m33_ble_start--> STARTING
STARTING --BTM_ENABLED success--> READY
STARTING --BTM_ENABLED failure--> FAILED
STARTING/READY --BTM_DISABLED--> FAILED
```

- `wiced_bt_stack_init()` 返回 success、pending 或 already-initialized，只表示启动请求被接受。
- `BTM_ENABLED_EVT` 成功是进入 READY 的唯一条件。
- FAILED 不会被迟到的成功回调重新激活。
- 当前不支持 stop/restart，也不会在失败路径中删除可能仍运行的 vendor 线程。

## 5. 软件验证记录

静态回归覆盖：

- BLE 默认关闭和唯一 Shell 启动入口；
- BLE 不直发 CAN、不直控电机、不绕过 `rehab_mode_manager`；
- NanoPi `0x320` 的 `joint_mask=msg->data[3]` 保持不变；
- GATT 指针、offset、CCCD 长度、断线清理和 runtime 快照；
- 异步 READY/FAILED、迟到事件和 READY 降级；
- 诊断不访问 live HCI queue/TCB。
- TX 只有单发送者，持久 buffer 同时等待完成与归还事件；
- 同一 `conn_id` 复用、错连接完成事件、两种完成事件顺序和会话切换；
- 严格 JSON 解析的正常、缺字段、重复字段、越界和恶意输入；
- RX worker 在严格解析前后都重新校验 session generation，且不存在旧 ASCII submit 路径。

最近一次主线程验证：GATT RX 重组 Host 测试、严格协议 Host 测试、App 租约 Host 测试和 BLE/CTL 安全边界静态测试通过，M33 SCons 全量构建和 HEX 重定位通过。本次构建尺寸为 `text=480180 data=15532 bss=311944`。

## 6. 后续顺序

1. 实现按 BLE session 分区的 `request_id` 重复/旧请求守卫，防止 ACK 丢失重传导致频繁重入模式。
2. 新增独立 bridge，把严格协议请求映射到 manager；GATT 断线回调只提交代际事件，不直接调用 rehab service。
3. 完成无动作 heartbeat 验收后，才允许把 runtime 临时改为 1 做硬件 BLE 测试。
4. 最后接 App 模式按钮和小智高层请求；主动、助力、抗阻固定使用 `joint_mask=0x38`，停止保持独立。

## 7. 后续硬件验收最低要求

在 runtime 仍为 0 时先验证：

```text
m33_ble_status
m33_ble_diag
```

预期：运行门显示 disabled/off，Shell 不死机，控制和 CAN 路径行为与整改前一致。

只有 worker、协议和来源租约完成后，才进入临时 runtime=1 的无动作测试。无动作测试至少覆盖：连续 100 次连接/断开、错误 MTU、半帧、超长帧、队列满、App 强退和蓝牙控制器启动失败；期间不得出现 HardFault、旧命令重放或电机动作。
