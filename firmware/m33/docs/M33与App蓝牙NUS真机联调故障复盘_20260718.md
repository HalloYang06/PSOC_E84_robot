# M33 与 App 蓝牙 NUS 真机联调故障复盘

## 1. 文档目的

本文记录 2026-07-18 在 M33 固件、英飞凌 CYW55500 蓝牙控制器和 Android App 之间进行 BLE NUS 真机联调时，实际出现并完成定位的工程问题。记录重点不是“最终能连接”，而是每个问题如何通过现象、证据、根因、最小修复和回归验证形成闭环，可用于项目复盘、比赛答辩和面试说明。

本轮遵循以下边界：

- 不改变已有 CAN、运动控制和 M33/M55 IPC 协议。
- BLE 默认不自动启动，必须显式编译启用并通过 Shell 启动，降低新增功能引起 HardFault 的风险。
- 每个修复独立提交，避免把构建、调度、广播和 App 修改混在一起。
- 手机“云端账号绑定”、Android 系统配对和 BLE GATT 连接是三件不同的事情，验收时分别判断。

## 2. 联调链路

```text
Android App
  -> Android BLE 扫描与 GATT 客户端
  -> CYW55500 广播与 HCI UART
  -> M33 BLE Worker / NUS Service
  -> 后续模式命令、控制安全门和 ACK（尚未完成端到端验收）
```

## 3. 已定位问题

### 3.1 修改了源文件，但实际构建没有包含

**现象**

代码已经修改，编译也没有报错，但烧录后行为完全没有变化，容易误判为运行时逻辑失效。

**证据**

- RT-Thread Studio 生成的 `Debug/*.mk` 存在陈旧源文件列表。
- 当前工程真正使用 SCons 的 `applications/m33/SConscript` 决定 M33 应用源文件和宏定义。
- 对最终 ELF/HEX 检查后，才能确认目标代码是否真正进入镜像。

**根因**

把 IDE 生成文件当成权威构建入口，导致“编辑的代码”和“烧录的镜像”不属于同一条构建链。

**最小修复**

在 `applications/m33/SConscript` 中加入显式环境开关 `M33_APP_BLE_RUNTIME=1`，仅在明确启用时注入 `M33_ENABLE_APP_BLE_RUNTIME=1`。不改默认行为，不依赖陈旧的 `Debug/*.mk`。

**验证**

- 使用 SCons 完整构建。
- 使用 Edge Protect 工具生成安全合并镜像。
- 对最终 `rtthread.hex` 执行写入后校验，并记录 SHA256。

**工程经验**

嵌入式项目不能只以“编译成功”作为证据，还要核对构建入口、链接产物、镜像生成和烧录文件是否一致。

### 3.2 C 函数缺少原型，存在真实 HardFault 风险

**现象**

蓝牙复位后需要读取平台配置。源码调用 `cybt_platform_get_config()`，但没有包含其声明头文件，编译器可能按隐式函数声明处理。

**根因**

调用点缺少 `cybt_platform_util.h`。在 32 位 ARM ABI 下，错误的返回类型或调用约定可能破坏返回值和寄存器，问题不一定在调用处立刻表现，可能最终以 BusFault、UsageFault 或 HardFault 暴露。

**最小修复**

- 包含平台提供的正式声明头文件。
- 删除本地重复定义的默认 HCI UART 波特率宏，避免配置来源分裂。

**验证**

- 增加静态测试，要求调用点必须包含正式头文件。
- 编译不再出现隐式声明风险。
- 实机蓝牙补丁加载、协议栈就绪和广播启动成功。

**工程经验**

嵌入式 C 中的 warning 可能是 ABI 错误，不应把“能链接”视为安全。函数原型、结构体布局和指针宽度都直接关系运行稳定性。

### 3.3 高优先级忙循环导致 Shell 线程饿死

**现象**

系统看似运行，但串口 Shell 无法输入；调试器暂停时经常停在 idle 或调度相关代码，容易误以为 CPU 卡死在 idle。

**证据**

- M33 `main` 线程优先级高于 `tshell`。
- 最小框架分支中的主循环持续执行空操作，没有阻塞、延时或让出 CPU。
- RT-Thread 固定优先级调度下，只要高优先级线程始终 ready，低优先级 `tshell` 就无法运行。

**根因**

`M33_XIAOZHI_MINIMAL_FRAMEWORK` 被设为 1，使固件进入未让出 CPU 的最小循环。问题本质是线程饥饿，不是 Shell 串口端口未打开，也不是 idle 函数本身故障。

**最小修复**

恢复可调度的完整框架，使控制逻辑由线程运行，主线程进入正常休眠/调度路径；同时同步静态测试的期望值。

**验证**

- 烧录后 `tshell` 可稳定进入。
- 可以执行 `m33_ble_start`、`m33_ble_status` 和 `m33_ble_diag`。
- CAN RX owner 静态测试和康复模式静态测试通过。

**工程经验**

“停在 idle”常表示当前没有更高优先级就绪任务，并不等于死机。判断 RTOS 卡死需要结合线程状态、优先级、栈使用和是否存在永久 ready 的忙循环。

### 3.4 BLE 运行时必须有显式安全门

**现象**

蓝牙功能涉及 HCI UART、补丁下载、协议栈回调和多个线程。若在系统启动早期无条件自动启动，会扩大与 M33/M55、外部 Flash 和现有控制线程之间的资源竞争范围。

**风险分析**

- 初始化顺序不稳定时，错误更容易表现为 HardFault。
- 蓝牙线程栈和堆占用会改变原有系统余量。
- 自动重试可能形成高频错误循环，使根因更难观察。

**最小处理**

- 编译时必须显式设置 `M33_APP_BLE_RUNTIME=1`。
- 上电后仍不自动启动，通过 `m33_ble_start` 单次启动。
- 保留状态、错误码、堆余量和关键线程栈诊断。

**验证**

启动后状态为 `enabled=1 state=3 last_error=0`，没有出现自动反复初始化。

**工程经验**

新外设首次接入成熟控制系统时，应先采用“默认关闭、显式启用、可诊断、可回退”的方式，完成长稳测试后再考虑自动启动。

### 3.5 广播包缺少 NUS UUID，且受 31 字节限制

**现象**

固件能够启动广播，但 App 无法可靠识别它是康复设备。只按设备名扫描会产生误匹配，也不利于协议演进。

**根因**

原广播数据没有携带 Nordic UART Service 的 128 位 UUID。BLE Legacy Advertising 的广播数据最多 31 字节，完整名称和 128 位 UUID 同时加入后必须精确控制长度。

**最小修复**

- 广播名称统一为 `RehabArm`。
- 广播中加入 NUS Service UUID `6e400001-b5a3-f393-e0a9-e50e24dcca9e`。
- 更新 GAP 广播元素数量，最终广播数据恰好 31 字节。

**验证**

PC 使用 Bleak 无过滤扫描得到：

```text
address=00:A0:50:11:44:77
name=RehabArm
rssi=-52（一次现场采样）
service_uuid=6e400001-b5a3-f393-e0a9-e50e24dcca9e
```

**工程经验**

协议标识应优先使用 Service UUID，设备名只适合作为用户展示或兼容条件。修改广播时必须计算长度，不能依靠肉眼估计。

### 3.6 Android 36 硬件 ScanFilter 没有回调

**现象**

- M33 已正常广播。
- PC 能扫描到 `RehabArm` 和正确 NUS UUID。
- Android App 使用 Service UUID 的硬件 `ScanFilter` 后，扫描周期内没有任何结果回调，页面为空。

**排除过程**

1. 通过 M33 Shell 确认协议栈和广播启动成功。
2. 用独立 PC Bleak 扫描确认射频、名称和 UUID 均正确。
3. 检查 Android 权限和扫描启动日志，扫描 API 已成功调用。
4. 去掉 Android 硬件过滤后，系统立即回调 M33 广播。

**根因判断**

现场 Android 36 设备/蓝牙栈没有按预期返回 128 位 Service UUID 硬件过滤结果。问题位于 Android 扫描过滤路径，而不是 M33 未广播。

**最小修复**

- 平台层启动无硬件过滤扫描。
- 在回调进入候选列表前执行软件白名单：名称必须严格等于 `RehabArm`，或广播中包含 NUS UUID。
- 连接后仍严格校验 NUS Service、RX 可写、TX 可通知和 CCCD，不能只凭名称建立可信连接。

**验证**

- App 页面显示 `RehabArm` 及实时 RSSI。
- GATT 连接和服务发现成功。
- TX Notification 与 CCCD 配置成功。
- MTU 协商到 247。
- Web 层收到 `link_ready`，云端设备绑定成功。

**工程经验**

跨平台 BLE 问题要分层证明：射频广播、OS 扫描、应用过滤、GATT 发现、特征能力和业务绑定。软件过滤提高兼容性，但连接后的协议能力校验不能省略。

### 3.7 Android 构建 JDK 版本不匹配

**现象**

使用 JDK 17 执行 Gradle 测试时出现 `invalid source release: 21`。

**根因**

App 当前 Android 构建配置要求 Java 21，而命令行环境指向旧 JDK。Android Studio 内置 JBR 与独立终端的 `JAVA_HOME` 不一致。

**最小修复**

命令行构建显式使用 Android Studio JBR：

```text
JAVA_HOME=F:\Andorid Studio\jbr
ANDROID_HOME=F:\Android\Sdk
```

**验证**

- BLE Contract 单元测试通过。
- `assembleDebug` 通过。
- APK 使用 ADB 覆盖安装成功。

**工程经验**

移动端和嵌入式混合项目需要固定工具链版本。IDE 能构建不代表 CI 或终端能构建，应把 JDK、SDK、Gradle 和构建命令纳入可复现记录。

### 3.8 OpenOCD 退出阶段报错不等于烧录失败

**现象**

OpenOCD 已完成写入和校验，但关闭会话后仍打印 acquisition error，容易让操作者误判烧录失败并重复写 Flash。

**证据**

- 主镜像写入后校验通过。
- XIP 镜像校验通过。
- 脚本最终返回成功，并打印 `Verified flash completed successfully`。
- 异常文本发生在目标已复位、调试会话退出之后。

**处理原则**

烧录脚本应以写入、verify、退出码和最终状态共同判断结果，不能只搜索日志中的 `error` 字符串。

### 3.9 页面切换后无法重新扫描，实际连接并未断开

**现象**

App 已连接 RehabArm，从训练页切换到其他页面再返回后，页面不再显示已连接设备；再次点击搜索会失败或一直找不到设备。

**证据**

- Capacitor 原生 `RehabArmBlePlugin` 的生命周期长于单个 HTML 页面，GATT 仍处于 `link_ready`。
- 页面跳转会重新创建 `device.html` 的 JavaScript 状态，`bleListenersAttached`、设备 Map 和页面提示全部回到初始值。
- 原生 `scan()` 只允许从空闲态开始；当前仍为 `LINK_READY`，因此返回 `BLE_INVALID_STATE`。
- 外设处于已连接状态时通常也不会继续以同样方式被扫描到。

**根因**

Web 层把页面内存状态误当成物理 BLE 状态。页面重建后没有调用原生 `status()` 恢复已有连接，随后又试图在连接态启动扫描。

**最小修复**

- 页面加载时读取原生 BLE `status()`。
- 若状态为 `link_ready` 且存在 `selected_address`，恢复 RehabArm 设备卡片和“蓝牙已连接”提示。
- 点击搜索时若已有有效连接，返回当前设备而不是强制断开重扫。
- 对同一地址执行账号绑定时跳过重复 GATT `connect()`，只继续云端绑定步骤。

**验证**

- App BLE Product Wiring QA 通过。
- App BLE 原生边界 QA 通过。
- Android Debug APK 全量 clean build 通过，并使用 ADB 覆盖安装成功。
- 页面切换后的真机人工复测仍需用户在当前安装包内执行并保留截图，因此不能仅凭静态测试写成完整真机 PASS。

**工程经验**

Hybrid App 至少存在 Web 页面状态、原生插件状态、Android GATT 状态和外设连接状态四层生命周期。页面销毁不等于蓝牙断线，恢复页面时必须从原生层重新获取权威状态。

### 3.10 M55 源码编译通过，但完整链接受后端和专有库影响

**现象**

新增 M55 BLE 状态接收和 LVGL 显示代码能够编译，但主工作区与干净 worktree 都无法直接完成最终链接。

**分层证据**

1. 主工作区构建先被未提交的 `voice_mode_intent.c` 中 `stati const` 拼写错误阻断。该文件不是本轮修改，未擅自回滚。
2. detached 干净 worktree 避开上述脏文件后，默认 `edge_impulse` 后端缺少 `ifx_deepcraft_wake_*` 适配符号。
3. 显式设置 `XIAOZHI_WAKE_BACKEND=deepcraft` 后，所有本轮 M55 源文件编译通过，但最终链接仍缺少 `ifx_afe_hpf_*`、`itsi_feature_*`、`SOD_*`、`ifx_wwd` 等专有音频/唤醒库符号。

**结论**

用户将 `stati const` 恢复为 `static const` 后，M55 主工作区使用默认 SCons 配置完成最终链接并生成可烧录 HEX：

```text
ELF: text=2074548, data=17560, bss=4537192
rtthread.hex bytes=5884674
rtthread.hex SHA256=EA759C5F82A2D6493764CFFA244CD0CF16CC3BC87635FEFF7B15B3C36DD82CD9
```

因此当前工作区的直接构建阻断已经消除，但 detached 干净 worktree 仍不能复现同一链接，说明唤醒后端选择或专有库依赖尚未完全纳入仓库/构建配置。当前可以声明“主工作区构建通过”，不能声明“任意干净环境可复现”。

2026-07-18 已使用板载 KitProg3 将上述同一 SHA256 的 M55 `rtthread.hex` 写入 SMIF Flash。OpenOCD 加载 `PSE84_SMIF.FLM`，识别到目标区间 `0x60000000-0x67FFFFFF`，实际写入 `2097152 bytes`，并成功校验 `2092108 bytes`。`reset run` 后 COM16 重新返回 `msh />` 提示符，说明 RT-Thread 已恢复运行。OpenOCD 在外部复位后关闭调试域时打印了 KitProg3 再次 acquire 失败，该日志出现在写入与 verify 均成功之后，不影响本次 Flash 结果。LVGL 上的蓝牙状态文案仍需人工观察屏幕验收。

**工程经验**

“单文件编译通过”和“完整固件可链接”必须分开记录。嵌入式 AI 工程还要冻结后端选择、模型、NPU runtime、专有静态库及其版本，否则干净环境不能复现构建。

## 5. 尚未完成和残余风险

以下项目不能宣称已经完成：

1. 上电后仍需人工执行 `m33_ble_start`，自动启动策略尚未经过长稳验证。
2. App 尚未稳定启动 BLE 心跳/业务流，M33 当前仍显示 `streaming=0`、`down=0`。
3. App 的主动、助力、抗阻三种模式和档位命令尚未完成 M33 ACK 与端到端控制验收。
4. 模式命令必须固定作用于 4、5、6 号关节，即 `mask=0x38`，并经过实时故障门、限幅和超时保护，不能绕过运动安全层。
5. 尚未完成多次断连重连、手机息屏、蓝牙开关、M33 重启和数小时连接压力测试。
6. Android 系统层“配对”并不是当前 NUS GATT 链路的必要成功条件；云端“绑定成功”也不能代替真机服务和特征校验。
7. App 连接成功后，候选设备按钮偶尔仍显示“连接中”，属于 UI 状态未完全收敛，不影响已建立的 GATT，但会误导用户。
8. HCI 内部任务栈使用量尚不可见，仍需通过厂商接口或更底层诊断补齐。

## 6. 后续整改顺序

1. 先让 App 心跳写入 NUS RX，并确认 M33 `down` 持续增长、超时后安全回落。
2. 定义主动、助力、抗阻和档位请求帧，只表达高层意图，不直接携带绕过安全层的电机电流命令。
3. 固定 `mask=0x38`，在 M33 统一经过状态机、故障阻断、速度/角度/电流限幅和命令租约。
4. 增加带序号、状态和拒绝原因的 ACK；App 只有收到 ACK 才更新最终模式状态。
5. 分别执行无电机、悬空低电流和人工施力验收，再逐步提高助力/抗阻档位。
6. 完成长稳、断连、重启、队列满、旧命令和 HardFault 回归测试后，再评估 BLE 自动启动。

## 7. 面试表达素材

### 7.1 STAR 案例：RTOS Shell 假死

- **Situation**：加入蓝牙功能后，固件可以启动，但串口 Shell 无法进入，调试器经常停在 idle。
- **Task**：判断是 HardFault、串口故障还是调度问题，并在不破坏 CAN 控制功能的前提下恢复诊断入口。
- **Action**：检查线程优先级和 ready 条件，发现高优先级 `main` 在最小框架中永久忙循环，导致低优先级 `tshell` 饿死；用最小提交恢复完整可调度框架。
- **Result**：Shell 恢复，可持续执行 BLE 状态、堆和线程栈诊断；同时保留静态测试防止配置再次回退。

### 7.2 STAR 案例：Android 扫不到已广播设备

- **Situation**：M33 显示广播成功，但 Android 端扫描列表为空。
- **Task**：确定问题属于固件射频、广播内容、Android 权限还是 App 过滤逻辑。
- **Action**：先用独立 PC Bleak 证明广播名称和 NUS UUID 正确，再对照 Android 扫描日志，逐步取消硬件过滤，定位到 Android 36 的 Service UUID ScanFilter 没有回调；改为无硬件过滤加严格软件白名单，并保留连接后的 NUS 能力校验。
- **Result**：手机发现 `RehabArm`，完成 GATT 服务发现、通知订阅、MTU 247 协商和 `link_ready`。

### 7.3 可回答的工程问题

- 为什么 RTOS 停在 idle 不一定是死机？
- 为什么函数缺少原型可能在 Cortex-M 上演变为 HardFault？
- 如何证明烧录的是刚编译出的镜像？
- 如何分层定位 BLE“扫不到”和“连不上”？
- 为什么软件扫描过滤可以作为兼容方案，但连接后仍需验证 Service/Characteristic/CCCD？
- 为什么医疗康复设备的 BLE 命令只能是高层意图，不能直接绕过实时控制安全门？
- 如何使用堆低水位、线程栈水位、消息计数和超时计数判断新增功能是否影响系统稳定性？
