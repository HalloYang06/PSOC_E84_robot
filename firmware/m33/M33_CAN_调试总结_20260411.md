## 1. 最后一次“不刷屏”修改的原理

这次刷屏现象是：

```text
[control] probe reply motor=0x01 uid=0x0137340c0230719f
[control] probe reply motor=0x7F uid=0x0137340c0230719f
```

它反复出现，看起来像有两个电机 ID。实际不是。`0x01` 是真实电机 ID，`0x7F` 是广播探测 ID。广播探测会让设备用自己的 UID 回包，所以同一个 UID 在 `0x01` 和 `0x7F` 之间出现，并不能说明有两个电机。

这次修改做了三个保护。

第一，增加“当前 probe 请求”状态：

```c
static rt_bool_t s_motor_probe_pending = RT_FALSE;
static rt_uint8_t s_motor_probe_expected_id = 0U;
```

发送 `cmd_motor_probe 0x01` 前，先记录当前只等待 `0x01`：

```c
s_last_motor_probe.valid = RT_FALSE;
s_motor_probe_expected_id = motor_id;
s_motor_probe_pending = RT_TRUE;
```

收到 Get_ID 回包时，如果当前没有 pending 请求，直接丢弃，不再打印：

```c
if (!s_motor_probe_pending)
{
    rt_mutex_release(&s_data_lock);
    return;
}
```

如果当前请求不是广播 `0x7F`，并且回包里的电机 ID 不等于期望 ID，也丢弃：

```c
if ((s_motor_probe_expected_id != MOTOR_PRIVATE_BROADCAST_ID) &&
    (probe.motor_id != s_motor_probe_expected_id))
{
    rt_mutex_release(&s_data_lock);
    return;
}
```

真正接收到期望回包后，保存结果并清掉 pending：

```c
s_last_motor_probe = probe;
s_motor_probe_pending = RT_FALSE;
```

所以现在 `cmd_motor_probe 0x01` 只会接受一次 `motor=0x01` 的回包，旧回包、广播回包、其他 ID 回包都不会刷屏。

第二，每轮 CAN 轮询增加最大 drain 限制：

```c
#define CONTROL_CAN_RX_DRAIN_LIMIT 16U
```

每次控制线程最多连续处理 16 帧：

```c
while ((drained < CONTROL_CAN_RX_DRAIN_LIMIT) &&
       (ifx_can_direct_recv(&msg) == (rt_ssize_t)sizeof(msg)))
{
    ctrl_handle_can_message(&msg);
    drained++;
}
```

这样即使 FIFO 或驱动状态异常，也不会让 `ctrl_can` 线程长时间占住 CPU 或疯狂打印。

第三，去掉底层重复 ACK：

当前 `CAN_config.h` 里 FIFO 配置是：

```c
.topPointerLogicEnabled = false,
```

查 PDL 源码后确认，`Cy_CANFD_ExtractMsgFromRXBuffer()` 在这种配置下会自己写 `RXF0A` 完成 ACK。因此我们不应该在 `ifx_can_direct_recv()` 里再调用一次 `Cy_CANFD_AckRxFifo()`。重复 ACK 可能造成跳帧或错过下一帧，虽然它不一定是刷屏主因，但这是底层正确性修复。

## 2. 本次 CAN 调试的关键问题

### 问题 1：设备名不是 `canfd0`，而是 `can0`

早期执行控制层初始化时出现：

```text
[control] can device canfd0 not found
```

后来确认 RT-Thread 注册出来的设备名是 `can0`，所以正确命令是：

```text
cmd_control_init can0
```

这个问题的教训是：底层外设叫 CANFD0，不代表 RT-Thread 设备名也叫 `canfd0`。要以 RT-Thread device 注册名为准。

### 问题 2：CANFD0 寄存器访问最开始会卡死

早期 `cmd_can_reg_probe` 会卡在：

```text
[can_probe] about to read CTL
```

后面加入 MMIO 初始化后，寄存器可以正常读取：

```text
[can_probe] init hsiom mmio
[can_probe] init gpio mmio
[can_probe] init canfd0 mmio
[can_probe] about to read CTL
[can_probe] ctl=0x00000000
[can_probe] about to read STATUS
[can_probe] status=0x00000000
[can_probe] about to read IR
[can_probe] ir=0x00000000
```

对应修复在 `ifx_canfd0_prepare_hw()`：

```c
Cy_SysClk_PeriGroupSlaveInit(CY_MMIO_HSIOM_PERI_NR,
                             CY_MMIO_HSIOM_GROUP_NR,
                             CY_MMIO_HSIOM_SLAVE_NR,
                             CY_MMIO_HSIOM_CLK_HF_NR);

Cy_SysClk_PeriGroupSlaveInit(CY_MMIO_GPIO_PERI_NR,
                             CY_MMIO_GPIO_GROUP_NR,
                             CY_MMIO_GPIO_SLAVE_NR,
                             CY_MMIO_GPIO_CLK_HF_NR);

Cy_SysClk_PeriGroupSlaveInit(CY_MMIO_CANFD0_PERI_NR,
                             CY_MMIO_CANFD0_GROUP_NR,
                             CY_MMIO_CANFD0_SLAVE_NR,
                             CY_MMIO_CANFD0_CLK_HF_NR);
```

这里的重点不是“CAN 协议错”，而是 M33 non-secure 侧访问 HSIOM/GPIO/CANFD0 这几个 MMIO 从设备前，需要把对应访问链路初始化好。

### 问题 3：时钟和引脚必须手动准备

最终工作版本里，CANFD0 PCLK、RX、TX 都在底层准备函数里显式配置。

PCLK：

```c
Cy_SysClk_PeriPclkAssignDivider(PCLK_CANFD0_CLOCK_CAN_EN0, CY_SYSCLK_DIV_8_BIT, 4U);
Cy_SysClk_PeriPclkAssignDivider(PCLK_CANFD0_CLOCK_CAN_EN1, CY_SYSCLK_DIV_8_BIT, 4U);
Cy_SysClk_PeriPclkSetDivider(PCLK_CANFD0_CLOCK_CAN_EN0, CY_SYSCLK_DIV_8_BIT, 4U, 4U);
Cy_SysClk_PeriPclkSetDivider(PCLK_CANFD0_CLOCK_CAN_EN1, CY_SYSCLK_DIV_8_BIT, 4U, 4U);
Cy_SysClk_PeriPclkEnableDivider(PCLK_CANFD0_CLOCK_CAN_EN0, CY_SYSCLK_DIV_8_BIT, 4U);
Cy_SysClk_PeriPclkEnableDivider(PCLK_CANFD0_CLOCK_CAN_EN1, CY_SYSCLK_DIV_8_BIT, 4U);
```

引脚：

```c
rx_pin_cfg.hsiom = P16_0_CANFD0_TTCAN_RX0;
Cy_GPIO_Pin_Init(P16_0_PORT, P16_0_PIN, &rx_pin_cfg);

tx_pin_cfg.hsiom = P16_1_CANFD0_TTCAN_TX0;
Cy_GPIO_Pin_Init(P16_1_PORT, P16_1_PIN, &tx_pin_cfg);
```

当前硬件接线对应：

```text
P16.0 -> CANFD0 RX
P16.1 -> CANFD0 TX
```

### 问题 4：电机不支持 CAN FD，必须按经典 CAN 发帧

用户确认电机不支持 CAN FD，所以驱动里强制 classic CAN：

```c
#define IFX_CAN_FORCE_CLASSIC 1
```

发送时使用标准 CAN 帧格式：

```c
can->tx_t1.brs = false;
can->tx_t1.fdf = CY_CANFD_FDF_STANDARD_FRAME;
```

这里容易混淆：MCU 外设名字叫 `CANFD0`，但它可以发送 classic CAN 帧。电机不支持 CAN FD 时，不能发 `FDF=CAN FD` 的帧。

### 问题 5：bitrate 公式和最终 1Mbps 参数

文档中的公式是：

```text
bitrate = pclk / ((prescaler + 1) * (timeSegment1 + timeSegment2 + 3))
```

当前 PCLK 打印为：

```text
pclk=20000000
```

当前参数为：

```c
#define BSP_CANFD0_NOMINAL_PRESCALER 0U
#define BSP_CANFD0_NOMINAL_TSEG1 14U
#define BSP_CANFD0_NOMINAL_TSEG2 3U
#define BSP_CANFD0_NOMINAL_SJW 3U
```

代入公式：

```text
20,000,000 / ((0 + 1) * (14 + 3 + 3)) = 1,000,000
```

所以当前 nominal bitrate 是 1Mbps。这个和电机上位机能通信的波特率匹配后，才收到有效回包。

### 问题 6：最小 PDL 链路先打通，再回灌控制层

最小链路命令是：

```text
cmd_can_init_min
cmd_can_send_probe 0x01
cmd_can_poll_once
```

当时收到：

```text
[can_min] poll ret=0 id=0x000001fe dlc=8 xtd=1
9F 71 30 02 0C 34 37 01
```

这个结果非常关键，说明硬件、接线、收发器、电机、波特率、协议帧格式都已经基本正确。后续如果 `cmd_control_init` 卡住，就不应该继续怀疑电机或接线，而应该回到 RT-Thread CAN 包装层和控制层线程上排查。

### 问题 7：RT-Thread CAN open/interrupt 路径会卡住

旧路径会进入：

```text
[control] init step6 open can flags=0x0503
...
[drv_can] set_int polling mode nvic disabled
[con
```

这条路径仍然经过 RT-Thread 的 `rt_device_open()`、CAN 设备 ops、rx/tx indicate 以及中断/轮询适配。bring-up 阶段为了先让电机控制层跑起来，我们没有继续死磕这条路径，而是把已经验证过的最小 PDL 收发封装成：

```c
rt_err_t ifx_can_direct_init(void);
rt_err_t ifx_can_direct_send(const struct rt_can_msg *msg);
rt_ssize_t ifx_can_direct_recv(struct rt_can_msg *msg);
```

控制层通过：

```c
#define CONTROL_CAN_USE_DIRECT_PDL 1
```

直接走 PDL 路径。初始化日志变为：

```text
[control] init step6 direct pdl can init
[drv_can] direct ready pclk=20000000 nbtp=0x06000e03 rxf0s=0x00000000
[control] init step7 direct open ret=0
[control] init step8 direct rx poll mode
```

这个策略是“先打通业务链路，再回头完善 RT-Thread CAN 驱动”。它不是最终最优雅的架构，但非常适合当前阶段推进电机联调。

### 问题 8：烧录过旧镜像导致现象回退

有一次板子日志仍然出现：

```text
[control] init step6 open can flags=0x0503
```

但本地最新 `rtthread.bin` 已经不包含 `open can flags`，而包含：

```text
direct pdl can init
direct ready
```

后来发现 `rtthread.hex` 时间比 `rtthread.bin` 旧，IDE 可能烧了旧 hex。重新从当前 elf 生成 hex 后，板子才跑到正确路径。

以后遇到“我明明改了但板子行为没变”，先做这个判断：

```text
如果日志里还有 open can flags，就是旧镜像。
如果日志里有 direct pdl can init，就是新镜像。
```

## 3. RobStride 私有 Get_ID 帧解析

请求帧扩展 ID 组成：

```text
ext_id = comm_type << 24 | master_id << 8 | motor_id
```

Get_ID 请求：

```text
comm_type = 0x00
master_id = 0xFD
motor_id = 0x01
```

所以请求 ID 是：

```text
0x0000FD01
```

广播探测：

```text
0x0000FD7F
```

电机回包曾经抓到：

```text
id=0x000001fe
data=9F 71 30 02 0C 34 37 01
```

这个 ID 的解析：

```text
低 8 位 0xFE：Get_ID 回复标记
中间 16 位低字节 0x01：真实电机 ID
高位 comm_type 0x00：Get_ID 类型
```

数据按小端拼 UID：

```text
9F 71 30 02 0C 34 37 01 -> 0x0137340c0230719f
```

所以上位机里看到 `CAN: id:137340c2` 或类似内容时，不要直接把它当电机 ID。真正的电机 CAN ID 在协议扩展 ID 的中间字段里，本次确认是 `0x01`。

## 4. 当前推荐测试流程

每次烧录后，先确认是不是新镜像：

```text
cmd_control_init can0
```

正确输出必须包含：

```text
[control] init step6 direct pdl can init
[drv_can] direct ready pclk=20000000 nbtp=0x06000e03
control_init ret=0
```

然后只测真实电机 ID：

```text
cmd_motor_probe 0x01
cmd_motor_probe_last
```

期望输出：

```text
[control] probe reply motor=0x01 uid=0x0137340c0230719f
MOTOR_PROBE: id=0x01 uid=0x0137340c0230719f
```

如果要测广播，单独测，不要和普通探测混在一起看：

```text
cmd_motor_probe 0x7F
cmd_motor_probe_last
```

广播收到真实电机 `0x01` 回包是正常的。不要把 `0x7F` 当成真实电机 ID。

如果控制层异常，再退回最小 PDL 链路：

```text
cmd_can_init_min
cmd_can_send_probe 0x01
cmd_can_poll_once
```

如果最小 PDL 能收到 `id=0x000001fe`，说明硬件链路没坏，应继续查控制层。如果最小 PDL 也收不到，再查接线、供电、收发器、终端电阻、波特率和电机状态。

## 6. 后续建议

短期建议继续使用 direct PDL polling 路径，把电机 enable、stop、读参数、写参数、速度控制、位置控制逐个验证完。这个路径已经证明可以初始化、发送、接收，并且不会卡在 `rt_device_open()`。

中期如果要回归 RT-Thread 标准 CAN device 路径，需要单独开一个任务处理 `rt_device_open()`、中断掩码、`rt_hw_can_isr()`、rx/tx event、FIFO ACK 的配合。不要在电机业务联调时混着修，否则问题域会重新变大。

长期建议保留 `cmd_can_init_min`、`cmd_can_send_probe`、`cmd_can_poll_once` 这组三个最小命令。它们是硬件链路的“听诊器”：以后任何 CAN 问题，都可以先用它们判断是底层链路问题，还是控制层/业务层问题。
