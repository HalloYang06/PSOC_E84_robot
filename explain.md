# 第一版 VLA 方案说明

## 1. 方案定位

当前第一版 VLA 方案以根目录下的 `server.py` 作为统一服务入口，目标是先把视觉输入到动作输出的推理链路跑通，并以最小成本接入现有机器人控制系统。

这一版的重点不是完整训练框架封装，也不是复杂的多模块编排，而是提供一个可以直接启动、加载模型、对外提供推理接口的轻量服务。

## 2. 整体思路

该方案基于 OpenVLA 的 Hugging Face AutoClass 推理方式实现：

- 服务启动后自动异步加载 VLA 模型。
- 外部系统通过 HTTP 接口提交图像和指令。
- 服务端将图像和自然语言指令整理成 OpenVLA 所需输入格式。
- 模型执行 `predict_action()`，返回动作向量。

整体链路可以概括为：

`相机图像 -> Base64 编码 -> /act 接口 -> OpenVLA 推理 -> 动作向量 -> 下游控制模块`

## 3. 当前实现内容

第一版已经具备以下能力：

- 基于 FastAPI 提供 REST 服务。
- 启动时后台加载模型，避免首次请求时才开始初始化。
- 提供根路径 `/` 用于查看服务状态和当前模型信息。
- 提供健康检查接口 `/healthz`。
- 提供动作推理接口 `/act`。
- 根据设备情况自动选择推理精度：
  - 支持 CUDA 且支持 bf16 时使用 `bfloat16`
  - 支持 CUDA 但不支持 bf16 时使用 `float16`
  - CPU 场景下使用 `float32`
- 在显存较小的 GPU 场景下，支持自动启用 4bit 加载。

## 4. 服务接口定义

### 4.1 根接口

`GET /`

用于查看当前服务状态，返回内容包括：

- 服务名称
- 模型状态
- 模型 ID
- CUDA 是否可用
- 是否启用 4bit
- 接口路径
- 当前错误信息

### 4.2 健康检查接口

`GET /healthz`

返回规则如下：

- 模型已经完成加载时返回 `200`
- 模型加载失败时返回 `500`
- 模型仍在加载中时返回 `503`

这个接口适合部署系统做探活和就绪检查。

### 4.3 动作推理接口

`POST /act`

请求体格式：

```json
{
  "image_base64": "......",
  "instruction": "pick up the cup"
}
```

返回格式：

```json
{
  "action": [0.1, 0.2, -0.3, 0.0, 0.5, -0.1, 1.0]
}
```

其中：

- `image_base64` 为 RGB 图像的 Base64 编码结果
- `instruction` 为自然语言任务描述
- `action` 为模型输出的连续动作向量，可直接交给下游机器人控制逻辑进一步解释和执行

## 5. 启动方式

默认启动命令：

```bash
python server.py
```

默认监听地址：

```text
0.0.0.0:8000
```

可通过环境变量调整运行参数：

```text
OPENVLA_MODEL_ID
OPENVLA_HOST
OPENVLA_PORT
OPENVLA_UNNORM_KEY
OPENVLA_LOAD_IN_4BIT
```

默认模型为：

```text
openvla/openvla-7b
```

## 6. 适合作为第一版的原因

这套方案适合作为第一版 VLA 方案，主要因为：

- 结构简单，服务入口清晰，便于快速接入和调试。
- 与 OpenVLA 官方的 Hugging Face 推理方式保持一致，后续迁移成本低。
- 对外只暴露标准 HTTP 接口，方便和机器人控制端、相机采集端、任务编排端解耦。
- 加载逻辑和推理逻辑集中在一个文件中，便于快速验证。
- 后续如果需要替换模型、增加鉴权、增加日志、增加并发控制，都有明确扩展点。

## 7. 当前边界与限制

第一版方案已经能满足“单模型推理服务”的目标，但当前仍有一些明确边界：

- 当前接口只支持单张图像加单条指令，不支持批量推理。
- 当前请求格式使用 `image_base64`，更偏向通用接入，尚未做高吞吐优化。
- 当前默认 `unnorm_key` 固定为 `bridge_orig`，更适合先围绕默认数据统计跑通流程。
- 当前实现主要针对 `openvla/openvla-7b` 使用场景。
- 如果后续切换到特殊版本模型或本地微调模型，建议补充更完整的 prompt 兼容和 `dataset_statistics.json` 加载逻辑。
- 当前版本只解决“VLA 推理服务”问题，不包含机器人动作执行、异常恢复、任务调度和闭环控制。

## 8. 适用场景

这一版最适合以下场景：

- 先验证 OpenVLA 是否能接入现有机器人系统
- 先打通感知到动作的基本调用链路
- 作为后续 VLA 平台化之前的验证版本
- 用于内部 Demo、联调和小规模实验

## 9. 后续建议迭代方向

在第一版跑通之后，建议按优先级逐步补充：

- 增加不同模型版本的 prompt 兼容逻辑
- 支持本地微调模型的 `dataset_statistics.json` 自动加载
- 支持请求级别传入 `unnorm_key`
- 增加更明确的错误码和日志输出
- 增加模型加载耗时、推理耗时等监控指标
- 增加鉴权、限流和并发保护
- 根据下游控制需求定义稳定的动作协议

## 10. 总结

第一版 VLA 方案的核心目标是：用最小实现，把 OpenVLA 作为一个可调用、可加载、可返回动作结果的服务跑起来。

从工程角度看，这一版已经能够承担“VLA 服务原型”的角色。它适合先作为基础版本落地，在验证链路稳定后，再逐步补齐模型兼容性、部署能力和生产化能力。
