# AI 协作平台第一版开发入口

版本：v0.1  
当前目标：只开发第一版 AI 合作平台 MVP  
原则：先跑通闭环，再逐步增强

---

## 1. 第一版定位

第一版要做的是一个能演示、能试用、能继续扩展的 AI 合作平台雏形。

第一版不追求完整产品化，重点是：

```text
人类可以管理 AI。
AI 可以认领角色。
平台可以创建任务。
Runner 可以执行任务。
代码可以进入任务分支。
人类可以审查 diff。
token 可以被统计。
上下文过长可以交接。
硬件操作必须人类确认。
首页能以研发基地方式展示状态。
```

---

## 2. 第一版核心闭环

必须优先打通：

```text
项目管理
  -> AI 成员管理
  -> Runner 注册
  -> 任务管理
  -> Git 分支
  -> Runner 执行
  -> 日志和 diff
  -> token 统计
  -> 审查
  -> 人工确认
  -> 合并/回滚
```

这个闭环没跑通前，其他功能都不能抢优先级。

---

## 3. 第一版必须开发的模块

### 3.1 前端

必须做：

```text
研发基地首页
项目管理页面
AI 成员管理页面
Runner 状态页面
任务列表和任务详情
任务 diff 和日志页面
token 成本简表
上下文健康显示
人工确认弹窗
```

第一版研发基地包含：

```text
顶部状态栏
老板办公室
任务大厅
AI 工位区
代码车间
硬件实验室确认卡
财务室 token 卡片
急救站失败/交接卡片
每日战报入口
```

暂不做：

```text
复杂地图拖拽
复杂角色动画
3D 场景
抽卡系统
无工程意义的游戏数值
```

### 3.2 后端

必须做：

```text
用户基础登录，可先简化
项目 CRUD
AI 成员 CRUD
Runner 注册和心跳
任务 CRUD
任务状态流转
Git 分支和 diff 服务
token 使用记录
上下文健康记录
交接包记录
人工确认记录
审计日志
```

暂不做：

```text
完整企业 RBAC
多租户
复杂审批流
商业计费系统
插件市场
```

### 3.3 Runner

必须做：

```text
Runner 注册
Runner 心跳
Runner 能力上报
Runner 拉取任务
Runner 创建工作区
Runner 执行安全命令
Runner 回传日志
Runner 回传任务结果
```

第一版 Runner 可以先支持：

```text
git
node
python
shell 受限命令
```

暂不做：

```text
真实硬件自动控制
自动烧录
复杂沙箱
多 Runner 调度优化
```

### 3.4 Git

必须做：

```text
绑定仓库
创建任务分支
查看 commit
查看 diff
记录任务分支
合并前人工确认
回滚到指定 commit
```

可以先简化：

```text
第一版可先用本地 Git 或 GitHub。
Gitea/Forgejo 镜像可以作为部署增强。
```

### 3.5 AI 协作

必须做：

```text
AI 身份认领
AI 角色状态表
任务指派给 AI
AI 输出任务报告
上下文健康显示
手动生成交接包
```

第一版需求管理库可以先做轻量：

```text
创建需求单
指定接收 AI
记录回复
关联任务
关闭需求
```

暂不做：

```text
复杂自动路由
知识图谱
跨项目记忆
自动模型竞标
```

### 3.6 硬件安全

必须做：

```text
人工确认等级 H0-H4
硬件任务确认卡
禁止 AI 自动执行真实硬件操作
硬件日志上传入口
硬件风险提示
```

第一版硬件相关只做辅助：

```text
AI 生成检查清单
人类填写测量结果
AI 分析日志
人类决定下一步
```

## 4. 第一版推荐 AI 角色

第一版只启用核心角色，不要一开始开太多。

### 必须角色

| 角色 | 第一版职责 | 开发思路 |
|---|---|---|
| AI-Boss | 统筹 AI 团队 | 看总进度、拆阶段、发现阻塞、安排接手，不直接写代码 |
| AI-PM | 任务管理 | 把文档拆成任务，维护优先级、阻塞项、验收标准 |
| AI-ARCH | 架构 | 确定模块边界、数据流、API 风格，避免发散 |
| AI-DEVOPS | 环境 | 搭 Docker Compose、环境变量、基础服务 |
| AI-BE-LEAD | 后端骨架 | 搭 FastAPI 框架、鉴权、错误格式、服务层 |
| AI-FE-LEAD | 前端骨架 | 搭 Next.js、路由、布局、组件规范 |
| AI-FE-GAME | 研发基地首页 | 实现游戏化首页和核心卡片 |
| AI-BE-PROJECT | 基础 API | 项目、AI、Runner API |
| AI-BE-TASK | 任务 API | 任务、需求、状态流转 API |
| AI-RUNNER-LEAD | Runner | 注册、心跳、执行任务、回传日志 |
| AI-GIT | Git | 分支、commit、diff、回滚 |
| AI-QA | 测试 | 建立测试和第一版验收清单 |
| AI-DOCS | 文档 | 维护 README、接口说明、使用说明 |

### 可选角色

| 角色 | 第一版职责 | 开发思路 |
|---|---|---|
| AI-CONTEXT-WATCHER | 上下文健康 | 先做 token 占用、健康等级、交接包 |
| AI-SECURITY | 安全权限 | 先做 H0-H4、审计日志和危险操作提示 |
| AI-SECRETS | 密钥 | 先做不暴露明文 token 的规范和接口 |
| AI-FE-REVIEW | 审查可视化 | 做 diff、日志、token、交接包页面 |
| AI-REQ | 需求单 | 做轻量需求单，不做复杂自动路由 |

## 5. 每个 AI 第一件事

每个 AI 进入项目后，第一件事必须是确认角色。

流程：

```text
1. 阅读 README.md。
2. 阅读本文件。
3. 阅读 AI身份认领与统一协作规则.md。
4. 查看角色是否已被认领。
5. 提交身份认领。
6. 等待人类总工程师确认。
7. 只做确认后的任务。
```

如果角色已被认领：

```text
不能抢角色。
可以申请副手。
可以申请临时接手某个任务。
可以请求人类总工程师替换。
```

## 7. 第一版开发顺序

建议按这个顺序执行：

```text
1. 建立项目代码骨架
2. 搭 Docker Compose
3. 搭后端 FastAPI
4. 搭前端 Next.js
5. 做项目/AI/Runner 基础管理
6. 做任务管理
7. 做 Git 分支和 diff
8. 做 Runner 注册和执行
9. 做 token 统计和上下文健康
10. 做人工确认和审计日志
11. 做研发基地首页
12. 做第一版演示流程
```

## 8. 第一版演示场景

演示场景：

```text
1. 打开研发基地首页。
2. 显示 AI-Boss、前端 AI、后端 AI、Runner AI、Git AI。
3. 创建任务：实现 Runner 心跳接口。
4. 指派给 AI-BE-PROJECT 和 AI-RUNNER-LEAD。
5. 平台创建任务分支。
6. Runner 执行测试。
7. 前端展示日志和 diff。
8. token 成本被记录。
9. 上下文健康显示为绿色。
10. 人类确认合并。
11. 每日战报显示任务完成。
```

硬件演示只做安全卡：

```text
创建硬件调试任务
  -> AI 生成检查清单
  -> 平台提示 H3 需要人工确认
  -> 人类填写测量结果
  -> AI 分析日志
```

不演示 AI 自动控制真实硬件。
