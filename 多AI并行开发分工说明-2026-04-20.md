# 多AI并行开发分工说明 - 2026-04-20

## 目标

当前项目已经不再按“产品页游戏化”思路推进，而是直接转向：

**开发一款真正像市面上大型经营页游的游戏。**

但这款游戏的世界观、经营循环、建筑功能、资源系统、角色职责，最终都要映射到：

- AI 协作开发
- 嵌入式 / 机器人项目研发
- 多人协作
- 多电脑 / 多 AI 调度
- 人工审批与安全边界

一句话：

**先做成真正好玩的经营游戏，再让经营行为自然等价于开发嵌入式机器人项目。**

---

## Boss 规则

我作为 `Boss / 总调度 AI`，只负责：

1. 定范围，不直接包办所有具体实现。
2. 拆任务，防止多人同时改同一块。
3. 控制“像游戏”与“对真实需求有映射”这两条线不要跑偏。
4. 统一验收标准、文档格式、Git 交流规则。
5. 决定谁先做、谁后做、谁依赖谁。

我不希望其他 AI 做的事：

1. 不要随便改总方向。
2. 不要直接重写别人负责的文件。
3. 不要在没有文档交接的情况下硬推到 `main`。
4. 不要把高风险硬件自动化做成默认行为。
5. 不要做“只有文字说明、没有真实系统”的假功能。

---

## 总体并行策略

现在不是单线推进，而是 5 条线同时推进：

1. 游戏主循环线
目标：先把它做成一个真的能上瘾的经营游戏。

2. 世界地图与场景线
目标：让地图、建筑、房间、出入口、层次关系像成熟页游。

3. 系统与数值线
目标：建立资源、产能、冷却、成长、产线、解锁、稀缺性。

4. 真实需求映射线
目标：把“开发嵌入式机器人”的真实动作映射到游戏系统，而不是写字面标签。

5. 协作与留痕线
目标：让多 AI 并行开发时，靠 Git 和文档交接，不互相打架。

---

## AI 角色分工

### AI-1：游戏主循环策划与实现

负责范围：

- 经营核心循环
- 新手期 5 分钟体验
- 中期成长循环
- 建筑解锁顺序
- 玩家每天/每轮要做什么
- “种植、养殖、生产、交付、升级”如何构成闭环

必须产出：

1. 一份 `游戏主循环设计文档`
2. 一份 `建筑解锁与成长节奏表`
3. 实际代码改动

不负责：

- 房间视觉美术细节
- Git 协作协议
- 后端联机架构

验收标准：

- 玩家进入后 30 秒内知道要干嘛
- 前 5 分钟内至少有 3 次明确反馈
- 不能只有说明板，必须有循环

建议文件边界：

- `apps/web/public/harvest-moon-phaser3-game/index.html`
- 如果拆脚本，可新建 `apps/web/public/harvest-moon-phaser3-game/game-systems/*`

---

### AI-2：地图、建筑、房间、场景玩法

负责范围：

- 外场地图的建筑语义
- 室内房间的功能分布
- 哪些房子能进，进去后做什么
- 建筑摆位和房间结构是否合理
- 入口提示、建筑等级外观、可互动对象

必须产出：

1. 一份 `地图建筑功能分配文档`
2. 一份 `房间功能对照表`
3. 实际截图对照说明

不负责：

- 数值平衡
- Git 协议
- 多人联机同步方案

验收标准：

- 玩家远看地图就能理解主要功能区
- 不同房子进入后要有明确不同玩法
- 不能只是“右侧抽屉标题不同”

建议文件边界：

- `apps/web/public/harvest-moon-phaser3-game/index.html`
- 若拆配置，优先抽成 `scene-config/*` 或 `buildings-config/*`

注意：

- 这个 AI 与 AI-1 会同时碰玩法，但 AI-2 不负责主循环，只负责场景承载。

---

### AI-3：资源、产能、升级、经营数值

负责范围：

- 资源种类设计
- 建筑产能
- 冷却、成长时间、解锁条件
- 任务转化效率
- AI 动物、电脑、协作者的经营属性
- 资源消耗与收益关系

建议的游戏内经营资源：

- 需求单
- 任务种子
- AI 精力
- 电脑算力
- 审批点
- 交付包
- 团队士气
- 知识值 / 工程经验

必须产出：

1. 一份 `经营资源与数值表`
2. 一份 `建筑升级树`
3. 一份 `前中期平衡草案`

不负责：

- 地图外观
- 联机架构
- 文档协作协议

验收标准：

- 至少有 3 类真正稀缺资源
- 至少有 2 条不同产线
- 玩家会面临取舍，不是全都点一遍就行

---

### AI-4：真实需求映射设计

负责范围：

- 把真实平台需求翻译成游戏系统
- 让“经营农场”真正等价于“开发嵌入式机器人项目”
- 把以下真实动作映射成系统：
  - 收需求
  - 拆任务
  - 派 AI
  - 派电脑
  - Runner 执行
  - 文件边界
  - 交接
  - 人工审批
  - 构建
  - 交付

必须产出：

1. 一份 `真实研发动作 -> 游戏机制映射表`
2. 一份 `高风险动作人工确认清单`
3. 一份 `不能游戏化错位的红线列表`

不负责：

- 前端视觉实现
- 数值平衡
- Git 协议

验收标准：

- 每个关键系统都能追溯到真实研发动作
- 高风险硬件相关动作必须保留人工门槛
- 不是简单把按钮换名字

---

### AI-5：前端 HUD、经营页游信息层与反馈

负责范围：

- 顶部资源条
- 左右功能栏
- 建筑升级反馈
- 飘字、产出、成熟、缺资源提示
- 玩家任务列表
- 活动面板、成就、阶段目标
- 页游式信息密度，但不能乱

必须产出：

1. 一份 `HUD 结构草图说明`
2. 一份 `反馈系统清单`
3. 对应截图

不负责：

- 经营数值规则
- 联机同步
- 需求映射总设计

验收标准：

- 像成熟经营页游，不像后台抽屉
- 有资源增长感、建筑经营感、成长感
- 页面信息多，但层次清楚

---

### AI-6：Git 留痕、交接文档、协作规则

负责范围：

- 统一分支规范
- 提交说明规范
- AI 间文档交接模板
- PR / commit 留痕要求
- 截图验收模板

必须产出：

1. 一份 `AI 协作 Git 规范`
2. 一份 `交接文档模板`
3. 一份 `截图验收模板`

不负责：

- 具体玩法实现
- 地图视觉
- 数值平衡

验收标准：

- 任何 AI 接手时，能直接看文档继续干
- 不依赖长聊天记录才能交接
- 每条线都能在 Git 上独立追溯

---

## Git 协作规则

所有其他 AI 必须遵守：

1. 不直接推 `main`
2. 一条线一个分支
3. 分支名必须清楚表达范围

推荐命名：

- `ai/game-loop-core`
- `ai/building-scenes`
- `ai/economy-balance`
- `ai/embedded-mapping`
- `ai/game-hud-feedback`
- `ai/collab-git-protocol`

提交要求：

1. 每次提交只做一类事情
2. commit 标题必须直白
3. 改完必须附文档或截图

推荐 commit 风格：

- `Build first pass of game economy loop`
- `Add building progression config for farm base`
- `Map embedded workflow to production buildings`
- `Refine HUD resource bar and feedback states`

---

## 文档交流规则

不靠长聊天，不靠口头说，统一靠文档。

每个 AI 都必须维护自己的一份文档：

- `docs/ai-handoffs/<role-name>.md`

文档里至少写：

1. 我负责什么
2. 我不负责什么
3. 我改了哪些文件
4. 我下一步准备做什么
5. 哪些点需要别人接
6. 当前风险是什么

推荐模板：

```md
# 角色：

## 负责范围

## 已完成

## 修改文件

## 当前截图/验证

## 待他人接力

## 风险
```

## 当前建议的第一轮并行拆法

### 第一优先级

1. AI-1 先做真正的经营主循环
2. AI-3 同时做资源和数值体系
3. AI-4 同时做真实需求映射表

这三条定了，游戏才会真的立住。

### 第二优先级

4. AI-2 把地图与房间按循环重排
5. AI-5 把 HUD 和反馈做成页游感

### 持续并行

6. AI-6 全程跟着做 Git 和文档规范，不要最后补

---

## 现阶段明确边界

### 可以先放一放的

- 多人联机实时同步代码
- 真后端生产级架构
- 硬件自动化接入
- 很远期的平台权限体系

### 现在必须立刻做实的

- 游戏主循环
- 资源系统
- 建筑成长
- 房间玩法差异
- HUD 与反馈
- 真实需求映射
- Git 文档交接规范

---

## Boss 当前调度结论

现阶段不是“继续补几个房间功能”，而是：

**重建成一款真正的经营页游原型。**

优先级如下：

1. 先立经营循环
2. 再立资源和升级
3. 再让建筑和房间承接循环
4. 再把嵌入式研发映射进去
5. 再做多人和更深的平台能力

如果其他 AI 来协作，我会按这份文档给他们分包，不再让大家同时乱改同一层。
