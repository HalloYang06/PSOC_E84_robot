# 第一版后端任务拆解

版本：v0.1  
范围：只覆盖第一版 MVP（以 `AI协作平台第一版开发入口.md` 为准）

---

## 1) 第一版后端模块清单（最小集合）

第一版后端必须支撑“项目 -> AI -> Runner -> 任务 -> 日志/diff -> token -> 审查/人工确认 -> 合并/回滚记录”的闭环。

模块最小集合：

1. `health`
   - 目标：服务存活检查
   - API：`GET /api/health`

2. `auth`（第一版可极简）
   - 目标：为未来权限预留钩子；第一版可先简单
   - API：可先只做占位或最简管理员登录

3. `projects`
   - 目标：创建/编辑/查询项目；绑定仓库信息
   - API：`/api/projects`

4. `agents`
   - 目标：管理 AI 成员（角色、接入方式、权限等级、预算、目录边界）
   - API：`/api/agents`

5. `runners`
   - 目标：Runner 注册、心跳、能力上报、在线状态
   - API：`/api/runners/register`、`/api/runners/heartbeat`、`/api/runners`

6. `tasks`
   - 目标：任务 CRUD、状态流转、事件记录、关联分支/AI/Runner
   - API：`/api/tasks`、`/api/tasks/{id}`、`/api/tasks/{id}/events`

7. `git`（第一版可以“记录为主，能力为辅”）
   - 目标：记录 task->branch->commit；可选提供 diff/commit 列表
   - API：`/api/projects/{id}/git/*`（或 `/api/git/*`）

8. `usage`
   - 目标：token/成本记录（第一版允许手动上报或 Runner 上报）
   - API：`/api/usage/*`

9. `context + handoffs`
   - 目标：上下文健康记录；交接包生成/查询
   - API：`/api/tasks/{id}/context-health`、`/api/tasks/{id}/create-handoff`、`/api/tasks/{id}/handoffs`

10. `approvals`（人工确认）
   - 目标：记录 H0-H4 确认；高风险操作必须有确认记录
   - API：`/api/approvals/*` 或 `POST /api/tasks/{id}/approve`

11. `audit`
   - 目标：记录关键操作审计日志（写入必须有；查询可后置）
   - API：`/api/audit/*`（查询可选）

第一版明确不做：

```text
多租户
企业 SSO
复杂 RBAC/审批流
插件市场
自动控制真实硬件
自动烧录
AI 自动合并 main
复杂知识图谱/跨项目记忆
```

## 2) API 开发顺序（按闭环优先级）

按“先可用、再完整、再增强”的顺序：

阶段 A：骨架与规范（第 1 天）

1. `GET /api/health`
2. 统一响应格式（success/list/error）
3. 统一错误码（`PERMISSION_DENIED`、`VALIDATION_ERROR` 等）
4. request_id（用于审计与排障）

阶段 B：项目与 AI（第 2 天）

1. Projects CRUD
   - `GET /api/projects`
   - `POST /api/projects`
   - `GET /api/projects/{id}`
   - `PATCH /api/projects/{id}`
2. Agents CRUD
   - `GET /api/agents`
   - `POST /api/agents`
   - `GET /api/agents/{id}`
   - `PATCH /api/agents/{id}`
   - `POST /api/agents/{id}/enable`
   - `POST /api/agents/{id}/disable`

阶段 C：Runner（第 3 天）

1. Runner 注册
   - `POST /api/runners/register`
2. Runner 心跳
   - `POST /api/runners/heartbeat`
3. Runner 列表/详情
   - `GET /api/runners`
   - `GET /api/runners/{id}`

阶段 D：任务（第 3-4 天）

1. Tasks CRUD
   - `GET /api/tasks`
   - `POST /api/tasks`
   - `GET /api/tasks/{id}`
   - `PATCH /api/tasks/{id}`
2. Tasks 状态流转（动作类）
   - `POST /api/tasks/{id}/plan`（可先占位）
   - `POST /api/tasks/{id}/run`
   - `POST /api/tasks/{id}/cancel`
   - `POST /api/tasks/{id}/review`
   - `POST /api/tasks/{id}/merge`（第一版只记录结果，合并动作仍由人类在 Git 执行亦可）
3. 任务事件
   - `GET /api/tasks/{id}/events`

阶段 E：人工确认 + 审计（第 4 天）

1. 人工确认记录
   - `POST /api/tasks/{id}/approve`
   - `GET /api/tasks/{id}/approvals`
2. 审计日志写入（对写操作统一落库）

阶段 F：上下文/交接 + token（第 4-5 天）

1. 上下文健康
   - `GET /api/tasks/{id}/context-health`
   - `POST /api/tasks/{id}/summarize-context`（第一版可先存摘要文本）
2. 交接包
   - `POST /api/tasks/{id}/create-handoff`
   - `GET /api/tasks/{id}/handoffs`
3. token/成本
   - `GET /api/usage/summary`
   - `POST /api/usage/log`（允许 Runner/人工上报）

阶段 G：Git 最小能力（第 5 天）

第一版建议先做到：

1. 记录 task 关联分支/commit 信息
2. 提供 commit 列表/简单 diff（可选）
3. 记录回滚动作（需要人工确认）

## 4) 审计日志与人工确认（最小实现）

### 4.1 审计日志最小实现

第一版要求：

1. 所有“写操作”（POST/PATCH/DELETE/动作类 POST）必须写入 `audit_logs`
2. 记录至少以下字段：
   - actor_type、actor_id
   - action
   - project_id、task_id（如适用）
   - success / error
   - request_id
3. `before/after` 可先只记录关键字段或空（第一版不强制完整 diff）

建议落库点：

```text
统一中间件/依赖注入写 audit（更一致）
或者在 service 层显式写入（更可控）
第一版建议：service 层显式写入关键动作 + 中间件写入兜底
```

### 4.2 人工确认最小实现

第一版要求：

1. 对以下动作必须创建 approval 记录：
   - merge（合并记录/合并动作）
   - rollback（回滚）
   - 任何标记为 H3/H4 的硬件动作（第一版只记录，不执行）
2. Approval 状态流转：
   - pending -> approved/rejected
3. 后端规则：
   - 未 approved 的 action，相关接口返回 `HUMAN_APPROVAL_REQUIRED`
   - 允许“先记录后执行”：第一版可先只记录 approval，不自动触发真实动作

## 6) 需要人类拍板的技术点（第一版必须明确）

1. 第一版认证策略：
   - A：内网无登录（最快，但风险需控制）
   - B：单管理员账号 + 简单 token/session（推荐）
   - C：完整 RBAC（第一版不建议）

2. Git 集成第一版范围：
   - A：只记录 branch/commit/diff 链接（最小）
   - B：后端读取本地仓库生成 diff（需要部署约束）
   - C：对接 GitHub/Gitea API（工作量更大）

3. Runner 第一版执行范围：
   - A：只注册/心跳/上报能力/上报日志（最小）
   - B：可拉取任务并执行受限命令（推荐）

4. token 统计来源：
   - A：手动记录（适合 Codex 线程模式）
   - B：Runner 上报（适合 CLI/自动执行）
   - C：API 自动统计（后续版本）

5. 数据库与存储部署位置：
   - A：一台机 Docker Compose 全跑（最快）
   - B：数据库/存储单独服务器（更稳定）

6. 任务状态最小集（第一版先定死，避免反复改前端）
   - 建议：draft/ready/running/reviewing/blocked/failed/merged/rolled_back/cancelled
