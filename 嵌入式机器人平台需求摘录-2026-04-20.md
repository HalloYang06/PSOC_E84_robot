## 核心定位

平台不是一次性工具，也不是普通后台，而是一个面向嵌入式、机器人、软硬件协作项目的长期产品。

一句话定位：
人类总工程师主导的多 AI、多电脑、多 Runner 协作研发平台。

## 必须满足的真实研发需求

1. 支持多个项目、多个团队、多个 AI、多个 Runner、多台电脑协作。
2. 支持 Git 仓库、任务、需求、审批、交付、知识库、上下文交接闭环。
3. 支持嵌入式、ROS、机器人、Web、App 等不同项目模板，尤其要重点支持嵌入式机器人研发。
4. 支持硬件实验记录、串口日志、烧录检查、设备状态、风险确认。
5. 支持真实硬件安全边界：
AI 可以辅助分析，但真实硬件操作必须由人类执行和确认。

## MVP 闭环

第一版必须跑通：
创建项目 -> 添加 AI -> 注册 Runner -> 创建任务 -> 指派 AI -> 创建任务分支 -> AI/Runner 执行 -> 查看日志和 diff -> 记录 token -> 审查 -> 人类确认 -> 合并或回滚

## 和嵌入式机器人最相关的模块

1. 项目管理
要能看项目状态、阶段、模块、风险、Git 绑定、负责人。

2. AI 成员管理
要能配置 AI 身份、职责、所属 Runner、模型、可读可写目录、权限和 token 预算。

3. Runner 与电脑节点
要能登记电脑、看在线状态、任务执行状态、日志回传、资源状态，后续支持专用嵌入式编译 Runner。

4. 任务系统
要支持任务状态流转、依赖、阻塞、验收标准、自动转派、从聊天和硬件调试日志生成任务。

5. 需求系统
要支持结构化需求、接收 AI 回复、按类型区分代码/接口/文档/硬件/测试/安全需求。

6. 上下文与交接
要能统计上下文健康度、生成摘要、生成交接包、推荐接手 AI，避免长任务断档。

7. Git 协作
要能建分支、看 diff、审查、回滚、后续生成 PR/MR 和 release notes。

8. 知识库
要能沉淀文档、接口契约、决策记录、硬件手册、调试记录，后续支持多模态资料。

9. 硬件实验室
要能创建设备卡、硬件调试任务、烧录前检查、上传串口日志和照片、人工确认下一步。

10. 安全与权限
要有人类确认门槛、危险操作拦截、审批流、审计日志，不能让 AI 直接越过硬件安全边界。

## 游戏化前端必须服务的真实动作

1. 进入平台后要一眼看懂项目状态、AI 状态、风险、预算和待确认事项。
2. 地图里的建筑必须映射真实功能，不是装饰：
   - 项目总览
   - 需求入口
   - 任务区
   - AI 工位
   - 电脑/Runner 机房
   - 硬件实验区
   - 审批门岗
   - 交付工坊
   - 档案/知识库
3. 玩法要让复杂研发流程更好理解、更顺手，而不是增加额外负担。

## 产品原则

1. 首要目标是方便开发。
2. 有趣感必须来自真实研发流程的空间化和状态化。
3. 不做只好看但拖慢工作的玩法。
4. 地图是导航和协作界面，不是背景图。
5. 多人联机的未来形态应当是：多人共同进入一个项目基地，管理同一套电脑、AI、任务和审批状态。

## 直接指导当前地图开发的结论

1. 优先搬原游戏里现成的交互节奏和日常循环。
2. 任务田块最适合承接任务播种、成长、收获。
3. 鸡棚和牛棚最适合承接不同 AI 类型与日常照料。
4. 机房/牛棚最适合承接电脑与 Runner。
5. 审批门岗必须保留“高风险动作要过人类确认”语义。
6. 好玩的前提是开发动作更顺，而不是地图更花。
