# AI 协作平台开发规范总览

版本：v0.1  
用途：统一所有人类工程师和 AI 工程师的开发、提交、审查、测试和交付规则  
适用范围：AI 协作开发平台本身

---

## 1. 为什么需要规范

本项目会有多个 AI 和人类共同开发。如果没有统一规范，会出现：

1. 不同 AI 写出不同风格代码。
2. 前后端字段不一致。
3. 任务分支混乱。
4. 重复实现同一功能。
5. AI 修改不属于自己的模块。
6. token 浪费和上下文失控。
7. 安全、权限、硬件边界不清。
8. 代码难以审查和回滚。

所以本项目必须先建立一套基础规范，所有 AI 认领身份后都必须遵守。

## 3. 全局硬规则

所有 AI 必须遵守：

1. 不直接修改 `main` 分支。
2. 不修改不属于自己职责的目录。
3. 不把 token、密钥、密码写进仓库。
4. 不绕过人工确认点。
5. 不自动执行真实硬件操作。
6. 不删除他人代码，除非任务明确要求且经过确认。
7. 不做大范围重构，除非有架构任务和人类确认。
8. 不无目标读取全仓库上下文。
9. 任务完成必须说明修改文件、测试结果、风险和下一步。
10. 上下文过长必须生成摘要或交接包。

---

## 4. 开发顺序

推荐开发主线：

```text
基础环境
  -> 用户/项目/AI/Runner 管理
  -> 任务管理
  -> Git 分支和 diff
  -> Runner 执行任务
  -> Agent Adapter
  -> 需求管理库
  -> 上下文健康和交接包
  -> 安全权限和人工确认
  -> 嵌入式调试辅助
  -> 游戏化研发基地前端
```

不要先做复杂动画、插件市场、复杂计费或真实硬件自动控制。

---

## 5. 第一版必须坚持的边界

第一版目标是打通主链路：

```text
创建项目
添加 AI
注册 Runner
创建任务
创建任务分支
AI 执行或辅助执行
查看 diff
记录 token
审查
人工确认
合并或回滚
```

第一版不做：

```text
复杂 3D 场景
自动控制真实机械臂
AI 自动合并主分支
复杂计费系统
完整插件市场
多租户商业化后台
```

---

## 6. 文档维护规则

1. 新增模块时必须补充对应文档。
2. 新增 API 时必须更新 API 规范。
3. 新增高风险能力时必须更新安全权限规范。
4. 新增前端页面时必须更新前端规范或页面清单。
5. 新增 Runner 能力时必须更新 Runner 规范。
6. 修改任务流时必须更新任务模板规范。
7. 每个阶段结束后由 AI-DOCS 生成一次文档审查报告。

---
