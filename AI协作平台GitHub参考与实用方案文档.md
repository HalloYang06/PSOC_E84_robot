# AI 协作平台 GitHub 参考与实用方案文档

版本：v0.1  
日期：2026-04-19  
用途：整理适合 AI 协作平台的 GitHub 参考项目，并结合嵌入式与机器人开发场景提出实用方案  

---

## 1. 文档目标

本平台的目标不是做一个普通聊天工具，也不是只做一个好看的前端，而是建设一个真正适合：

1. 嵌入式开发
2. 机器人开发
3. 机械臂项目
4. 多人协作
5. 多 AI 分工
6. 多电脑 / 多 Runner / 多设备管理

的协作平台。

因此参考 GitHub 开源项目时，重点不在于“页面好不好看”，而在于：

1. 是否适合真实研发协作。
2. 是否适合任务、代码、设备、实验、知识的统一管理。
3. 是否能支持嵌入式和机器人开发中的特殊流程。
4. 是否能减少会议和人工同步成本。
5. 是否能为后续产品化提供成熟思路。

---

## 2. 参考原则

参考 GitHub 项目时，不建议整套照搬，而是按模块吸收：

1. Git 协作层
2. Agent 执行层
3. 需求与审批流
4. 知识库与决策沉淀
5. 机器人监控与实验流
6. 数据、模型、设备版本管理

每个参考仓库都应该回答两个问题：

1. 能借鉴什么
2. 为什么适合这个平台

---

## 3. GitHub 参考项目清单

### 3.1 Git 协作与仓库管理

#### 1. Gitea

仓库：

[go-gitea/gitea](https://github.com/go-gitea/gitea)

可借鉴点：

1. 自托管 Git 服务。
2. 组织、成员、仓库、Issue、PR、权限模型。
3. 轻量，适合中小团队部署。
4. 适合作为 GitHub 的本地协作层或镜像层。

为什么适合本平台：

1. 你们的平台不应该自己重造 Git 基础设施。
2. 可以把 Gitea 当作代码协作底座，把平台精力放在 AI、任务、审批、实验和设备管理上。
3. 对嵌入式和机器人团队来说，局域网环境和私有仓库都很常见，Gitea 比直接依赖 GitHub 更稳。

建议融合方式：

1. GitHub 继续做远端主仓库。
2. 本平台接入 Gitea 作为本地高频协作层。
3. 平台只增强任务、AI、Runner、审批、回滚，不替代 Git。

---

### 3.2 AI 执行与代码代理

#### 2. OpenHands

仓库：

[All-Hands-AI/OpenHands](https://github.com/All-Hands-AI/OpenHands)

可借鉴点：

1. AI 可以读取代码、执行命令、修改仓库。
2. 任务到结果有完整执行链。
3. 很适合做代码型 Agent 的执行层。

为什么适合本平台：

1. 你们的平台里“线程工位 / AI 工位”最终需要接真实执行 Agent。
2. OpenHands 的价值不在 UI，而在执行闭环。
3. 它能成为“AI 打工者”的执行后端之一。

建议融合方式：

1. 平台负责任务、角色、权限、审批、日志。
2. OpenHands 负责一部分代码执行任务。
3. 对硬件相关任务仍保留人类确认，不让 Agent 直接控制真机。

---

### 3.3 工单、需求和审批流

#### 3. Request Tracker

仓库：

[bestpractical/rt](https://github.com/bestpractical/rt)

可借鉴点：

1. 工单生命周期管理。
2. 状态流转、责任人、评论线程、队列。
3. 权限与流程非常成熟。

为什么适合本平台：

1. 你们的需求信箱、审批单、交接单、本质上都不是聊天，而是工单。
2. RT 的价值在于把沟通变成可追踪、可关闭、可回溯的正式记录。
3. 特别适合把“人 -> AI”、“AI -> AI”的协作做成结构化流程。

建议融合方式：

1. 平台内部的需求单、审批单、交接单都参考工单系统建模。
2. 让“结构化单据”成为协作核心，而不是自由聊天。

---

### 3.4 知识库与决策沉淀

#### 4. Wiki.js

仓库：

[requarks/wiki](https://github.com/requarks/wiki)

可借鉴点：

1. 自托管知识库。
2. Markdown、权限、搜索、结构化内容沉淀。
3. 适合团队长期积累文档。

为什么适合本平台：

1. 你们的平台要沉淀接口契约、芯片手册、原理图说明、实验记录、问题复盘。
2. 知识图书馆不能只是列表，要能长期成长。
3. Wiki.js 的结构很适合“按项目、按模块、按设备、按主题”收纳知识。

建议融合方式：

1. 平台内建轻量知识页。
2. 后期可对接独立知识库服务。
3. AI 沟通结论默认沉淀到知识图书馆，而不是留在聊天里。

#### 5. Log4brains

仓库：

[thomvaill/log4brains](https://github.com/thomvaill/log4brains)

可借鉴点：

1. ADR（架构决策记录）管理。
2. 决策文档和代码仓一起维护。
3. 可视化查看决策历史。

为什么适合本平台：

1. 机器人和嵌入式项目最怕“为什么之前这样设计没人记得”。
2. 多 AI 协作时，架构决策更容易反复被问。
3. ADR 是减少重复沟通、减少上下文浪费的关键。

建议融合方式：

1. 接口变更、协议选择、关键模块边界都必须写 ADR。
2. AI 在回答架构问题时，优先查 ADR。

---

### 3.5 机器人监控与操作台

#### 6. rosboard

仓库：

[dheera/rosboard](https://github.com/dheera/rosboard)

可借鉴点：

1. 浏览器直接看 ROS topic、图像和状态。
2. 轻量，手机也能看。
3. 适合现场查看设备状态。

为什么适合本平台：

1. 你们要让手机、电脑、开发板都能看前端。
2. 硬件实验楼如果只能看文本，会不够直观。
3. rosboard 很适合借鉴“实验楼实时状态页”的设计思路。

建议融合方式：

1. 平台的硬件实验楼接入 ROS 状态可视化。
2. 把 topic 状态、摄像头、节点在线状态纳入实验楼。

#### 7. robotics-ui

仓库：

[sciotaio/robotics-ui](https://github.com/sciotaio/robotics-ui)

可借鉴点：

1. React 前端 + ROS2 数据接口。
2. 配置驱动的机器人操作面板。
3. 插件式布局思路。

为什么适合本平台：

1. 你们的平台不能只看代码，也要能看运行中的机器人状态。
2. robotics-ui 提供了“机器人操作台”的很好参考。
3. 适合借鉴到机房车间、硬件实验楼、控制台区域。

建议融合方式：

1. 平台总控是庄园地图。
2. 实验楼/机房车间参考 robotics-ui 做更专业的控制面板。

---

### 3.6 实验记录、数据与回放

#### 8. Robometry

仓库：

[robotology/robometry](https://github.com/robotology/robometry)

可借鉴点：

1. 机器人遥测、日志、实验数据采集。
2. 更偏工程化数据流。
3. 适合做实验记录和回溯。

为什么适合本平台：

1. 机器人开发不能只有任务，还必须有实验记录。
2. 真实调试中，日志、遥测、状态、视频、结果都要保留。
3. Robometry 的思路很适合“实验记录流”。

建议融合方式：

1. 平台增加实验记录页。
2. 每次实验都有参数、日志、照片、结论和版本快照。

#### 9. ROS2 Message Flow Analysis

仓库：

[christophebedard/ros2-message-flow-analysis](https://github.com/christophebedard/ros2-message-flow-analysis)

可借鉴点：

1. ROS2 消息链路分析。
2. 时序、延迟、消息流追踪。
3. 适合分析复杂分布式机器人系统。

为什么适合本平台：

1. 机器人问题常常不是单点 bug，而是跨节点时序问题。
2. 这个方向特别适合你们后面做“链路诊断”和“性能排查”。

建议融合方式：

1. 在实验楼增加链路分析视图。
2. 后续支持节点级、topic 级时序追踪。

---

### 3.7 设备与版本管理

#### 10. Mender

仓库：

[mendersoftware/mender-server](https://github.com/mendersoftware/mender-server)

可借鉴点：

1. 设备注册、分组、版本管理、远程更新。
2. 适合大规模设备运维。
3. 设备层抽象成熟。

为什么适合本平台：

1. 你们以后不会只管代码和线程，还要管开发板、NanoPi、实验设备。
2. 嵌入式项目需要“整机版本快照”和设备分组。

建议融合方式：

1. 把每块开发板、每台边缘设备都纳入“电脑/设备节点”体系。
2. 平台支持设备版本、固件版本、部署状态、回滚状态查看。

---

### 3.8 数据、模型和实验管理

#### 11. DVC

仓库：

[treeverse/dvc](https://github.com/treeverse/dvc)

可借鉴点：

1. 数据和实验版本管理。
2. 与 Git 兼容。
3. 适合保存日志、数据集、实验结果、模型文件。

为什么适合本平台：

1. 机器人开发后面一定会涉及实验数据、标定文件、模型、日志。
2. 不能只管理代码，不管理数据。

建议融合方式：

1. 平台记录实验数据和模型版本。
2. 任务和实验快照能关联到 DVC 风格的数据版本。

#### 12. ClearML

仓库：

[clearml/clearml](https://github.com/clearml/clearml)

可借鉴点：

1. 实验跟踪。
2. 流水线编排。
3. 指标和结果回溯。

为什么适合本平台：

1. 如果你们后面接触视觉模型、VLA、训练任务，这类实验追踪会很重要。
2. 也适合借鉴任务 -> 运行 -> 结果 -> 回溯的可视化链路。

建议融合方式：

1. 平台实验楼和账房页后续可以吸收 ClearML 思路。
2. 把模型任务和普通代码任务分层管理。

## 4. 对平台真正有帮助的专项方案

下面这些不是花哨功能，而是特别适合嵌入式和机器人开发的实用功能。

### 4.1 硬件调试卡系统

目标：

把每次硬件问题都变成结构化调试记录。

建议内容：

1. 板卡名称
2. 芯片型号
3. 固件版本
4. 当前现象
5. AI 建议检查项
6. 人类测量结果
7. 串口日志
8. 示波器/万用表数据
9. 照片上传
10. 下一步建议

价值：

1. 减少“只靠口头描述”的调试。
2. 适合实验室和多人协作。

---

### 4.2 模块责任地图

目标：

把项目拆成模块，并明确谁负责什么。

建议模块：

1. MCU 固件
2. 驱动与传感器
3. 通信协议
4. Linux 节点
5. ROS 节点
6. 上位机
7. App
8. 服务器
9. 安全系统
10. 文档与实验

每个模块展示：

1. 负责人类
2. 负责 AI
3. 所属电脑
4. 相关目录
5. 当前状态
6. 风险等级

价值：

1. 减少开会问“这块谁管”。
2. 让平台适合复杂机器人项目。

---

### 4.3 整机版本快照

目标：

不只管理代码版本，还要管理整机状态。

建议快照包含：

1. Git commit
2. 固件版本
3. 上位机版本
4. ROS 包版本
5. 配置文件版本
6. 标定文件版本
7. 模型版本
8. 测试结果
9. 审批记录

价值：

1. 方便回滚。
2. 方便复现实验。
3. 方便现场支持。

---

### 4.4 接口契约中心

目标：

减少跨模块反复沟通。

建议纳入：

1. 串口协议
2. CAN 协议
3. ROS topic
4. ROS service
5. WebSocket / HTTP API
6. GPIO / PWM / ADC 约定
7. 传感器字段定义

每条契约绑定：

1. owner
2. consumer
3. 示例数据
4. 版本号
5. 相关文件
6. 最近变更人

价值：

1. 降低 token 消耗。
2. 降低跨模块误解。

---

### 4.5 实验记录流

目标：

让实验也成为平台核心对象，而不是散落在聊天和表格里。

建议记录：

1. 实验目的
2. 设备环境
3. 参数
4. 操作步骤
5. 照片/视频
6. 日志
7. 结果
8. 结论
9. 是否可复现

价值：

1. 真实服务机器人和机械臂项目。
2. 方便复盘和知识沉淀。

---

### 4.6 任务接力模式

目标：

让跨模块任务可以按链条接力完成。

示例：

```text
协议设计
-> MCU 实现
-> Linux 接收
-> ROS 发布
-> UI 展示
-> 真机验证
```

每一棒记录：

1. 输入
2. 输出
3. 接手人
4. 接手 AI
5. 交接包
6. 风险

价值：

1. 特别适合机器人项目。
2. 让多线程、多 AI 协作更自然。

---

## 5. 最推荐优先落地的六项能力

如果只挑最值的，建议优先做：

1. 硬件调试卡系统
2. 模块责任地图
3. 整机版本快照
4. 接口契约中心
5. 实验记录流
6. 任务接力模式

原因：

1. 这些能力最贴近嵌入式和机器人开发的真实痛点。
2. 它们不是花哨展示，而是直接减少沟通和返工。
3. 后续也最容易成为产品差异化卖点。

---

## 6. 最终建议

这个平台后续不要只学“通用 AI 协作平台”，而应该走自己的路线：

**Git 协作底座 + 结构化工单流 + 多 AI 工位 + 硬件实验楼 + 实验记录 + 整机版本快照**

最建议的组合方式：

1. 用 Gitea 的思路做 Git 协作层。
2. 用 RT 的思路做需求单、审批单、交接单。
3. 用 OpenHands 的思路做 AI 执行层。
4. 用 rosboard / robotics-ui 的思路做机器人实验楼。
5. 用 DVC / ClearML / Robometry 的思路做实验、数据、模型与日志管理。
6. 用 Mender 的思路做设备和版本管理。

这样做出来的平台，不只是“AI 协作平台”，而会更像：

**面向嵌入式和机器人团队的人类主导、多 AI 协作、设备与实验一体化研发平台。**
